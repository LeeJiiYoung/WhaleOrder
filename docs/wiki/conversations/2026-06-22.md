### 4. **분산 락 leaseTime 3초가 너무 짧음**
`domain/stock/service/StockLockFacade.java:25,45` — `tryLock(5, 3, SECONDS)`. DB 부하 시 재고 차감 트랜잭션이 3초 안에 커밋되지 못하면 락이 자동 해제되어 **같은 메뉴를 두 스레드가 동시 차감**. 한정 수량 오버셀 발생.
→ leaseTime을 보수적으로 늘리거나(예: 10~30초) 락 갱신 전략 사용. 또는 `Stock` 엔티티에 `@Version` 추가해서 낙관적 락으로 이중 방어.

### 5. **SSE Emitter `onError` 누락 → 메모리 누수**
`domain/order/service/OrderSseService.java:47-63, 87-93` — `register()`와 `registerStatusStream()`에 `onError` 콜백이 없음. 클라이언트 네트워크 끊김/브라우저 강제 종료 시 좀비 Emitter가 `emitters`/`statusEmitters` 맵에 누적 → 장시간 운영 시 OOM. (참고: `registerAdmin()`은 `onError` 있음)

### 6. **IdempotencyService delete+insert race**
`global/idempotency/IdempotencyService.java:31-38` — 타임아웃된 PROCESSING 레코드를 `delete → flush → insert`로 재획득하지만, 두 요청이 동시에 진입하면 둘 다 delete → 둘 다 insert 시도 → 한쪽은 PK 위반으로 실패하지만, 그 사이 비즈니스 로직이 두 번 실행될 가능성. ON CONFLICT의 원자성이 delete+insert 시퀀스에는 적용되지 않음.
→ Redis SETNX(`SET key val NX EX`)로 단일 원자 연산 사용 권장.

### 7. **금액 필드 Integer overflow / 음수 검증 누락**
`Orders.totalPrice`, `Payment.amount` 가 `Integer`. 대량 주문 시 약 21억원 초과로 overflow → 음수 저장 가능. 또한 `@Min(0)` 미설정이라 클라이언트 조작으로 음수 금액도 통과 가능.
→ `Long` 변환 + `@Positive` 검증 + 도메인 setter에서 invariant 체크.

## 자동화 옵션 3가지

**A. SessionStart hook — N일 넘은 메모리 목록 주입**
```bash
find ~/.claude/projects/E--WhaleOrder/memory -name "*.md" -mtime +7
```

**B. Stop hook — 코드 수정 후 메모리 재검토 트리거**
세션 끝에 "이번에 만진 파일 = 메모리에 언급된 파일이면 갱신할지 물어봐" 식 메시지 주입. 갱신 누락을 줄이지만 가끔 시끄러움.

## `@Min(0)`이 효과 없는 이유 두 가지

**1) 사용자가 금액을 보내지 않음**
`PaymentRequest.java` 를 보면 `method`/`storeId`/`orderType`/`customerRequest` 만 받고 금액 필드가 없어요. 서버가 `cartService.getCart(memberId).totalPrice()` 로 직접 계산합니다. 즉 클라이언트 입력 검증으로 막을 게 없음.

**2) 엔티티에 붙여도 JPA persist 시 자동 검증 안 됨**
`@Min(0)` 같은 Bean Validation 어노테이션은 컨트롤러의 `@Valid` 파라미터에서만 자동 실행됩니다. `Orders.totalPrice` 필드에 붙여도 `orderRepository.save()` 가 그냥 통과시킴 (Hibernate가 validator를 호출하려면 `javax.persistence.validation.mode=AUTO` + 클래스패스 검증기 필요한데 기본값에서도 엔티티에 어노테이션만으로는 불충분한 경우 많음).

## ⚠️ 더 중요한 동반 패치: Integer overflow

`@Min(0)` 만 붙이면 음수만 막을 뿐, **양수가 overflow 되어 음수로 떨어지는 케이스는 검증 통과**합니다. 즉:

- `unitPrice(int) * quantity(int)` 계산 도중 overflow 발생 → 음수 결과가 totalPrice 로 전달됨
- 이때 생성자의 `< 0` 검사는 통과(이미 음수 상태) 안 함, 통과함... 잠시. 음수면 막힘. 하지만 **두 음수가 곱해져 양수가 된 케이스**나 **딱 overflow되어 0보다 큰 값이 된 케이스**는 통과.

진짜 해결책은 `Integer` → `Long` 으로 타입 변경 + 계산 시 `Math.multiplyExact()` / `Math.addExact()` 사용. 다만 이건 DB 컬럼 타입 마이그레이션이 따라옴.

## 보통의 패턴: "표시 금액 확인(amount confirmation)"

```
클라이언트 → 서버: "제가 본 금액 ₩12,500 으로 결제 진행할게요"
서버 내부 계산:    cart 기반으로 ₩12,500 직접 산출
서버 검증:        두 값이 같으면 진행, 다르면 거절
서버 → PG:        서버가 계산한 ₩12,500 만 청구 (클라값 절대 사용 안 함)
```

핵심은 **클라가 보낸 값은 "내가 본 화면이 이거였다"는 선언일 뿐, 실제 청구에는 안 쓰는 것**입니다.

### 왜 보내는가 — 세 가지 이점

1. **가격 변동 감지**: 사용자가 결제 화면에 5분 머무는 동안 메뉴 가격이 바뀌었으면 차단
2. **장바구니 동시 수정 감지**: 다른 탭/모바일에서 카트가 바뀐 채로 결제 누르면 차단
3. **UX 명확성**: "표시된 금액과 다릅니다 — 다시 확인해주세요" 라는 명확한 에러 가능

### 실제 PG사 사례

- **토스페이먼츠**: 클라가 `amount` 보내고, 서버 `confirm` API에서 다시 검증 (필수)
- **Stripe PaymentIntent**: 서버에서 amount 지정해 PaymentIntent 생성, 클라는 ID만 들고 confirm
- **카카오페이**: 가맹점 서버가 준비 API에 amount 보내고 토큰 받음, 클라는 토큰으로만 진행

전부 "**계산은 서버, 확인은 양방향**" 구조입니다.

## 2. 이 한 줄짜리 리다이렉트가 토큰을 유출시키는 경로 6가지

| # | 경로 | 무엇이 새는가 | 누가 보는가 |
|---|------|---------------|-------------|
| 1 | **브라우저 히스토리** | URL 전체가 영구 저장 | PC 공유 사용자, 동기화된 다른 기기, 브라우저 익스텐션 |
| 2 | **Referer 헤더** | 콜백 페이지에서 외부 리소스(이미지·analytics·CDN) 로드 시 Referer 에 그대로 실림 | 외부 도메인(GA, 광고 SDK, 이미지 CDN 등) |
| 3 | **서버 액세스 로그** | nginx / ALB / Tomcat / API Gateway 등 경유한 모든 미들박스의 access log 에 query string 기록 | 인프라 운영자, 로그 수집기 |

| 5 | **북마크 / 공유** | 사용자가 주소창 URL 을 복사해서 동료에게 보내면 토큰이 동봉됨 | 무차별 |
| 6 | **CDN / 프록시 캐시** | URL 단위로 캐시되는 환경(특히 GET) 에서 응답이 캐시되면 토큰이 다른 사용자에게 노출 가능 | 가장 사고 큼, 드물지만 치명적 |

## 3. 어떤 토큰이 새느냐에 따라 피해 정도

- **accessToken** — 짧은 만료(보통 30분~1시간). 탈취 시점부터 만료까지만 악용 가능
- **refreshToken** — 긴 만료(7일~30일). 탈취되면 **그 기간 동안 무한정 accessToken 재발급 → 사실상 계정 완전 장악**

지금 코드는 **둘 다** 보내므로 사실상 계정 인수(Account Takeover) 가능한 상태입니다.

## 4. 공격 시나리오 구체적 예

**시나리오 A — Referer 유출**
1. 사용자가 카카오 로그인 → `?accessToken=X&refreshToken=Y` URL 로 콜백 페이지 도착
2. 콜백 페이지에 Google Analytics 스크립트 한 줄 박혀 있음
3. GA 가 자기 도메인으로 hit 보낼 때 Referer 헤더에 우리 URL 그대로 첨부
4. **Google 의 로그에 사용자 토큰이 평문으로 기록됨** (Google이 악용한다는 게 아니라, 우리 자산이 외부 통제권 밖으로 나감)

**시나리오 B — 인프라 운영자**
- 운영자가 nginx access.log 를 grep 하다가 `?accessToken=` 패턴이 보임
- 평문 토큰 복사 → Postman 으로 그대로 호출 → 임의 사용자로 행세 가능

**시나리오 C — 사용자 본인의 부주의**
- 사용자가 "로그인 안 되는데 확인 좀" 하며 콜백 URL 을 카카오톡으로 친구에게 보냄
- 친구가 그 링크 열면 본인 세션이 됨

## 5. 업계 표준 해결책 3가지

### A. httpOnly Cookie (가장 안전, 표준)
서버가 `Set-Cookie: access_token=...; HttpOnly; Secure; SameSite=Strict` 로 토큰 발급. JavaScript 가 접근 불가하므로 URL/Referer/로그에 절대 안 노출. XSS 방어도 자동.
- **장점**: 가장 안전, SPA에서 자동 첨부
- **단점**: CSRF 대비 필요, 백엔드 인증 필터 + 프런트 axios 클라이언트 모두 수정, JWT 필터를 cookie 기반으로 전환

### B. 단기 1회용 Authorization Code (점진 전환에 적합)
1. OAuth 성공 → 서버가 30초 수명의 임시 `code`(랜덤 UUID) 발급, Redis 에 `code → (accessToken, refreshToken)` 저장
2. 리다이렉트 URL 은 `?code=ABC123` 만 (토큰 노출 없음)
3. 프런트가 즉시 `POST /api/auth/oauth2/exchange { code: "ABC123" }` 호출
4. 서버는 Redis 에서 코드 꺼내 토큰 응답 + Redis 코드 삭제(1회용)
- **장점**: 현재 토큰 저장 구조(localStorage 등) 그대로 유지, 변경 범위 작음
- **단점**: 한 번의 추가 라운드트립, Redis 의존(이미 사용 중이라 무료)

### C. URL Fragment (`#` 사용)
`?accessToken=` 대신 `#accessToken=` 으로 변경. Fragment 는 HTTP 요청에 포함되지 않아 서버 로그/Referer 에는 안 새지만 **브라우저 히스토리·우리 자체 로그·북마크 위험은 그대로** 남음.
- **장점**: 한 줄 수정으로 #3·#4(서버/우리 로그) 만 해결
- **단점**: 절반의 해결책. 다른 4가지 경로는 여전. 권장 안 함

## 1. 멱등성 시스템의 기본 컨셉

```
idempotency_key (PK) | status     | response_body | created_at | expires_at
─────────────────────┼────────────┼───────────────┼────────────┼───────────
SHA256(...)          | PROCESSING | null          | T0         | T0+24h
SHA256(...)          | COMPLETED  | "{json...}"   | T0         | T0+10m  ← complete 시 갱신
```

흐름:
- 같은 요청은 같은 키 → PROCESSING 이면 "처리 중", COMPLETED 면 캐시 응답 반환
- **30초 이상 PROCESSING 상태면 "타임아웃" 으로 보고 재획득 허용** ← 이게 위험의 근원

`IdempotencyRecord.java:60-63`:
```java
public boolean isProcessingTimedOut() {
    return status == IdempotencyStatus.PROCESSING
        && createdAt.plusSeconds(PROCESSING_TIMEOUT_SECONDS).isBefore(LocalDateTime.now());
}
```

## 2. `markProcessing()` 의 두 단계 동작

`IdempotencyService.java:24-39`:

```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public boolean markProcessing(String key) {
    // 1단계: 원자적 INSERT (PostgreSQL ON CONFLICT DO NOTHING)
    int inserted = idempotencyRepository.insertProcessingIfAbsent(key, now, now.plusHours(24));
    if (inserted == 1) return true;              // 새 키 → 처리권 획득

    // 2단계: 이미 있는 키 → 만료/타임아웃이면 DELETE 후 재 INSERT
    return idempotencyRepository.findById(key)
        .filter(r -> r.isExpired() || r.isProcessingTimedOut())
        .map(record -> {
            idempotencyRepository.delete(record);
            idempotencyRepository.flush();
            return idempotencyRepository.insertProcessingIfAbsent(...) == 1;
        })
        .orElse(false);
}
```

**1단계는 원자적입니다** (PostgreSQL `ON CONFLICT` 가 row lock 으로 동시성 처리). 진짜 문제는 **2단계의 "delete → flush → insert" 시퀀스**.

## 4. 왜 이런 일이 일어나는가 — 두 가지 잘못된 가정

### 가정 ① **"30초 PROCESSING = 죽은 트랜잭션"**
실제로는 단지 느린 것뿐일 수 있어요. PG 응답 지연, DB 부하, GC pause, 네트워크 글리치 모두 30초쯤 가볍게 넘김. **느린 트랜잭션 == 죽은 트랜잭션 이라는 가정이 깨지면 곧장 중복 결제.**

### 가정 ② **"멱등성 키 = 비즈니스 트랜잭션 lock"**
`markProcessing` 은 `REQUIRES_NEW` 트랜잭션에서 commit 후 종료. 그 다음 시작하는 비즈니스 트랜잭션과는 **완전히 독립**. 즉 멱등성 레코드가 있다는 사실이 "비즈니스 처리가 진행 중이다" 를 보장하지 못함.

→ 이 두 가정 때문에 "타임아웃 감지 → 새 요청 허용" 로직이 위험.

## 5. 추가 잔존 문제

### ⚠ 실패 시 즉시 키 삭제
`PaymentService.java:197-200`:
```java
} catch (Exception e) {
    idempotencyService.delete(key);   // 클라가 재시도 허용
    throw e;

```

A 가 PG 호출 중 예외 → catch 에서 delete(key) → 그 직후 B 가 들어오면 즉시 markProcessing 성공 → 둘 다 결제될 수 있음. (A 의 비즈니스 트랜잭션은 이미 롤백됐을 거라 OK 처럼 보이지만, PG 측이 이미 청구 처리한 케이스라면 보상 트랜잭션 누락)

### ⚠ delete + insert 자체의 비원자성 (원래 보고된 결함)
극단적으로 짧은 윈도우라 잘 안 일어나지만, 두 요청이 거의 동시에 타임아웃 감지하면 둘 다 delete 시도. PostgreSQL row lock 으로 직렬화는 되지만, 한쪽이 delete + insert 완료 후 commit 한 직후 두 번째가 findById 하면 새 createdAt 의 PROCESSING 을 보고 `isProcessingTimedOut() == false` → 안전. **이 시나리오는 사실상 안전**. 진짜 위험은 위의 시나리오 ① (느린 트랜잭션).

## 6. 해결책 — 3가지 옵션

### A. **Redis SETNX + TTL (강력 추천)**
```java
public boolean markProcessing(String key) {
    Boolean ok = redisTemplate.opsForValue()
        .setIfAbsent("idem:" + key, "PROCESSING", Duration.ofSeconds(60));
    return Boolean.TRUE.equals(ok);
}

public <T> void saveResult(String key, T result) {
    redisTemplate.opsForValue()
        .set("idem:" + key, serialize(result), Duration.ofMinutes(10));
}
```

- **원자적**: `SET ... NX EX` 가 단일 명령
- **TTL 자동**: 60초 후 자동 만료 → "재획득 로직" 자체가 사라짐 → race condition 원천 차단
- **PROCESSING 동안 새 요청은 무조건 거절** → 가정 ① 의 위험 사라짐
- Redis 이미 사용 중 (분산 락에 사용) → 추가 의존성 없음
- 단점: Redis 장애 시 멱등성 보장 깨짐 (현재 DB 기반보다 가용성 낮음). 다만 분산 락도 Redis 의존이라 동일 신뢰 모델

### B. PostgreSQL Advisory Lock 으로 비즈니스 트랜잭션 전체를 lock
```java
pg_try_advisory_xact_lock(hash(key))
```
비즈니스 트랜잭션 전체를 lock → 진짜로 "한 번에 하나만 처리". 트랜잭션 종료 시 자동 해제. 가정 ② 도 자연 해결.
- 단점: PostgreSQL 종속, lock 폭증 시 DB 부하

### C. 현 구조 유지 + 타임아웃 제거 + 보상 잡 추가
PROCESSING 타임아웃 30초 → 영구화. 별도 스케줄러가 24시간 넘은 PROCESSING 정리. 타임아웃으로 인한 재획득 자체를 제거.
- 단점: 첫 요청이 진짜 죽으면 사용자가 24시간 동안 같은 카트로 결제 못함 (UX 망함)

## 7. WhaleOrder 에 추천

**A 안 (Redis SETNX)**. 이유:
- Redis 분산 락이 이미 `StockLockFacade` 에서 검증된 인프라

## "괜찮은" 정도가 아니라 권장되는 이유 4가지

### 1. SPOF 가 새로 생기는 게 아님
이미 결제 흐름의 Redis 의존성은 다음과 같이 있습니다:
- `CartService` → Redis (장바구니)
- `StockLockFacade` → Redis (분산 락)
- (예정) `IdempotencyService` → Redis

여기에 멱등성을 빼서 PostgreSQL 에 둔다 해도, **카트와 락이 Redis 라 어차피 Redis 죽으면 결제 자체가 안 됩니다.** 즉 멱등성만 분리해봤자 가용성 이득 없음. **장애 도메인(blast radius)이 이미 동일** → 굳이 분리할 이유 없음.

### 2. 일관성 모델이 통일됨
지금 "분산 락 = Redis 가 살아 있을 때만 정확" 이라는 약속을 이미 받아들였어요. 멱등성도 같은 약속으로 통일되면 **운영 시 의사결정이 단순해집니다** — "Redis 장애 = 결제 전면 차단" 한 가지 시나리오만 대응하면 됨.

### 3. 인프라/모니터링/장애대응 노하우 그대로 활용
- 같은 Redis 클러스터 → 모니터링 대시보드 하나
- 백업/Failover/Sentinel 설정 그대로
- "Redis latency 가 튀면 결제도 영향" 이라는 분석 패턴 동일

### 4. 명령 부하 무시할 만함
- 분산 락: `LOCK` (Redisson 내부적으로 Lua 스크립트), 짧은 TTL
- 멱등성: `SET NX EX`, `GET`, 짧은 명령
- 둘 다 O(1), 평균 0.1ms 수준 → 결제 TPS 가 수천 단위라도 Redis 한 대로 충분

## 한 가지 인식해야 할 점

**같은 Redis 인스턴스를 쓰니 key 네임스페이스만 분리** 하면 됩니다. 충돌 위험 없음:

```
stock:lock:{storeId}:{menuId}    ← 분산 락 (StockLockFacade)
idem:{sha256}                    ← 멱등성 (신규)
```

## 구현 — Redisson API 활용 가능

이미 `RedissonConfig` 가 있으니 `RedisTemplate` 대신 **Redisson 의 `RBucket`** 을 쓰면 코드 스타일이 일관됩니다:

```java
@Service
@RequiredArgsConstructor
public class IdempotencyService {

    private final RedissonClient redisson;
    private final ObjectMapper objectMapper;

    private static final Duration PROCESSING_TTL = Duration.ofSeconds(60);
    private static final Duration RESULT_TTL     = Duration.ofMinutes(10);
    private static final String   PREFIX         = "idem:";

    public boolean markProcessing(String key) {
        RBucket<String> bucket = redisson.getBucket(PREFIX + key);
        // SET NX EX 의 Redisson 버전 — 원자적, TTL 자동
        return bucket.setIfAbsent("PROCESSING", PROCESSING_TTL);
    }

    public <T> T getResult(String key, Class<T> type) {
        String value = redisson.<String>getBucket(PREFIX + key).get();
        if (value == null || "PROCESSING".equals(value)) return null;
        return deserialize(value, type);
    }

    public <T> void saveResult(String key, T result) {
        // COMPLETED 로 덮어쓰면서 TTL 도 10분으로 재설정
        redisson.getBucket(PREFIX + key).set(serialize(result), RESULT_TTL);
    }

    public void delete(String key) {
        redisson.getBucket(PREFIX + key).delete();
    }
}
```

기존 호출부 (`PaymentService.pay()`) 는 시그니처가 동일해서 **수정 불필요**.

## ⚠ 함정 하나 — "같은 음료 한 잔 더" 시나리오

이 멱등성 설계의 흔한 부작용입니다:

```
T+0분  : 카페라떼 1잔 카트에 담고 결제 성공
T+1분  : clearCart 로 카트 비워짐
T+3분  : "한 잔 더!" → 카페라떼 1잔 다시 담고 결제 → ❌ 차단됨
         (cart JSON 이 동일 → 같은 키 → COMPLETED 캐시 hit → 이전 결과 반환)
```

`IdempotencyRecord.complete()` 가 완료 후 10분간 결과를 보관하기 때문에 (`expiresAt = now().plusMinutes(10)`, `IdempotencyRecord.java:48`), **결제 직후 10분 안에 똑같은 카트로 또 시키면 막힙니다.**

## 이 함정을 해결하는 표준 패턴 3가지

| 방식 | 설명 | 변경 범위 |
|------|------|----------|
| **A. 클라이언트 `Idempotency-Key` 헤더** | 프런트가 결제 시도마다 UUID 생성해서 헤더에 실음. 같은 UUID = 재시도, 다른 UUID = 새 결제. **Stripe, 토스페이먼츠 표준** | 프런트 + 서버 키 생성 로직 변경 |
| **B. 키에 timestamp 또는 nonce 포함** | 서버가 카트 내용 + `LocalDate.now()` 또는 `cart.lastModified` 같은 변화 요소를 키에 섞음 | 서버만, 단 정확도 떨어짐 |
| **C. 결제 성공 시 즉시 키 expire** | `saveResult` 후 곧장 `delete(key)` — 캐시 효과는 잃지만 재주문은 자유 | 서버만, 가장 단순 |

## 10초가 가진 실제 효과

현재 재고 차감 트랜잭션:
```
tryLock(5s, 3s) → stockService.deductStock() → @Transactional → UPDATE 1건 → commit
```

일반적인 UPDATE 한 건은 1–50ms, P99 도 100ms 미만. 그래서 **3초도 평상시엔 충분**해 보입니다. 문제는 **튀는 경우** — GC pause, DB 부하 폭증, 커넥션 풀 고갈, 네트워크 마이크로 끊김. 3초 → 10초로 늘리면 이런 케이스 99.X% 가 커버되긴 합니다.

하지만 **트래픽 폭주 + DB 부하 동시 발생** 시 10초도 넘길 수 있어요. 그 순간 다음 시나리오:

```
T+0초:  A tryLock 성공 → UPDATE 진행 중 (DB 느림)
T+10초: leaseTime 만료 → Redis 가 락 키 자동 삭제
T+10초: B tryLock → 성공 (락 없음) → 또 UPDATE
T+12초: A commit → 재고 -5
T+15초: B commit → 재고 -5
                       💥 같은 락 안에서 처리됐어야 할 두 작업이 동시에 → 오버셀
```

즉 leaseTime 늘리는 건 사고 확률을 줄일 뿐, **"길어지는 트랜잭션" 이라는 근본 원인을 해결하지 못함**.

## 더 나은 두 방향

### A. Redisson Watchdog 모드 (정석)
leaseTime 인자를 **생략**하면 Redisson 이 자동 갱신(watchdog)을 켭니다.

```java
// Before
if (!lock.tryLock(5, 3, TimeUnit.SECONDS)) { ... }

// After — leaseTime 인자 제거
if (!lock.tryLock(5, TimeUnit.SECONDS)) { ... }
```

동작:
- 락 획득 후 watchdog 스레드가 10초마다 자동 갱신 → 트랜잭션이 얼마나 길든 보유 유지
- 보유자가 죽으면 30초 안에 자동 해제 (Redis lock timeout 기본값)
- **트랜잭션 길이에 무관하게 안전**

장점: race 원천 차단. Redisson 공식 권장.
단점: 보유자가 hang 되면 30초 동안 다른 요청도 hang. → `@Transactional(timeout = 10)` 같이 잡아주면 트랜잭션 자체가 10초에 강제 종료 → 곧장 finally unlock → 30초 hang 안 됨.

### B. `Stock` 엔티티에 `@Version` 추가 (이중 방어)
분산 락은 그대로 두고, DB 레벨 낙관적 락 추가.

```java
@Entity
public class Stock extends BaseEntity {
    @Version
    private Long version;
    ...
}
```

동작:
- A 가 SELECT → version=1 읽음 → UPDATE 시 `WHERE version=1` → 성공 → version=2
- B 가 (race 로 들어와) SELECT → version=1 읽음 → UPDATE 시 `WHERE version=1` → **0 rows affected** → `OptimisticLockException`
- 둘 중 하나는 실패 → 재시도 또는 사용자에게 "잠시 후 다시 시도" 안내

장점: leaseTime 무관, race 발생해도 두 번째 차감 차단. DB 가 진실의 원천.
단점: race 직격 시 사용자가 재시도 화면 봐야 함 (대신 오버셀은 없음)

## 가장 견고한 조합 — A + B

1. **Watchdog 모드** 로 락 leaseTime 게임 자체에서 빠짐 (락이 트랜잭션 길이에 자동 적응)
2. **@Version** 으로 DB 레벨 이중 방어 (혹시 watchdog 으로도 새어나가는 경우 차단 + Redis 장애 시에도 일부 보호)
3. **@Transactional(timeout = 10)** 추가하면 hang 자동 복구

세 줄짜리 변경으로 락 관련 race 가 사실상 사라집니다.

## 1. ALTER TABLE (한 번)

`@Version` 컬럼 추가는 PostgreSQL 입장에서:

```sql
ALTER TABLE stock ADD COLUMN version BIGINT;   -- nullable, default 없음
```

이 패턴은 **PostgreSQL 11+ 에서 메타데이터만 수정** → 데이터 페이지 안 건드림 → 거의 즉시 완료 + ACCESS EXCLUSIVE 락 짧음.

위험한 패턴(피해야 함):
- `ALTER TABLE ... SET DEFAULT now()` 같은 **비-상수 default** → 모든 row 재작성
- `ALTER TABLE ... SET NOT NULL` (기존 row 가 NULL 인 상태로) → 전체 row 검증 필요

다행히 Hibernate 의 `@Version private Long version` 기본 매핑은 **nullable + default 없음** 이라 메타데이터 변경 한 번으로 끝납니다.

추가로 `Stock` 테이블은 `store × menu` 조합이라 row 수가 작아요 (매장 100 × 메뉴 50 = 5,000 row 수준). 큰 테이블도 아니라 ALTER 비용은 더더욱 무시 가능.

## 2. 매 쿼리 비용 (영구)

### UPDATE
- 기존: `UPDATE stock SET quantity=? WHERE stock_id=?`
- 후: `UPDATE stock SET quantity=?, version=? WHERE stock_id=? AND version=?`

차이:
- **컬럼 하나 더 쓰기**: 8 bytes → I/O 영향 0에 가까움
- **WHERE 조건 한 줄 추가**: PK 로 row 찾은 후 in-memory 비교 → 측정 불가 수준
- UPDATE 결과가 `0 rows affected` 면 `OptimisticLockException` → race 가 실제 발생했을 때만 (평상시 0회)

### SELECT
- 컬럼 하나 더 읽음 → row 한 줄에 8 bytes 추가 → 무시 가능
- 인덱스 영향 0 (version 컬럼에 인덱스 만들 필요 없음)

### 디스크
- 5,000 row × 8 bytes = **40 KB**. 농담 수준.

## 3. 정말 신경 써야 할 한 가지 — 적용 절차

부하보다 **schema 변경 절차가 더 위험**합니다.

**현재 ddl-auto 설정에 따라:**
- `update` 또는 `create` → 앱 재시작만으로 자동 추가됨. 편하지만 **운영에서 이렇게 두면 의도치 않은 변경이 적용될 수 있어 위험**
- `validate` 또는 `none` → 명시 마이그레이션 필요

권장 절차 (Zero-downtime):
1. **마이그레이션 SQL 먼저 실행** — `ALTER TABLE stock ADD COLUMN version BIGINT;`
2. **앱 재배포** — `@Version` 추가된 코드 + Hibernate 가 자동으로 새 row 부터 version=0 부터 시작
3. **기존 row 의 version=NULL** 은 Hibernate 가 처음 UPDATE 할 때 자연스럽게 0/1 로 초기화

이러면 다운타임 없이 안전하게 추가 가능. 이미 `Orders.version` 에 적용된 패턴(`Orders.java:31-32`) 이라 일관성도 ↑.

## 결정 요약 (12:20) — 금액 음수 차단 (엔티티 invariant + DB CHECK)

### 배경
- `Orders.totalPrice`, `Payment.amount`, `OrderItem.unitPrice` 가 모두 `Integer` 인데 어떤 검증도 없었음
- Bean Validation 어노테이션은 컨트롤러 `@Valid` 에서만 자동 발동 → JPA persist 시점 보호 없음

### 적용 — 2중 방어

**1) 엔티티 생성자 invariant** (코드 레벨)
- `Orders.java` — `totalPrice >= 0`
- `Payment.java` — `amount >= 0`
- `OrderItem.java` — `quantity >= 1` (이미 DB CHECK 있었음), `unitPrice >= 0`

**2) DB CHECK 제약** (스키마 레벨 — 마지막 방어선)
- `db/schema.sql`:
  - `orders` 테이블에 `CONSTRAINT chk_orders_total_price CHECK (total_price >= 0)` 추가
  - `order_item` 테이블에 `CONSTRAINT chk_order_item_price CHECK (unit_price >= 0)` 추가 (기존 `chk_order_item_qty` 와 병기)
  - `payment` 테이블에 `CONSTRAINT chk_payment_amount CHECK (amount >= 0)` 추가

### 운영 적용 시 주의
ddl-auto=update 는 컬럼·테이블 추가만 자동 적용, **CHECK 제약 자동 추가 X**. 새 환경은 schema.sql 기반으로 적용되지만, **기존 운영 DB 에는 수동 마이그레이션 필요**:

```sql
ALTER TABLE orders     ADD CONSTRAINT chk_orders_total_price CHECK (total_price >= 0);
ALTER TABLE order_item ADD CONSTRAINT chk_order_item_price   CHECK (unit_price  >= 0);
ALTER TABLE payment    ADD CONSTRAINT chk_payment_amount     CHECK (amount      >= 0);
```

기존 데이터에 음수가 있으면 ALTER 실패 → 사전 점검 SQL:
```sql
SELECT count(*) FROM orders     WHERE total_price < 0;
SELECT count(*) FROM order_item WHERE unit_price  < 0;
SELECT count(*) FROM payment    WHERE amount      < 0;
```

## 금액에 `DOUBLE`/`FLOAT` 을 쓰면 안 되는 이유

부동소수점은 **정확하지 않아요.**

```
0.1 + 0.2 == 0.30000000000000004   ← Java/JS/C/Python 모두 동일
```

원 단위 계산이 누적되면 1원, 0.99원 오차가 생기고:
- 결제 1000건 집계 → 합계가 진짜 합계와 0.5원 어긋남
- 환불 계산 → 부분 환불 금액이 0.001원 차이로 모자라거나 남음
- 회계 정산 → 매출이 안 맞아서 분기 결산이 깨짐

PostgreSQL `DOUBLE PRECISION` 도 같은 IEEE 754 8-byte 부동소수점 → DB 에 들어가는 순간 정밀도 손실. 한 번 깨진 회계는 복구가 거의 불가능하니 **무조건 정수 또는 BigDecimal/NUMERIC**.

## 올바른 선택지 3가지

| 타입 | 한계 | 적합 케이스 |
|------|------|-------------|
| **`Integer` / `INTEGER`** (현재) | 약 **21억원** | 단일 결제 한 건에는 충분, 하지만 누적 집계 overflow 위험 |
| **`Long` / `BIGINT`** ✅ 추천 | 약 **9,200경원** (사실상 무한) | 원화(KRW) 처럼 소수점 없는 통화 |
| **`BigDecimal` / `NUMERIC(p, s)`** | 임의 정밀도 | 외화 환산, 이자율, 부가세 분배 등 소수점 필요한 계산 |

## WhaleOrder 는 `Long` (BIGINT) 가 정답

이유:
- KRW(원) 는 소수점 단위 없음 → 정수면 충분
- 누적 집계(매출, 결제 합계)·이론적 overflow 까지 모두 차단
- Java 연산 단순 (`+`, `-`, `*`, `/` 그대로)
- BigDecimal 의 단점(equals 가 scale 까지 비교, 비교가 compareTo 만 가능, 직렬화 부담) 회피

만약 외화(USD 등) 결제까지 확장한다면 **센트 단위 Long** 패턴 (Stripe 표준) — `$12.34` 를 `1234L` 로 저장. BigDecimal 안 써도 됨.

## "잘못 만들었네" 까지는 아닙니다

`INTEGER` 가 한 잔에 21억원이면 평범한 음료 결제 단위에는 사실 부족하지 않아요 (한 잔 5천원 → 42만 잔까지 안전). 다만:
- **단일 주문 totalPrice = unitPrice × quantity** 계산 중 overflow 가능 (큰 단가 × 큰 수량)
- **매장별 일/월 매출 집계** 시 SUM 결과가 INT 범위 넘으면 깨짐

성장 가능성을 고려하면 BIGINT 가 보수적으로 안전. "잘못 만들었다" 가 아니라 "초기에는 무리 없고 지금 격상 시점" 정도예요.

## ⚠ 운영 적용 시 수동 마이그레이션 필수

`ddl-auto=update` 는 컬럼 타입 변경을 자동으로 적용하지 않습니다. **앱 재배포 전에** 운영 DB 에 실행:

```sql
ALTER TABLE menu        ALTER COLUMN base_price       TYPE BIGINT;
ALTER TABLE menu_option ALTER COLUMN additional_price TYPE BIGINT;
ALTER TABLE orders      ALTER COLUMN total_price      TYPE BIGINT;
ALTER TABLE order_item  ALTER COLUMN unit_price       TYPE BIGINT;
ALTER TABLE payment     ALTER COLUMN amount           TYPE BIGINT;
ALTER TABLE goods       ALTER COLUMN price            TYPE BIGINT;
```

PostgreSQL 의 INTEGER → BIGINT 는 row 재작성 + ACCESS EXCLUSIVE 락 → 현재 row 수에서는 빠르지만, 큰 테이블이면 `pg_repack` 같은 온라인 마이그레이션 도구 고려.

## 왜 키에 storeId 가 없는가

`cart:{memberId}:{storeId}` 로 가면 매장별 카트가 따로 만들어집니다. 사용자가 매장 A 의 메뉴 5개 + 매장 B 의 메뉴 3개를 동시에 카트에 가지고 있을 수 있게 됨. 그러나 사이렌오더의 결제 모델은 **한 번에 한 매장 주문**입니다. 그래서 설계 결정:

- **키는 `cart:{memberId}` 한 가지로 유지** (매장별 분리 X)
- 대신 **`CartService.ensureSingleStore()` 가 다른 매장 메뉴 담기 시도를 차단**
- 다른 매장 메뉴를 담으려 하면 `DifferentStoreCartException` (HTTP 412) → 프런트가 confirm → 동의 시 `force=true` 로 기존 카트 비우고 새 매장 메뉴 담음

같은 정보가 `docs/wiki/domains/cart.md` 의 "단일 매장 정책" 섹션에 명시되어 있어요. `redis-usage.md` 는 키 구조만 표시하고 정책은 cart.md 로 위임한 형태.

## Dockerfile 동작 방식

`Dockerfile` 과 `frontend/Dockerfile` 둘 다 **멀티스테이지 빌드** 로 소스를 이미지에 **굽는** 구조:

```dockerfile
FROM eclipse-temurin:21-jdk-alpine AS builder
COPY src src                                    # 빌드 시점 스냅샷
RUN ./gradlew bootJar -x test --no-daemon       # JAR 생성

FROM eclipse-temurin:21-jre-alpine              # 런타임 단계
COPY --from=builder /app/build/libs/*.jar app.jar
```

→ `volumes:` 마운트 없음 → 한 번 빌드된 이미지는 **불변(immutable)**. 코드 바꾸면 새 이미지를 빌드해야 함.

Prometheus 에는 메트릭이 들어오는데(쿼리에서 보임) Grafana 대시보드만 No data 라면, **대시보드 패널의 쿼리가 k6 메트릭명을 정확히 사용 안 하는** 경우입니다.

k6 prometheus-rw output 의 메트릭명은 **`k6_` 접두사** 가 붙어요:
- `k6_http_req_duration_seconds` (히스토그램)
- `k6_vus` (현재 VU)
- `k6_iterations_total`
- 커스텀 카운터: `k6_payment_success_total`, `k6_system_error_total`
- 커스텀 Trend: `k6_payment_duration_ms`

대시보드 패널 쿼리가 다른 이름(예: `http_req_duration_seconds` 처럼 prefix 없이) 으로 되어 있으면 No data. Grafana 패널 편집 → 쿼리 확인 후 알려주시면 대시보드 JSON 도 같이 고치겠습니다.
