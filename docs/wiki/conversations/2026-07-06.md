## 문서가 말하는 3중 방어의 실체

`docs/wiki/architecture/concurrency-control.md:33-44` 에 정확히 나와 있습니다:

```
[1차] Redisson 분산 락 (watchdog)          ← 락
[2차] @Transactional(timeout = 10)          ← 락 아님 (트랜잭션 강제 종료)
```

## "3중 방어"라고 부르는 게 타당한가

**타당한 관점** — "동시성 사고를 막는 방어선"이라는 넓은 의미면 3개 맞습니다:
1. 진입 자체를 직렬화 (분산 락)
2. 진입한 트랜잭션이 무한정 hang 못 하게 (tx timeout)
3. 그래도 뚫린 경우 DB에서 lost update 차단 (@Version)

각각 다른 실패 모드(race / hang / Redis 장애)를 커버하므로 계층 방어로 부를 만함.

**엄밀한 관점** — "동시성 **락** 3중"이라고 하면 부정확. 락은 분산 락 + 낙관 락 **2개**, 나머지 하나는 tx 타임아웃입니다. 실제로 6/23 대화 로그에서도 README가 "Redisson 분산락 하나만" 언급한 것을 지적하며 "watchdog + tx timeout + @Version 3중"으로 갱신했다는 기록이 남아 있습니다 (`conversations/2026-06-23.md:1339, 1360, 1366`).

## 권장

- 발표/문서에 쓸 때는 **"3중 방어 (분산 락 + 트랜잭션 타임아웃 + 낙관적 락)"** 처럼 세 요소를 나열해 주면 오해가 없습니다.

아니요, **동시성 제어(concurrency control)**는 상위 개념이고 비관적 락은 그중 한 가지 구현 방식입니다.

## 개념 계층

```
동시성 제어 (Concurrency Control)
├─ 락 기반 (Locking)
│   ├─ 비관적 락 (Pessimistic Lock)   ← 미리 잠근다
│   │   ├─ DB 행 락 (SELECT ... FOR UPDATE)
│   │   └─ 분산 락 (Redisson RLock 등)
│   └─ 낙관적 락 (Optimistic Lock)    ← 커밋 시점에 검증
│       └─ @Version, CAS 등
├─ 비락 기반 (Lock-free)
│   ├─ MVCC (PostgreSQL 스냅샷)
│   └─ 원자 연산 (Redis SET NX, INCR)
└─ 격리 수준 (Isolation Level)
    └─ READ_COMMITTED / REPEATABLE_READ / SERIALIZABLE
```

## 용어 구분

| 용어 | 의미 |
|------|------|
| **동시성 제어** | 여러 트랜잭션/스레드가 같은 데이터에 동시 접근할 때 정합성을 지키는 **모든 기법의 총칭** |
| **비관적 락** | "충돌이 자주 날 것이다"라고 가정하고 **미리 행을 잠가서** 다른 트랜잭션을 대기시키는 방식 (예: `SELECT ... FOR UPDATE`) |
| **낙관적 락** | "충돌은 드물 것이다"라고 가정하고 **일단 진행 → 커밋 시점에 version 비교**로 검증. 충돌 시 예외 |

## 이 프로젝트에서

- **동시성 제어 = 상위 목표** ("수천 명 동시 주문에도 재고 정합성 유지")
- 이 목표를 달성하기 위해 아래 3가지 기법을 조합:
  - 분산 락 (Redisson) — 넓게 보면 비관적 락의 분산 버전
  - 낙관적 락 (`@Version`)
  - 트랜잭션 타임아웃

그래서 README 제목 `동시성 제어 — Redisson 분산 락 + 낙관적 락` 는 "동시성 제어라는 큰 주제 아래, 구체적으로 이 두 락을 씁니다"라는 뜻이지 "동시성 제어 = 비관적 락"이라는 뜻이 아닙니다.

## 헷갈리기 쉬운 이유

## 왜 필요한지 (역할 재확인)

`StockService.java:90-91` 주석에도 명시돼 있습니다:

> `timeout=10 으로 트랜잭션이 hang 되어도 10초 안에 강제 종료 → finally unlock 까지 빠르게 도달.`
> `watchdog 자동 갱신 모드와 결합해 락이 무한정 보유되는 상황도 방지.`

즉 1차 방어(watchdog 분산 락)의 부작용을 억제하는 안전장치입니다.
- watchdog은 트랜잭션이 아무리 길어져도 락을 계속 붙잡아주는 게 장점
- 반대로 말하면 **트랜잭션이 hang 되면 락도 무한정 hang**됨 → 다른 사용자 전체 블로킹
- `timeout = 10`이 이 시나리오를 10초 만에 강제 종료 → `finally { lock.unlock() }` 도달 보장

## 결론

- **주문(재고) 경로**: 분산 락 + tx timeout + @Version → 3중 방어 성립
- **주문 생성(`PaymentService.pay`)** 자체엔 tx timeout 없음 (`@Transactional(noRollbackFor = ...)`만 있음). 여긴 재고를 안 건드리고 결제·주문 레코드만 만드는 짧은 트랜잭션이라 hang 위험이 낮아서 안 붙임.
- **취소 경로(`OrderCancelService.cancelOrder`)**도 tx timeout 없음. 재고 복구는 여기서 직접 안 하고 `StockLockFacade.restoreStock` → `StockService.restoreStock`(여기 timeout=10)으로 위임하는 구조.

즉 "재고 차감·복구가 실제로 일어나는 곳"에만 정확히 tx timeout이 들어가 있고, 그 앞뒤 얇은 트랜잭션에는 없습니다. 필요한 자리에 정확히 배치된 상태입니다.

## 왜 엄밀한 Saga가 아닌가

Saga의 핵심 정의는 **여러 로컬 트랜잭션에 걸친 분산 트랜잭션 + 각 단계 실패 시 앞선 단계를 되돌리는 보상 액션**입니다.

라인 193~211은:
- **한 개의 DB 트랜잭션** 안에서 실행됨 (outer `@Transactional`)
- 결제 시도와 주문 취소가 **같은 커밋 경계**
- 재고 차감·외부 결제 승인 같은 **다른 서비스/리소스 롤백이 없음**

즉 이건 그냥 "실패를 DB에 기록"이지 여러 로컬 트랜잭션을 잇는 Saga의 보상 단계는 아닙니다. 굳이 이름 붙이면 **"in-transaction failure recording"** 정도가 정확합니다.

## 진짜 Saga는 여기입니다

**결제는 성공했는데 그 뒤 재고 차감이 실패하는 경우** — 이때 결제/주문이 이미 커밋된 뒤라 되돌릴 수 없고, 그래서 진짜로 보상이 필요합니다.

```
PaymentService.pay()               ← 트랜잭션 1 (Order 생성, Payment SUCCESS) commit
  → AFTER_COMMIT Kafka 발행
    → OrderKafkaConsumer.consume()
      → OrderProcessingService.process()   ← 트랜잭션 2 (재고 차감)
          └ 실패 3회 → DLT
              ↓
    → OrderKafkaConsumer.consumeDlt()
      → OrderProcessingService.compensate() ← ★ 여기가 진짜 Saga 보상
          ├─ StockLockFacade.restoreStock() (재고 원복)
          ├─ paymentService.cancelPayment() (Payment → CANCELLED)
          └─ orderCancelService.cancelOrder() (Orders → CANCELLED)
```

`OrderKafkaConsumer.java:38-51`, `OrderProcessingService.compensate()` — **이미 커밋된 다른 트랜잭션들을 뒤에서 되돌리는** 코드가 여기 있습니다. 이게 교과서적 Saga입니다.

| 위치 | 성격 |
|------|------|
| `PaymentService.pay()` line 193~211 | 같은 tx 안에서 결제 실패 상태 기록 — **Saga 아님**. 주석은 표현이 느슨. |
| `OrderProcessingService.compensate()` (DLT 컨슈머가 호출) | 커밋된 결제/주문을 뒤에서 되돌림 — **진짜 Saga 보상** |

```
[1] OrderProcessingService.process() 에서 예외 throw
        ↓
[2] KafkaConfig.kafkaErrorHandler (KafkaConfig.java:41)
      = DefaultErrorHandler + DeadLetterPublishingRecoverer
      → FixedBackOff(1초, 3회) 재시도
        ↓ 3회 모두 실패
[3] DeadLetterPublishingRecoverer
      → 원본 토픽명 + ".DLT" 로 KafkaTemplate.send()
      → "order-created.DLT" 토픽에 메시지 발행
        ↓
[4] Kafka 브로커: order-created.DLT 파티션에 저장
        ↓
[5] Spring Kafka 컨테이너 (KafkaListenerContainerFactory)
      → 앱 시작 시 @KafkaListener 스캔해 "order-created.DLT" 구독
```
