## 1. 통합 테스트 vs 단위 테스트

### 왜 통합 테스트인가

동시성 제어, 멱등성, Saga 보상 트랜잭션은 **단위 테스트로는 검증 불가능**하다.

```
단위 테스트 (Mock 사용)
  ┌──────────────────────────────────────────┐
  │  stockService.deductStock() → Mock 처리  │
  │  → "락이 실제로 걸리는지" 알 수 없음      │
  └──────────────────────────────────────────┘

통합 테스트 (실제 DB + Redis)
  ┌──────────────────────────────────────────┐
  │  실제 PostgreSQL에서 SELECT FOR UPDATE    │
  │  실제 Redis에서 분산락 획득/해제           │
  │  → 동시성 문제가 실제로 발생하는지 검증    │
  └──────────────────────────────────────────┘
```

| 검증 항목 | 단위 테스트 | 통합 테스트 |
|---|---|---|
| 비즈니스 로직 | O | O |
| 실제 DB 반영 여부 | X | O |
| Redis 락 동작 | X | O |
| 트랜잭션 원자성 | X | O |
| 동시성 경쟁 조건 | X | O |

## 2. Testcontainers

### 개념

테스트 실행 시 실제 Docker 컨테이너(PostgreSQL, Redis 등)를 띄워서 테스트하는 라이브러리.

```
기존 방식: H2 인메모리 DB
  → 실제 운영 DB(PostgreSQL)와 SQL 문법이 달라서 놓치는 버그 발생
  → ON CONFLICT DO NOTHING, FOR UPDATE 등 PostgreSQL 전용 기능 테스트 불가

Testcontainers: 실제 PostgreSQL 컨테이너
  → 운영 환경과 동일한 DB로 테스트
  → PostgreSQL 전용 기능도 그대로 테스트 가능
```

### 이 프로젝트의 구조

```java
// TestContainerBase.java
// static으로 선언 → JVM 전체에서 컨테이너 한 번만 시작, 모든 테스트가 공유
public abstract class TestContainerBase {

    static final PostgreSQLContainer<?> POSTGRES;
    static final GenericContainer<?> REDIS;

    static {
        POSTGRES = new PostgreSQLContainer<>("postgres:16-alpine");
        REDIS = new GenericContainer<>("redis:7-alpine").withExposedPorts(6379);
        POSTGRES.start(); // 병렬 시작으로 대기 시간 단축
        REDIS.start();
    }

    @DynamicPropertySource
    static void overrideProperties(DynamicPropertyRegistry registry) {
        // 컨테이너가 랜덤 포트를 사용하므로 실행 후 주소를 동적으로 주입
        registry.add("spring.datasource.url", POSTGRES::getJdbcUrl);
        registry.add("spring.data.redis.host", REDIS::getHost);
        registry.add("spring.data.redis.port", () -> REDIS.getMappedPort(6379));
    }
}
```

### static 선언의 이유

```
static X → 테스트 클래스마다 컨테이너 새로 시작 → 테스트 100개 = 컨테이너 100번 시작
static O → JVM 전체에서 1번만 시작 → 테스트 100개를 같은 컨테이너에서 실행
```

테스트 격리는 컨테이너를 새로 띄우는 게 아니라 `@BeforeEach`에서 데이터를 삭제하는 방식으로 해결한다.

## 3. EmbeddedKafka

### 왜 필요한가

`application-test.yaml`에서 `kafka.enabled`를 별도로 설정하지 않으면 `KafkaConfig`가 자동 로드된다(`matchIfMissing = true`).  
실제 Kafka 브로커가 없으면 컨텍스트 로딩 실패.

```yaml
# application-test.yaml
spring:
  kafka:
    # @EmbeddedKafka가 있으면 embedded broker 주소 사용, 없으면 localhost:9092
    bootstrap-servers: ${spring.embedded.kafka.brokers:localhost:9092}
```

```java
// @EmbeddedKafka 사용 → 테스트 JVM 안에 실제 Kafka 브로커 내장 실행
@EmbeddedKafka(partitions = 1, topics = {"order-created", "order-created.DLT"})
class SagaCompensationTest extends TestContainerBase { }
```

### Kafka 관련 테스트 2가지 방식

| | `SagaCompensationTest` | `OrderIntegrationTest` |
|---|---|---|
| **목적** | 서비스 로직 직접 검증 | Kafka 경유 E2E 검증 |
| **호출 방식** | `process()`, `compensate()` 직접 호출 | `kafkaProducer.publish()` |
| **속도** | 빠름 | 느림 (Consumer 처리 대기) |
| **검증 범위** | 모든 보상 분기 | Kafka 메시지 흐름 |

## 4. 동시성 테스트 — CountDownLatch 패턴

### 왜 CountDownLatch를 쓰는가

단순히 스레드를 여러 개 만들면 동시성 문제가 재현되지 않는다.

```
스레드 생성 타이밍이 다 달라서 실제로는 순차 실행에 가까워짐
→ 동시성 버그가 발생하지 않아 테스트가 통과해버림
```

3개의 Latch를 써서 모든 스레드를 같은 순간에 출발시킨다.

```java
CountDownLatch ready  = new CountDownLatch(threadCount); // 준비 완료 신호
CountDownLatch start  = new CountDownLatch(1);           // 스타팅 건
CountDownLatch finish = new CountDownLatch(threadCount); // 종료 대기

// 모든 스레드 준비 → 스타팅 건 당기기 → 동시 출발
ready.await();
start.countDown(); // 50개 스레드 동시 출발
finish.await(30, TimeUnit.SECONDS);
```

### 타임아웃 설정의 이유

```java
finish.await(30, TimeUnit.SECONDS); // 무한 대기 대신 30초 타임아웃
```

스레드가 예외로 종료되면 `finish.countDown()`이 호출되지 않아 테스트가 영원히 멈춘다.  
타임아웃을 걸어두면 이 경우에도 테스트가 FAILED로 종료된다.

## 5. 멱등성 (Idempotency)

### 개념

같은 요청이 여러 번 와도 결과가 한 번 처리한 것과 동일한 성질.

```
클라이언트가 주문 요청 → 서버 응답 전에 네트워크 끊김
→ 클라이언트: "실패했나?" → 재시도
→ 서버: 이미 처리한 요청인가? 아닌가?
```

### 핵심 메커니즘 — PostgreSQL ON CONFLICT DO NOTHING

```sql
INSERT INTO idempotency_key (idempotency_key, status, created_at, expires_at)
VALUES (:key, 'PROCESSING', :now, :expiresAt)
ON CONFLICT (idempotency_key) DO NOTHING
```

- 키가 없으면 삽입 → 반환 행 수 1 → **처리 권한 획득**
- 키가 이미 있으면 아무것도 안 함 → 반환 행 수 0 → **다른 요청이 처리 중**

이 SQL이 원자적으로 실행되기 때문에 50개 스레드가 동시에 INSERT를 시도해도 단 하나만 성공한다.

### 상태 흐름

```
없음 → [markProcessing] → PROCESSING → [saveResult] → COMPLETED → (10분 후 만료)
                              ↓
                         [delete]  ← 처리 실패 시
                              ↓
                            없음 (재시도 가능)
```

### getResult 반환 규칙

```java
// COMPLETED + 만료 전 → 캐시된 결과 반환
// PROCESSING → null (아직 처리 중)
// 레코드 없음 → null (새 요청)
// 만료 → null (TTL 지남)
```

## 6. Saga 패턴 — 보상 트랜잭션

### 개념

MSA에서 여러 서비스에 걸친 작업의 일관성을 유지하는 방법.  
하나의 큰 트랜잭션 대신, 각 단계마다 "실패 시 되돌리는 로직(보상 트랜잭션)"을 정의한다.

### 이 프로젝트의 Saga 흐름

```
결제 성공(PaymentService)
    ↓
Kafka 발행 (order-created 토픽)
    ↓
Consumer 재고 차감 시도(OrderProcessingService.process)
    ├─ 성공 → stockDeducted = true, 주문 정상 진행
    └─ 실패 (재고 부족)
         ↓
    보상 트랜잭션
         → 주문 CANCELLED
         → 결제 CANCELLED (환불)
```

### DLT (Dead Letter Topic) 시나리오

```
Kafka Consumer 처리 실패
    ↓
1초 간격 3회 자동 재시도 (KafkaConfig.kafkaErrorHandler)
    ↓
3회 후에도 실패 → order-created.DLT 토픽으로 이동
    ↓
OrderKafkaConsumer.consumeDlt() 수신
    ↓
orderProcessingService.compensate(orderId) 호출
    ↓
보상 트랜잭션: 주문 + 결제 취소
```

### 보상 트랜잭션이 멱등성을 가져야 하는 이유

```java
@Transactional
public void compensate(Long orderId) {
    cancelOrder(orderId);
}

protected void cancelOrder(Long orderId) {
    orderRepository.findById(orderId).ifPresent(order -> {
        if (order.getStatus() == OrderStatus.CANCELLED) {
            return; // 이미 취소됨 → 스킵
        }
        order.cancel();
        // ...
    });
}
```

DLT 메시지는 네트워크 재전송 등으로 중복 도착할 수 있다.  
이미 취소된 주문에 `compensate()`를 다시 호출해도 예외 없이 스킵되어야 한다.

## 7. stockDeducted 플래그

### 왜 필요한가

```
문제: 고객이 주문 취소 시 재고를 복구해야 하는가?
     "Kafka Consumer가 재고를 차감했는가?" 를 어떻게 알 수 있는가?
```

Kafka는 비동기이므로 주문 생성 시점에 재고 차감 여부를 알 수 없다.

```java
// Orders 엔티티
@Column(nullable = false, columnDefinition = "boolean not null default false")
private boolean stockDeducted = false; // Consumer가 차감 완료 후 true로 설정

// 주문 취소 시 재고 복구 여부 판단
if (order.isStockDeducted()) {
    stockLockFacade.restoreStock(storeId, menuId, quantity);
}
```

| 시점 | stockDeducted | 취소 시 재고 복구 |
|---|---|---|
| Consumer 처리 전 | false | 복구 안 함 (차감된 적 없음) |
| Consumer 처리 후 | true | 복구함 |

## 8. 테스트 격리 — FK 역순 삭제

### 왜 역순인가

FK(외래 키) 제약 조건 때문에 참조 당하는 테이블을 먼저 삭제하면 에러 발생.

```
payment_history → payment → order_item → orders → stock → menu → store → member
                                                                  ↑
                                              menu_id FK로 참조 → menu를 먼저 삭제하면 에러
```

```java
@BeforeEach
void setUp() {
    // 참조하는 쪽(자식)부터 삭제
    paymentHistoryRepository.deleteAll();  // payment를 참조
    paymentRepository.deleteAll();         // orders를 참조
    orderHistoryRepository.deleteAll();    // orders를 참조
    orderRepository.deleteAll();           // orders cascade → order_item 함께 삭제
    stockRepository.deleteAll();           // store, menu를 참조
    menuRepository.deleteAll();
    storeRepository.deleteAll();           // member를 참조
    memberRepository.deleteAll();          // 참조 없음
}
```

### Orders cascade = ALL

```java
@OneToMany(mappedBy = "orders", cascade = CascadeType.ALL, orphanRemoval = true)
private List<OrderItem> orderItems;
```

`orderRepository.deleteAll()`을 호출하면 `order_item`이 자동으로 함께 삭제된다.  
별도로 `orderItemRepository.deleteAll()`을 호출할 필요 없다.
