## 왜 Kafka가 아니라 DB인가

Kafka의 DLT는 **"실패한 메시지를 재처리·보상하기 위한 파이프라인"**이지, 관리자가 브라우저로 페이징·검색하며 들여다보는 조회 저장소가 아닙니다. 그래서 이 프로젝트는:

- **자동 보상**(DLT Consumer → `compensate()` → 주문 취소·환불)까지는 Kafka로 처리하고,
- 그마저도 안 되는 **최종 실패건**은 조회·수동보정을 위해 **관계형 DB 테이블에 별도 적재**해 두는 구조입니다. (역할 분담: 주문/결제/재고 정합성 데이터 = PostgreSQL)

## 권장 경로: Elastic IP + 도메인 A레코드 + Caddy(자동 HTTPS)

### 1. Elastic IP 할당 (필수 — 안 하면 DNS가 깨짐)
t3.micro는 stop→start 하면 public IP가 바뀝니다(런북 C-1의 재부팅 절차). 도메인이 옛 IP를 가리키게 되니 **Elastic IP를 할당해 인스턴스에 연결**하세요. AWS 콘솔 → EC2 → Elastic IP 할당 → 인스턴스에 연결.

### 2. 도메인 A레코드 연결
- 가비아/Route53 등 어디든 도메인의 **A 레코드 → Elastic IP**.
- `dig <도메인>` 또는 `nslookup`으로 전파 확인.

### 3. 보안그룹 인바운드에 80, 443 추가
현재 런북 표엔 8080/3000만 있고 443은 "Caddy 붙이면"으로만 있음 → [ec2-runbook.md:88-95](docs/wiki/operations/ec2-runbook.md#L88-L95). **80, 443을 0.0.0.0/0로 개방**하세요(HTTP→HTTPS 리다이렉트 + 인증서 발급에 80 필요).

### 4. Caddy 리버스 프록시 추가 (HTTPS 자동)
Let's Encrypt 인증서를 자동 발급/갱신해줘서 t3.micro 데모에 가장 편합니다. `docker-compose.prod.yml`에 서비스 하나 추가:

```yaml
  caddy:
    image: caddy:2
    container_name: whale-caddy
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile
      - caddy-data:/data
      - caddy-config:/config
    depends_on:
      - frontend
    restart: unless-stopped
    networks:
      - whale-net
```

`Caddyfile` (레포 루트 또는 `~/whale-order/`에 생성):

```
your-domain.com {
    encode gzip

    # SSE(실시간 주문 상태 푸시) 때문에 버퍼링을 꺼야 함 — 아래 주의 참고
    reverse_proxy frontend:80 {
        flush_interval -1
    }
}
```

그리고 `volumes:`에 `caddy-data:` / `caddy-config:` 추가. 이렇게 하면 **`https://your-domain.com` → Caddy → frontend nginx → (`/api`) backend** 순으로 흐릅니다. 프론트 포트(`3000:80`)는 이제 외부 노출이 필요 없으니 내려도 됩니다(내부 통신은 `whale-net`).

### 5. 배포
`docker-compose.prod.yml` + `Caddyfile` 커밋 → `main` push → GitHub Actions 자동 배포([deployment.md:80-88](docs/wiki/operations/deployment.md#L80-L88)). 또는 EC2에서 `docker compose -f docker-compose.prod.yml up -d`.

## ⚠️ 꼭 짚어둘 함정 — SSE

이 프로젝트는 주문 상태를 **SSE로 실시간 푸시**합니다([OrderSseService](src/main/java/com/whale/order/domain/order/service/OrderSseService.java)). 그런데:

- **프론트 [nginx.conf:13-17](frontend/nginx.conf#L13-L17)의 `/api/` 프록시에 `proxy_buffering off`가 없습니다.** 프록시가 응답을 버퍼링하면 SSE 이벤트가 실시간으로 안 내려가고 뭉쳐서 옵니다. Caddy는 위 `flush_interval -1`로 해결되지만, **프론트 nginx 쪽도** SSE 경로에 다음이 필요합니다:

```nginx
location /api/ {
    proxy_pass http://backend:8080;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_buffering off;          # SSE 스트리밍 유지
    proxy_read_timeout 3600s;     # 롱리브드 커넥션 타임아웃 방지
    proxy_http_version 1.1;
}
```

지금도 IP:3000 접속 시 SSE가 동작한다면 nginx 기본 버퍼가 우연히 넘어간 것일 수 있는데, 프록시 계층(Caddy)이 하나 더 끼면 문제가 드러나기 쉽습니다. 도메인 붙일 때 함께 정리하는 걸 권합니다.

## 먼저 개념 구분 — "등록기관" vs "DNS 호스팅"

이 둘은 원래 별개 역할인데 헷갈리기 쉽습니다.

- **도메인 등록기관(Registrar)**: 도메인 소유권을 파는 곳. 가비아, 후이즈, GoDaddy, Route53(Registrar 기능도 있음).
- **DNS 호스팅(네임서버)**: 그 도메인의 레코드(A, CNAME 등)를 실제로 응답해주는 곳. 가비아 DNS, **Route53**, Cloudflare.

**핵심: 가비아에서 도메인을 사도, 네임서버만 Route53으로 위임하면 DNS는 Route53으로 굴릴 수 있습니다.** 둘은 양자택일이 아닙니다.

## 비교

| 항목 | Route53 (AWS) | 가비아 DNS |
|------|---------------|-----------|
| 성격 | AWS 관리형 DNS 호스팅 | 도메인 등록기관이 무료 제공하는 DNS |
| 비용 | 호스팅 존 **$0.50/월** + 쿼리 과금(소액) | 도메인 구입 시 **무료** |
| AWS 통합 | **Alias 레코드**로 ELB·CloudFront·S3에 무료 별칭 연결 | 없음 (일반 A/CNAME만) |
| 고급 라우팅 | 가중치/지연/지역/장애조치(Failover)/Health Check | 없음 (기본 레코드만) |
| 자동화(IaC) | Terraform·CLI·API 완비 | API 제한적, 콘솔 위주 |
| UI/지원 | 영어, AWS 콘솔 | **한국어, 국내 고객지원** |
| 전파 속도 | 글로벌 Anycast, 빠름 | 무난 |

## 이 프로젝트 기준 판단

지금 구성은 **EC2 1대 + Elastic IP 하나**입니다. 필요한 건 `A레코드 → Elastic IP` **딱 하나**뿐이라, Route53의 강점(Alias, 라우팅 정책, Health Check)이 쓰일 데가 없습니다.

- **가비아에서 도메인을 살 예정 → 가비아 DNS 그대로 쓰세요.** A레코드 하나면 끝, 무료, 한국어. 데모엔 이게 정답입니다.
- **Route53을 굳이 쓸 이유**가 생기는 경우:
  - 나중에 **ALB/CloudFront/S3 정적호스팅**으로 확장 → Alias 레코드가 편함(IP 안 바뀌고 무료 별칭).
  - **Terraform 등으로 인프라를 코드화**하고 싶음.
  - Elastic IP를 안 쓰고 로드밸런서 뒤로 갈 계획.

## 추천

**지금 단계: 가비아 도메인 + 가비아 DNS(A레코드 → Elastic IP).** 가장 싸고 간단하며, 앞서 말한 배포 절차(Elastic IP 고정)와 바로 맞물립니다.

> 참고: 가비아 DNS에서 A레코드의 호스트(이름)에 `@`(루트 도메인)와 `www`를 각각 걸어두면 `example.com`·`www.example.com` 둘 다 잡힙니다. Route53처럼 루트 도메인에 CNAME은 못 거니(RFC 제약), 루트는 반드시 A레코드로.

이후 ALB나 CloudFront로 확장하는 시점이 오면 그때 네임서버를 Route53으로 위임하는 것도 언제든 가능합니다.

## 2. 무료 서브도메인 (도메인 구입 없이 데모)

포트폴리오/시연 목적이면 **굳이 도메인을 안 사도** 됩니다. IP를 가리키는 무료 서브도메인을 받으면 Let's Encrypt HTTPS까지 됩니다.

- **DuckDNS** — `yourname.duckdns.org`, Elastic IP에 바로 연결, 무료, 가장 간단. **데모 추천.**
- **No-IP** — `yourname.ddns.net`, 무료(주기적 갱신 필요).

## 각자 하는 일

| 도구 | 계층 | 하는 일 |
|------|------|---------|
| **DuckDNS** | DNS | 이름 → IP 변환. `whaleorder.duckdns.org` 를 물으면 **Elastic IP를 알려줌** |
| **Caddy** | 리버스 프록시 + TLS | 그 IP로 들어온 요청을 받아 **HTTPS 인증서 발급**하고 내부 컨테이너로 전달 |

DuckDNS는 "주소록"이고, Caddy는 "건물 정문 경비 + 안내데스크"라고 보면 됩니다. 하나가 다른 하나를 대체할 수 없습니다.

## 왜 둘 다 필요한가

- **DuckDNS만 있으면**: `whaleorder.duckdns.org` 로 접속은 되지만 `http://`(암호화 X)이고, 포트도 `:3000` 같은 게 붙습니다.
- **Caddy를 얹으면**: `https://whaleorder.duckdns.org` (포트 없이, 자물쇠 O). Caddy가 Let's Encrypt에서 인증서를 자동 발급/갱신합니다.

## 전체 흐름 (이 프로젝트 기준)

```
브라우저
  │  https://whaleorder.duckdns.org
  ▼
DuckDNS ── "그 이름? Elastic IP 12.34.56.78 야" (DNS 응답)
  │
  ▼
EC2:443 ── Caddy (TLS 종료, 인증서 자동)
  │
  ▼
frontend nginx (:80) ──/api/──► backend:8080
```

DuckDNS는 처음 이름을 IP로 바꿔줄 때 **한 번** 관여하고, 이후 실제 트래픽은 Caddy → nginx → 백엔드로 흐릅니다.

## 붙이는 순서

1. **DuckDNS**: 계정 만들고 서브도메인 생성 → **Elastic IP 입력**. (이게 되면 `ping whaleorder.duckdns.org` 시 그 IP가 나와야 함)
2. **보안그룹**: 80, 443 개방.
3. **Caddy**: 앞서 드린 `Caddyfile`의 도메인 자리에 DuckDNS 주소를 넣으면 끝.

```
whaleorder.duckdns.org {
    encode gzip
    reverse_proxy frontend:80 {
        flush_interval -1        # SSE 실시간 푸시 유지
    }
}
```

Caddy가 뜰 때 이 도메인으로 자동으로 인증서를 받아오는데, **그러려면 1번(DNS가 이 EC2를 가리킴) + 2번(80 포트 개방)이 먼저 되어 있어야** 발급이 성공합니다. 순서가 중요합니다.

## 정리

- **DuckDNS = 무료 DNS**(이름↔IP), **Caddy = HTTPS 리버스 프록시**. 역할이 달라 **같이 사용**합니다.
- 굳이 안 겹치니 "둘 중 뭐 쓰지?"가 아니라 "**둘 다**"가 맞습니다.
- HTTPS가 필요 없으면 Caddy는 빼고 DuckDNS + 프론트 포트 `80:80`만으로도 `http://whaleorder.duckdns.org` 는 됩니다.

## 짚어둘 점

- **인증서 rate limit**: Caddy가 재기동마다 재발급하지 않도록 `caddy-data` 볼륨을 뒀습니다. 테스트 중 도메인/설정을 반복해서 틀리면 Let's Encrypt 주간 한도(도메인당 5회)에 걸릴 수 있으니, 실패가 반복되면 Caddyfile에 `acme_ca https://acme-staging-v02.api.letsencrypt.org/directory` 를 잠깐 넣어 스테이징으로 테스트 후 지우세요.
- **백엔드 8080은 여전히 직접 노출**(평문 http)됩니다. 모든 트래픽을 Caddy TLS로 강제하려면 나중에 `docker-compose.prod.yml`에서 backend의 `ports: 8080` 도 `expose`로 바꾸면 됩니다. 지금은 호환성 위해 남겨뒀습니다.
- **커밋은 하지 않았습니다.** 검토 후 원하시면 커밋/푸시 진행하겠습니다.
