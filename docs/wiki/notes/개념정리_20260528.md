# 개념 정리 - 2026.05.28

## 1. Kafka 핵심 개념

### Kafka란?
대용량 실시간 데이터 스트림을 처리하는 **분산 메시지 브로커**.
서비스 간에 메시지(이벤트)를 주고받는 중간 게시판 역할을 한다.

> RabbitMQ와 비슷하지만 다름.
> RabbitMQ = 메시지 전달 후 삭제 / Kafka = 메시지를 로그처럼 보관 (재처리 가능)

---

### 핵심 용어

#### Topic (토픽) — 게시판 이름
```
"order-created"  = 신규 주문 게시판
"order-cancelled" = 취소 주문 게시판
```
목적별로 토픽을 분리해서 관리한다.

#### Producer (프로듀서) — 게시판에 글 올리는 쪽
- 이벤트를 발생시키는 쪽
- WhaleOrder에서는 **결제 서비스** (결제 완료 시 주문 ID 발행)
- **고객(브라우저)이 아님** — 서버 내부 코드

#### Consumer (컨슈머) — 게시판을 읽고 처리하는 쪽
- 토픽을 구독하다가 메시지가 오면 즉시 처리하는 쪽
- WhaleOrder에서는 **주문 처리 서비스** (주문 ID 수신 → 재고 차감)
- **고객(브라우저)이 아님** — 서버 내부 코드
- Producer와 Consumer 모두 우리 서버 코드, 고객은 Kafka를 전혀 모름

```
고객(브라우저) → 우리 서버(Producer) → Kafka → 우리 서버(Consumer)
                      발행                          소비
```

#### Partition (파티션) — 토픽 내 병렬 처리 단위
```
"order-created" 토픽
├── partition 0  ← Consumer 1번 담당
├── partition 1  ← Consumer 2번 담당
└── partition 2  ← Consumer 3번 담당
```
- 파티션 수 = 동시에 처리 가능한 최대 Consumer 수
- orderId 기준으로 파티션에 자동 분배됨

#### Offset (오프셋) — 내가 어디까지 읽었는지 북마크
```
게시판 글 목록:
1번 글 - 주문 #100  ← Consumer가 여기까지 처리함 (offset = 1)
2번 글 - 주문 #101  ← 아직 미처리
3번 글 - 주문 #102  ← 아직 미처리
```
- Redis 큐는 읽으면 사라지지만, Kafka는 읽어도 메시지가 남아있음
- Consumer가 장애 후 재시작해도 offset 기준으로 이어서 처리 가능

#### Consumer Group (컨슈머 그룹)
- 같은 그룹에 속한 Consumer들이 파티션을 나눠서 처리
- WhaleOrder: group-id = "whale-order"

## 2. Redis 폴링 방식 (기존)

### 구조
```
결제 완료
    ↓
Redis Sorted Set에 orderId 저장 (score = 타임스탬프)
    ↑
OrderQueueWorker가 200ms마다 확인
    "새 주문 있어?" "없어" "있어?" "없어" (폴링)
    ↓
주문 발견 → OrderProcessingService → 재고 차감 → SSE 알림
```

### 코드 흐름
```
PaymentService.pay()
    → orderQueueService.enqueue(orderId)    // Redis에 저장

OrderQueueWorker (@Scheduled 200ms)
    → orderQueueService.dequeue()           // Redis에서 꺼냄
    → orderProcessingService.processNext()  // 재고 차감
```

### 단점
| 문제 | 설명 |
|---|---|
| 지연 | 최대 200ms 지연 (워커가 방금 확인하고 지나갔을 경우) |
| 처리량 한계 | 워커 1개가 순차 처리 → 동시 주문 폭발 시 병목 |
| 메시지 유실 | 서버가 Redis에서 꺼낸 직후 죽으면 주문 사라짐 |
| 확장 어려움 | 서버를 여러 대로 늘려도 워커 충돌 위험 |

## 3. Kafka 방식 (변경 후)

### 구조
```
결제 완료
    ↓
Kafka "order-created" 토픽에 orderId 발행 (Producer)
    ↓ (즉시)
OrderKafkaConsumer 수신
    → OrderProcessingService → 재고 차감 → SSE 알림
```

### 코드 흐름
```
PaymentService.pay()
    → orderKafkaProducer.publish(orderId)   // Kafka에 발행

OrderKafkaConsumer (@KafkaListener)
    → 메시지 수신 즉시 실행
    → orderProcessingService.process(orderId)  // 재고 차감
```

### 장점
| 항목 | Redis 폴링 | Kafka |
|---|---|---|
| 처리 속도 | 최대 200ms 지연 | 거의 즉시 |
| 동시 처리량 | 워커 1개 순차 | 파티션 수만큼 병렬 |
| 장애 복구 | 메시지 유실 가능 | offset으로 재처리 가능 |
| 확장성 | 서버 증가 시 충돌 위험 | Consumer 추가로 선형 확장 |
| 처리량 | 수만 건/초 | 수백만 건/초 |

## 4. WhaleOrder 변경 내용

### 교체 대상
| 기존 | 변경 후 | 역할 |
|---|---|---|
| `OrderQueueService` | `OrderKafkaProducer` | 주문 발행 |
| `OrderQueueWorker` | `OrderKafkaConsumer` | 주문 수신 및 처리 |

### 취소 처리 방식 변경
기존 Redis 방식은 `removeFromQueue()`로 큐에서 삭제해서 재고 차감 여부를 판단했음.
Kafka는 메시지를 삭제할 수 없으므로, `Orders` 엔티티에 `stockDeducted` 플래그를 추가해서 판단.

```
기존: cancelOrder() → removeFromQueue() 성공 여부로 재고 복구 결정
변경: cancelOrder() → order.isStockDeducted() 값으로 재고 복구 결정
```

### 파티션 전략
```
"order-created" 토픽: partition=3
orderId를 key로 사용 → 같은 매장 주문이 같은 파티션으로 몰릴 수 있음
→ 필요하면 storeId를 key로 바꿔서 매장별 순서 보장 가능
```

## 5. Kafka vs RabbitMQ 비교

| | RabbitMQ | Kafka |
|---|---|---|
| 메시지 보관 | 소비 후 삭제 | 디스크에 장기 보관 |
| 재처리 | 어려움 | offset으로 쉽게 가능 |
| 처리량 | 수만/초 | 수백만/초 |
| 적합한 용도 | 작업 분배, 이메일/알림 | 이벤트 스트리밍, 로그, 대용량 |
| 운영 복잡도 | 낮음 | 높음 |

## 관계 정리

```
Redis 폴링  →  단순하지만 지연 있고 확장 어려움
Kafka       →  즉시 처리, 병렬 확장, 장애 복구 가능
             단, 인프라 복잡도 증가 (Zookeeper 필요)

WhaleOrder 선택 이유:
  원래 목표에 "Kafka로 대용량 주문 스트림 처리" 명시
  동시 주문 폭발 시 병렬 처리로 대응
  Saga 패턴과 함께 장애 내성 설계 구현
```
