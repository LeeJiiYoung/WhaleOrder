## 이게 왜 "정상"인가

**모든 웹사이트가 이렇게 동작합니다.** 네이버, 구글, 은행 로그인도 개발자도구 Network 탭을 열면 본인 비번이 평문으로 보입니다. 이유는:

- 개발자도구는 **"내 브라우저가 내 요청을 보내는 순간"** 을 보여줍니다. 이건 **내 컴퓨터 안**입니다. 남이 볼 수 있는 게 아니에요.
- 실제 보안은 그 요청이 **네트워크를 타고 나갈 때 HTTPS(TLS)가 암호화**해주는 겁니다. 즉 브라우저 밖으로 나가는 순간부터 암호화됩니다.

**내 개발자도구에 내 비번이 보이는 것 자체는 취약점이 아닙니다.** 공격자가 내 브라우저 개발자도구를 열 수 있는 상황이면 이미 내 컴퓨터가 장악된 거라, 비번 노출은 문제 축에도 안 껴요.

## ⚠️ 하지 말아야 할 "가짜 해결책"

여기서 흔히 **"그럼 JS로 비번을 해싱/암호화해서 보내자"** 는 생각을 하는데, 이건 **안티패턴**입니다:

1. **프론트에서 해싱하면 그 해시값이 곧 비밀번호가 됩니다(pass-the-hash).** 서버는 넘어온 값을 그대로 비교하니까, 공격자는 원본 비번을 몰라도 그 해시값만 가로채 재전송하면 로그인됩니다. 보안이 개선되기는커녕 그대로거나 더 나빠집니다.
2. TLS가 어차피 전 구간을 암호화하므로 **JS 암호화는 중복이고 헛수고**입니다.
3. 비번 해싱은 **서버에서** BCrypt로 해야 맞습니다 (이 프로젝트는 이미 `SecurityConfig`에서 그렇게 하고 있을 겁니다 — 확인해드릴 수 있어요).

## ✅ 진짜 고쳐야 할 것 — HTTPS

**딱 하나 실제 리스크가 있습니다:** 지금 앱은 **평문 HTTP**로 돕니다.

`application.yaml`에 `server.ssl` 설정이 전혀 없고, 프론트도 `http://localhost:8080`으로 직접 호출합니다 ([LoginPage.jsx:107](frontend/src/pages/LoginPage.jsx#L107)). 로컬 개발은 상관없지만, **이 상태로 실제 서버에 배포하면** 비번이 진짜로 네트워크에 평문으로 흘러서 같은 와이파이·중간 경로의 누구나 가로챌 수 있습니다(sniffing). **그게 진짜 노출입니다** — 개발자도구가 아니라 전송 구간이 문제예요.

그래서 실제 액션 아이템은:
1. **프로덕션에서 HTTPS 강제** — 보통 앱 자체에 TLS를 넣기보다, 앞단에 Nginx/ALB(로드밸런서)를 두고 거기서 TLS 종료 + HTTP→HTTPS 리다이렉트. (지금 `docker-compose.yml`을 열어두신 것도 이 맥락이면 딱 맞습니다.)
2. Spring Security에 `requiresChannel().anyRequest().requiresSecure()` 같은 HTTPS 강제 옵션 추가 검토.

## 이 프로젝트(EC2 + docker-compose)에 맞는 방법 3가지

지금 구조는 EC2에 docker-compose로 Spring Boot를 띄우는 형태죠. 앞단에 **리버스 프록시를 하나 세워서 거기서 TLS를 처리(TLS termination)** 하는 게 정석입니다. Spring Boot 자체는 그대로 HTTP로 두고요.

| 방법 | 난이도 | 인증서 자동갱신 | 추천도 |
|---|---|---|---|
| **Caddy** | ⭐ 가장 쉬움 | ✅ 완전 자동 (설정 2줄) | **토이/포폴엔 최고** |
| **Nginx + Certbot** | ⭐⭐ 중간 | ✅ (cron 갱신) | 실무 표준, 이력서에 쓰기 좋음 |
| **AWS ALB + ACM** | ⭐⭐ | ✅ 자동 | ALB가 월 ~$16 유료라 지금 구조엔 오버 |

### 추천: Caddy
Caddy는 **도메인만 적어주면 Let's Encrypt 인증서 발급·갱신·HTTP→HTTPS 리다이렉트를 전부 자동**으로 합니다. docker-compose에 컨테이너 하나 추가하고 `Caddyfile`에 이 정도면 끝:

```
whaleorder.duckdns.org {
    reverse_proxy app:8080
}
```

이 두 줄이 인증서 발급 + HTTPS + 리다이렉트를 다 처리합니다. 배우기도 좋아요.

### 실무 티 내려면: Nginx + Certbot
면접·이력서용이면 Nginx가 더 흔해서 설명하기 좋습니다. 설정이 Caddy보다 길고 Certbot으로 갱신을 따로 걸어야 하지만, "실제 배포에서 Nginx로 TLS 종료했다"가 말하기 좋은 경험이 됩니다.

## 왜 Caddy인지 (Nginx와 뭐가 다른가)

두 방식이 실제로 시켜야 하는 일 목록입니다:

| 해야 할 일 | Nginx 방식 | Caddy 방식 |
|---|---|---|
| 인증서 발급 | Certbot 따로 설치·실행 | **Caddy가 자동** |
| 인증서 갱신(90일마다) | cron/타이머 직접 설정 | **Caddy가 자동** |
| HTTP→HTTPS 리다이렉트 | nginx.conf에 직접 작성 | **Caddy가 자동** |
| 리버스 프록시 설정 | nginx.conf 여러 줄 | `Caddyfile` **2줄** |
| 설정 파일 개수 | nginx.conf + certbot 볼륨 + 갱신 스크립트 | `Caddyfile` **하나** |

Nginx는 "직접 해야 할 일"이 4~5개, Caddy는 사실상 **도메인만 적으면 나머지 전부 자동**입니다. 실무에선 Nginx가 더 흔하지만, 지금처럼 **도메인도 없고 처음 붙이는 상황**이면 Caddy가 압도적으로 편하고, 실패 지점도 적어요. 나중에 여유될 때 Nginx로 바꿔보면 됩니다.

## SSE는 Kafka와 무관한 계층

SSE 푸시는 **인메모리 `ConcurrentHashMap` + `SseEmitter`** 로만 돌아갑니다 ([OrderSseService.java:21,34](src/main/java/com/whale/order/domain/order/service/OrderSseService.java#L21)). Kafka가 껴있든 없든, 실제 SSE 전송은 이 맵을 통해서만 일어나요. Kafka는 "재고차감을 **언제/어느 스레드에서** 트리거하느냐"만 담당하지, SSE 전송 자체엔 관여하지 않습니다.

## Kafka 끄면 흐름이 이렇게 바뀜

```
[Kafka ON]  pay() 커밋 → Kafka 발행 → (다른 스레드)Consumer → process() → SSE notify
[Kafka OFF] pay() 커밋 → AFTER_COMMIT 리스너에서 곧바로 process() → SSE notify  (동기)
```

폴백 지점: [OrderEventListener.java:42-45](src/main/java/com/whale/order/domain/order/event/OrderEventListener.java#L42-L45) — Producer 빈이 없으면 `orderProcessingService.process()`를 **그 자리에서 동기 호출**. 그리고 `process()` 안에서 하는 일은 Kafka 모드와 **완전히 동일**합니다:

- [OrderProcessingService.java:91](src/main/java/com/whale/order/domain/order/service/OrderProcessingService.java#L91) — `orderSseService.notify(...)` (고객에게 결과)
- [:96](src/main/java/com/whale/order/domain/order/service/OrderProcessingService.java#L96) — `broadcastNewOrder(...)` (관리자에게 새 주문)

**즉 SSE를 쏘는 코드는 그대로 실행됩니다.** 경로가 "Consumer 스레드"에서 "결제 요청 스레드"로 바뀔 뿐이에요.

## 타이밍은 오히려 더 안전 — pendingResults 덕분

동기 모드에선 `process()`가 결제 응답이 클라이언트에 돌아가기 **전에** 실행됩니다(AFTER_COMMIT은 pay() 호출 스택 안에서 돎). 그래서 재고차감·`notify()`가 **브라우저가 아직 SSE `/result`에 연결하기 전에** 끝나버릴 수 있는데 — 이게 바로 질답 Q13의 그 race condition이죠.

근데 이건 이미 해결돼 있습니다:

- [OrderSseService.java:74-78](src/main/java/com/whale/order/domain/order/service/OrderSseService.java#L74-L78) — `notify()`가 연결된 emitter를 못 찾으면 **`pendingResults` 맵에 결과를 보관**
- [:47-56](src/main/java/com/whale/order/domain/order/service/OrderSseService.java#L47-L56) — 브라우저가 나중에 `register()`로 붙는 순간 **맵에서 꺼내 즉시 전송**

동기 모드에선 이 pendingResults 경로를 거의 항상 타게 되는데, 그게 정확히 이 상황을 위해 만든 안전장치라 **결과가 유실 없이 전달**됩니다.

## 정리 — SSE 3종 다 정상

| SSE 종류 | Kafka OFF에서 |
|---|---|
| 고객 결과(`/result`) | ✅ 정상 (pendingResults가 타이밍 커버) |
| 관리자 새 주문(`broadcastNewOrder`) | ✅ 정상 (연결된 관리자에게 브로드캐스트, Kafka 무관) |
| 상태 변경(`notifyStatusUpdate`) | ✅ 정상 (관리자 액션 기반, 애초에 Kafka와 무관) |

## 지금 파이프라인 전체 흐름

[deploy.yml](.github/workflows/deploy.yml) 기준, `main`에 push하면:

```
main push
  │
  ├─ [build-and-push] (GitHub 러너에서)
  │    ├─ 백엔드 이미지 빌드 → Docker Hub 푸시
  │    └─ 프론트 이미지 빌드 → Docker Hub 푸시
  │
  └─ [deploy] (appleboy/ssh-action이 EC2에 SSH 접속)
       ├─ git clone 또는 git pull        ← 최신 소스(compose 파일 포함) 가져옴
       ├─ docker-compose pull backend frontend   ← Docker Hub에서 새 이미지 받음
       └─ docker-compose up -d --force-recreate --remove-orphans backend frontend
```

즉 **당신이 EC2에 손대는 게 아니라, Actions가 대신 SSH 들어가서 docker-compose를 돌립니다.** 흔하고 유효한 패턴이에요. (44~52행이 그 SSH 스크립트)

## 방금 바꾼 파일들이 이 파이프라인을 어떻게 타는가

두 파일이 서로 **다른 경로**로 반영됩니다 — 이게 중요해요:

| 바꾼 파일 | 반영 경로 |
|---|---|
| `docker-compose.prod.yml` | EC2의 **`git pull`**(48행)로 반영 — 이미지 재빌드 불필요 |
| `application-prod.yaml` | **backend 이미지 안에 빌드됨** → 이미지 재빌드·푸시·pull로 반영 |

`application-prod.yaml`은 Spring Boot 리소스라 **이미지에 구워집니다.** 그래서 이 변경은 `main` push → 이미지 재빌드가 일어나야 적용돼요. 두 파일 다 push 한 번이면 자동 반영됩니다.

## 좋은 부수 효과 — 옛날 무거운 컨테이너가 자동 정리됨

52행의 **`--remove-orphans`** 덕분에, 새 compose 파일에서 사라진 서비스(kafka·kafka-ui·prometheus·grafana)가 **다음 배포 때 자동으로 제거**됩니다. 즉 지금 EC2에 이미 kafka·모니터링이 떠 있어도, 이 커밋을 배포하면 **알아서 내려가고 메모리가 회수**됩니다. 따로 정리 안 해도 돼요.

## 확인/주의할 점 3가지

1. **postgres·redis는 자동으로 뜹니다.** 스크립트가 `backend frontend`만 명시하지만, backend가 이 둘에 `depends_on`이라 `up`이 의존 서비스를 같이 띄웁니다. 데이터 볼륨도 그대로 유지되고요.

2. **스왑은 파이프라인에 없습니다.** 스왑 2GB는 인프라 1회 설정이라 CI/CD로 안 되고 **EC2에서 직접** 걸어야 합니다(앞서 준 명령). 이건 배포와 별개예요.

3. **모니터링 오버레이는 자동 배포 대상이 아닙니다.** `docker-compose.monitoring.yml`은 필요할 때 EC2에서 수동으로:
   ```bash
   # EC2는 docker-compose(v1)일 가능성이 커서 하이픈 버전으로:
   docker-compose -f docker-compose.prod.yml -f docker-compose.monitoring.yml up -d
   ```
   (제가 문서에 적은 `docker compose`(v2, 띄어쓰기)가 EC2에서 안 되면 `docker-compose`로 쓰세요.)

## A. 인스턴스를 재시작(stop → start)한 경우 — 확인 3가지

이미 세팅된 인스턴스면 손댈 게 거의 없어요. 컨테이너는 자동으로 뜹니다. 순서대로 확인만:

**1️⃣ 스왑 확인 (제일 먼저)**
```bash
free -h        # Swap 항목이 2.0Gi로 잡혀 있어야 정상
```
- `/etc/fstab`에 등록해뒀으면 재부팅 후에도 자동 유지됩니다.
- 만약 `Swap: 0B`면 → fstab 등록을 안 한 것. 아래 B-1을 다시 실행하세요. **(이게 다운의 흔한 원인)**

**2️⃣ 컨테이너 자동 기동 확인**
```bash
docker ps      # whale-backend, whale-frontend, whale-postgres, whale-redis 4개가 Up 인지
```
- `restart: unless-stopped` 라 보통 알아서 떠 있습니다.
- 안 떠있으면 수동으로:
  ```bash
  cd ~/whale-order && docker-compose -f docker-compose.prod.yml up -d
  ```

**3️⃣ 로그/접속 확인**
```bash
docker logs -f whale-backend       # "Started ... in N seconds" 뜨는지, 에러 없는지
```
브라우저로 `http://<EC2-IP>:3000` (프론트) / `:8080` (API) 접속되면 끝.

> 재시작 시엔 **배포를 다시 돌릴 필요 없습니다.** 코드 변경 배포는 `main`에 push하면 Actions가 알아서 해요.

## B. 새 인스턴스를 처음 세팅하는 경우 — 이 순서 그대로

**1️⃣ 스왑 2GB (반드시 맨 먼저 — 안 하면 이후 단계가 OOM으로 멈춤)**

```bash

sudo apt-get install -y docker.io docker-compose
sudo usermod -aG docker ubuntu     # sudo 없이 docker 쓰려면 (재로그인 필요)
```

```bash
git clone https://github.com/LeeJiiYoung/WhaleOrder ~/whale-order
cd ~/whale-order
nano .env
```
`.env`에 넣을 내용 (현재 `docker-compose.prod.yml`이 참조하는 3개):
```
JWT_SECRET=32자이상랜덤문자열
KAKAO_CLIENT_ID=카카오앱키
KAKAO_CLIENT_SECRET=카카오시크릿
```
- **`.env`는 gitignore라 레포에 없습니다.** EC2에 직접 만들어야 해요. 없으면 `JWT_SECRET`이 비어서 토큰 발급이 깨집니다.
- **경로 주의:** 클론 위치가 `~/whale-order`(소문자·하이픈)입니다. `.env`는 반드시 이 폴더 안(`~/whale-order/.env`)에 둬야 compose가 읽습니다. *(트러블슈팅 문서엔 `~/WhaleOrder`로 적혀 있는데 실제 워크플로 경로는 `~/whale-order`예요 — 문서가 안 맞습니다.)*

**4️⃣ AWS 보안그룹 인바운드 포트 열기** (콘솔에서)
| Type | Port | 용도 |
|---|---|---|
| SSH | 22 | 접속 (내 IP만 권장) |
| Custom TCP | 8080 | 백엔드 API |
| Custom TCP | 3000 | 프론트 |
| HTTPS | 443 | *(나중에 Caddy 붙이면)* |

**5️⃣ 첫 배포**
- `main`에 push → GitHub Actions가 이미지 빌드 + EC2 SSH 배포 자동 실행.
- 또는 수동으로: `docker-compose -f docker-compose.prod.yml up -d`

## (A) SSH도 안 될 정도로 느리면 — AWS 콘솔에서 재부팅

## 1. Docker Hub에서 새 토큰 발급

1. https://app.docker.com/settings/personal-access-tokens 접속 (또는 Docker Hub → 우측상단 계정 → **Account Settings → Personal access tokens**)

   - Expiration: 길게(1년) 또는 없음
4. 생성되면 **토큰 문자열 복사** (이 화면 벗어나면 다시 못 봄!)

- **Actions 탭 → 실패한 "배포 수정" 실행 → 우측 `Re-run jobs` → Re-run failed jobs** (재push 없이 재시도)
- 또는 아무 커밋이나 push

---

이건 코드 문제가 아니라 **자격증명 만료**라 제가 대신 못 해드리고 위 3단계는 직접 하셔야 해요. 토큰 갱신하고 재실행하면 이번엔 이미지 빌드·푸시·EC2 배포까지 쭉 갑니다.

> 참고: `docker/login-action`이 실패하면 이후 이미지 빌드·push·배포 스텝이 전부 스킵됩니다. 그래서 **로그인만 고치면** 나머지는 원래대로 돌아가요.

토큰 갱신 후 재실행해서 결과 보여주시면 이어서 확인하겠습니다.


브라우저로 `http://<EC2-IP>:3000` 접속해서 **로그인·주문까지 실제로 되면** 완전히 끝입니다.

## 오늘 정리

| 문제 | 해결 |
|---|---|
| 스왑 없어 OOM 다운 | 스왑 2GB (fstab 등록 ✅) |
| 무거운 8개 서비스 | 슬림 compose 4개로 배포 ✅ |
| kafka 기동 구멍 | `kafka.enabled=false` ✅ |
| **Docker Hub 토큰 만료** | 새 토큰 발급 + 시크릿 갱신 ✅ |

이제 재부팅해도, 재배포해도 **다시 안 터집니다.**

> 💡 참고: Docker Hub 토큰에 만료기간을 짧게 잡으면 몇 달 뒤 또 이 에러가 나요. 이번에 발급할 때 만료를 길게(1년) 잡았으면 한동안 신경 안 써도 됩니다.
