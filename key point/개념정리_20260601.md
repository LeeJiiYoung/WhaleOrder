# 개념 정리 - 2026.06.01

## 1. 카카오 소셜 로그인 (OAuth2 Authorization Code Flow)

### 개념

OAuth2는 사용자가 비밀번호를 직접 넘기지 않고, 카카오 같은 외부 인증 서버에 위임해서 로그인하는 표준 프로토콜이다.  
이 프로젝트에서는 **Authorization Code Flow** 방식을 사용한다.

### 전체 흐름

```
[사용자]          [프론트 (5173)]      [백엔드 (8080)]       [카카오]
   |                    |                    |                  |
   | 카카오 버튼 클릭     |                    |                  |
   |----window.location.href--------------->|                  |
   |              GET /oauth2/authorization/kakao              |
   |                    |                    |---302 redirect-->|
   |                    |                    |    카카오 로그인 URL|
   |<-------------------------------------------------------카카오 로그인 화면
   | 로그인/동의                              |                  |
   |-----------------------------------------------------302-->|
   |                    |            /login/oauth2/code/kakao  |
   |                    |             ?code=ABC&state=XYZ      |
   |                    |                    |<---code 전달-----|
   |                    |          [카카오 API로 access_token 교환]
   |                    |          [카카오 /v2/user/me 호출]    |
   |                    |          [Member 조회 or 생성]        |
   |                    |          [JWT 발급]                   |
   |                    |<--302 /oauth2/callback?accessToken=--|
   |                    |                    |                  |
   |  OAuth2CallbackPage 렌더링              |                  |
   |  localStorage에 토큰 저장              |                  |
   |  → /stores 이동                        |                  |
```

### 핵심 개념 3가지

| 개념 | 설명 |
|---|---|
| Authorization Code | 카카오가 발급하는 일회성 코드. 백엔드가 이걸로 실제 access_token과 교환한다 |
| state | CSRF 방지용 랜덤값. 요청 시 생성해서 세션에 저장, 콜백에서 일치 여부 검증 |
| providerId | 카카오 회원 고유번호. 동일 사용자가 재로그인해도 항상 같은 값 |

## 2. 구현 구조

### 파일 역할

```
global/auth/oauth2/
├── KakaoOAuth2UserService.java   # 카카오 응답 파싱 + Member 조회/생성
├── CustomOAuth2User.java         # OAuth2User + CustomUserDetails 래퍼
└── OAuth2SuccessHandler.java     # JWT 발급 + 프론트로 리다이렉트

global/config/
└── SecurityConfig.java           # .oauth2Login() 설정
```

### KakaoOAuth2UserService — 핵심 로직

```java
// 1. 카카오 API로 사용자 정보 가져오기 (super.loadUser)
OAuth2User oAuth2User = super.loadUser(userRequest);

// 2. 카카오 응답 구조 파싱
// { id: 12345, kakao_account: { profile: { nickname: "홍길동" } } }
String providerId = String.valueOf(attributes.get("id"));
String nickname = profile.get("nickname");

// 3. 기존 회원이면 닉네임 업데이트, 없으면 신규 생성 (자동 회원가입)
memberRepository.findByProviderAndProviderId(KAKAO, providerId)
    .map(existing -> { existing.updateNickname(nickname); return existing; })
    .orElseGet(() -> memberRepository.save(
        Member.builder().provider(KAKAO).providerId(providerId)...build()
    ));
```

### OAuth2SuccessHandler — JWT 발급 후 리다이렉트

```java
// JWT 생성
String accessToken  = jwtProvider.generateAccessToken(member.getMemberId(), member.getRole());
String refreshToken = jwtProvider.generateRefreshToken(member.getMemberId());

// 프론트로 리다이렉트 (nickname은 URL에 넣지 않음 — 이유는 아래 트러블슈팅 참고)
String targetUrl = UriComponentsBuilder.fromUriString(redirectUri)
    .queryParam("accessToken", accessToken)
    .queryParam("refreshToken", refreshToken)
    .queryParam("role", member.getRole().name())
    .build().toUriString();

getRedirectStrategy().sendRedirect(request, response, targetUrl);
```

### Member 테이블 설계 — LOCAL / KAKAO 통합

```
LOCAL 회원 : userId + password 필수 / providerId = null
KAKAO 회원 : providerId 필수      / userId = null, password = null
공통       : memberId (PK), name, nickname, role
```

JWT의 subject는 항상 `memberId`이므로 로그인 방식에 관계없이 동일한 인증 필터를 사용한다.

## 3. 프론트엔드 처리 (OAuth2CallbackPage)

```
백엔드 302 리다이렉트
  → localhost:5173/oauth2/callback?accessToken=eyJ...&refreshToken=eyJ...&role=CUSTOMER
  → OAuth2CallbackPage 렌더링
  → localStorage에 accessToken, refreshToken, role 저장
  → /api/members/me 호출로 nickname 저장
  → CUSTOMER → /stores, ADMIN → /admin/store-create 이동
```

## 4. 트러블슈팅

### 문제 1: 흰 화면 (Tomcat Location 헤더 오류)

**원인**  
리다이렉트 URL 쿼리 파라미터에 한글 닉네임을 넣었더니 Tomcat이 HTTP 헤더에 비ASCII 문자를 넣을 수 없어 Location 헤더 자체를 제거해버렸다.

```
The HTTP response header [Location] has been removed because it is invalid
```

**해결**  
nickname은 URL에서 제거. 로그인 후 `/api/members/me`로 별도 조회.

---

### 문제 2: accessToken이 항상 null (로그인 페이지로 돌아옴)

**원인**  
React 18 `<StrictMode>`는 개발 환경에서 `useEffect`를 두 번 실행한다.

```
1차 실행: URL에 accessToken 있음 → 저장 → navigate('/stores') → URL이 /stores로 변경
2차 실행: URL이 이미 /stores → accessToken 없음 → navigate('/login')
```

**해결**  
`useRef`로 이미 처리됐으면 건너뛰고, URL 파라미터를 컴포넌트 초기화 시점에 캡처.

```js
const processed = useRef(false)
const params = useRef(new URLSearchParams(window.location.search)) // 최초 URL 캡처

useEffect(() => {
  if (processed.current) return  // 두 번째 실행 차단
  processed.current = true
  
  const accessToken = params.current.get('accessToken')
  ...
}, [navigate])
```
