## CountDownLatch

특정 작업이 모두 완료될 때까지 스레드를 대기시키는 동기화 도구.

```java
CountDownLatch ready = new CountDownLatch(20); // 카운터 = 20
CountDownLatch start = new CountDownLatch(1);  // 카운터 = 1

// 각 스레드: "나 준비됐어"
ready.countDown(); // 20 → 19 → ... → 0

// 메인 스레드: 20명 전부 준비될 때까지 블로킹
ready.await();

// 동시 출발 신호
start.countDown(); // 1 → 0, 대기 중이던 모든 스레드 동시 출발
```

`start.await()`을 각 스레드가 기다리다가, `start.countDown()` 한 번으로 전부 동시에 풀린다.  
→ 동시 주문 시뮬레이션에 사용.

## CopyOnWriteArrayList

멀티스레드 환경에서 안전하게 쓸 수 있는 리스트.

일반 `ArrayList`는 여러 스레드가 동시에 `add()`하면 데이터가 유실되거나 예외가 난다.  
`CopyOnWriteArrayList`는 `add()` 할 때마다 내부 배열을 복사한 뒤 새 원소를 추가해서 충돌을 방지한다.

```java
// 20개 스레드가 동시에 add() → 안전
CopyOnWriteArrayList<long[]> arrivals = new CopyOnWriteArrayList<>();
arrivals.add(new long[]{cNum, System.nanoTime()});
```

쓰기보다 읽기가 훨씬 많을 때 적합. 쓸 때마다 복사하므로 쓰기가 잦으면 비효율적.

## ExecutorService (스레드 풀)

스레드를 직접 생성하지 않고, 미리 만들어둔 스레드 풀에 작업을 위임하는 도구.

```java
ExecutorService executor = Executors.newCachedThreadPool();

executor.submit(() -> {
    // 무거운 작업 (DB 조회, sleep, SSE 이벤트 전송...)
});

return emitter; // HTTP 요청 스레드는 즉시 반환
```

HTTP 요청 스레드가 직접 무거운 작업을 하면 수십 초간 블로킹된다.  
`executor.submit()`으로 백그라운드 스레드에 넘기면 요청 스레드는 즉시 `emitter`를 반환하고 끝난다.

## SSE (Server-Sent Events)

서버가 클라이언트에게 데이터를 단방향으로 스트리밍하는 방식.

```
HTTP 연결 유지
서버 → 클라이언트: event: result\ndata: {"success":true}\n\n
서버 → 클라이언트: event: complete\ndata: {"finalStock":0}\n\n
연결 종료
```

- WebSocket은 양방향, SSE는 서버→클라이언트 단방향
- `Content-Type: text/event-stream` 헤더로 구분
- Spring에서 `SseEmitter`로 구현

### Race Condition (경쟁 조건)

워커가 SSE 연결보다 먼저 처리를 끝내면 이벤트가 유실된다.

```
워커 처리 완료 → notify() 호출 → emitter 없음 → 이벤트 유실
브라우저 SSE 연결 → 이미 늦음
```

해결: `pendingResults` 버퍼에 결과를 임시 저장 → 연결 시 즉시 전송.

## 멱등성 (Idempotency)

같은 요청을 여러 번 보내도 결과가 한 번만 처리되는 성질.

```
클라이언트: 주문 요청 (key: abc123)
서버: PROCESSING 레코드 삽입
서버: 주문 처리
서버: COMPLETED 레코드로 업데이트

클라이언트: 네트워크 오류로 재요청 (같은 key: abc123)
서버: 이미 COMPLETED → 저장된 결과 그대로 반환
```

DB의 `ON CONFLICT DO NOTHING`으로 원자적으로 처리.  
실패 시 키를 삭제해서 클라이언트가 재시도할 수 있게 한다.

## 분산 락 vs 대기열

| | 분산 락 | 대기열 |
|---|---|---|
| 방식 | 모든 스레드가 락 경쟁 | 도착 순서대로 순차 처리 |
| 처리 순서 | 랜덤 (락을 먼저 잡는 쪽) | FIFO (먼저 도착한 순서) |
| 공정성 | 없음 | 있음 |
| 처리 속도 | 빠름 (병렬) | 느림 (순차) |

분산 락 데모에서 Thread 순서가 뒤섞이는 이유: 어떤 스레드가 락을 먼저 획득하느냐는 OS 스케줄러가 결정하기 때문.

## Redis 분산 락 (Redisson)

Redis의 `SET NX`(SET if Not eXists)를 기반으로 동작.

```
스레드 A: SET lock:menu:1 "A" NX EX 5  → 성공 (락 획득)
스레드 B: SET lock:menu:1 "B" NX EX 5  → 실패 (이미 존재)
스레드 B: 재시도...

스레드 A: 재고 차감 완료 → DEL lock:menu:1 (락 해제)
스레드 B: SET 성공 → 락 획득
```

- `NX`: 키가 없을 때만 SET → 원자적으로 단 하나의 스레드만 성공
- `EX 5`: 5초 후 자동 만료 → 락 해제 못 해도 데드락 방지
- Redisson이 재시도, 만료, 해제를 자동 처리
