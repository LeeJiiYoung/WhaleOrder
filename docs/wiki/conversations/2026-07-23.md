**동작 방식**

- 그 외 예외면 → `FixedBackOff(1000L, 3)` 규칙대로 1초 간격 3회 재시도 후 DLT

**왜 이렇게 나누나 — 재시도의 의미**

재시도는 "일시적 문제(transient failure)" 를 가정합니다. 예:
- DB 커넥션 순간 끊김
- Redis 락 획득 실패 (`StockLockException`) → 잠깐 후엔 풀림
- 외부 API 타임아웃

반면 "결정적 실패(deterministic failure)" 는 몇 번을 다시 해도 같은 결과가 나옵니다:
- `IllegalArgumentException` → 메시지 페이로드가 잘못됨 (`orderId=null` 같은 것). 100번 재시도해도 여전히 null
- `IllegalStateException` → 이 프로젝트에선 재고 부족 신호로 씀. 재고가 0인 상태는 1초 뒤에도 0

이런 예외를 재시도하면 **CPU/네트워크만 낭비하고 DLT 도착이 3초 늦어질 뿐** 이라, 바로 DLT 로 보내서 `consumeDlt()` → `compensate()` (보상 트랜잭션) 을 즉시 실행시키는 게 낫습니다.

**대응되는 반대 개념**

`addRetryableExceptions(...)` 도 있어요. 이건 화이트리스트 방식 — "이 예외들만 재시도, 나머지는 즉시 DLT". 이 프로젝트는 블랙리스트(`addNotRetryableExceptions`)를 골랐는데, 알 수 없는 예외는 일단 재시도해 보자는 관대한 정책이에요.

**주의점**

- 매칭은 `instanceof` 기준이라 하위 클래스도 포함됩니다. `IllegalStateException` 을 넣으면 `StockLockException extends IllegalStateException` 이면 얘도 재시도 안 됨 — 그래서 프로젝트에서 `StockLockException` 을 별도 클래스로 분리한 거예요(주석에도 "IllegalStateException(재고 부족)과 구분하여 Kafka가 재시도하도록 한다" 라고 명시).
- 즉 예외 계층 설계가 이 설정과 연동되어 있어요: **재시도 가능한 일시적 실패는 별도 예외 타입**, **결정적 실패는 `IllegalArgumentException`/`IllegalStateException` 계열** 로 던지는 규칙입니다.

`id` 를 지정하지 않으면 스프링이 자동으로 유니크한 이름을 붙여줘서 이 문제는 안 생겨요.

지금은 `order-created` 를 `OrderKafkaConsumer.consume()` 하나만 구독 중인데, 만약 "주문 생성 시 매장 사장님에게 푸시 알림" 기능을 추가한다면:

```java
// 기존 — 주문 처리 (재고 차감, 결제 확정)
@KafkaListener(topics = "order-created", groupId = "whale-order")
public void consume(Long orderId) { ... }

// 신규 — 사장님 알림 (독립된 groupId 로 팬아웃)
@KafkaListener(topics = "order-created", groupId = "whale-notification")
public void notifyStoreOwner(Long orderId) { ... }
```

이렇게 두면 주문 처리 로직이 실패해서 재시도/DLT 로 가더라도 알림은 독립적으로 발송됩니다. 관심사 분리가 자연스럽게 되는 거예요.

**정리**

| 상황 | 결과 |
|------|------|
| 같은 topic, **다른** groupId | 팬아웃 — 각 메서드가 모든 메시지 받음 |
| 같은 topic, **같은** groupId | 경쟁 — 메시지가 나눠져서 하나에만 감 |
| 같은 topic, 같은 groupId, 같은 id | 시작 실패 |

거의 항상 케이스 1(다른 groupId) 이 정답입니다.

**User (09:44):**

팬아웃이 있으면 팬인도 있어?

**Assistant (09:45):**

네, 있어요. 다만 Kafka 세계에서는 **팬인이 팬아웃보다 훨씬 흔하고 자연스러운 패턴**입니다.

**팬아웃(fan-out) — 1 → N**

한 이벤트를 여러 컨슈머가 각자 받아서 처리:

```
                    ┌── whale-order (주문 처리)
order-created ──────┼── whale-notification (알림)
                    └── whale-analytics (통계)
```

**팬인(fan-in) — N → 1**

여러 소스에서 오는 이벤트를 하나의 리스너/토픽으로 모음. 두 가지 층위에서 얘기할 수 있어요:

**① 리스너 레벨 팬인 — 여러 토픽 → 하나의 메서드**

앞서 나온 `topics = {...}` 문법이 정확히 이 케이스입니다:

```java
// 감사 로그 — 여러 도메인 이벤트를 하나의 리스너에 모아 저장
@KafkaListener(topics = {"order-created", "payment-completed", "stock-updated"},
               groupId = "whale-audit")
public void audit(ConsumerRecord<String, Object> record) {
    auditRepository.save(record.topic(), record.value());
}
```

```
order-created ──────┐
payment-completed ──┼──> audit()
stock-updated ──────┘
```

**② 토픽 레벨 팬인 — 여러 프로듀서 → 하나의 토픽**

사실 Kafka 는 **기본적으로 팬인 구조**예요. 하나의 토픽에 여러 프로듀서가 자유롭게 발행할 수 있으니까요:

```
KioskA ──┐
KioskB ──┼──> order-created ──> Consumer
MobileApp ┘
```

이 프로젝트에서 `PaymentService.pay()` 가 `order-created` 를 발행하는데, 나중에 관리자 페이지에서 수동 주문을 넣는 기능이 생겨도 같은 토픽에 발행하면 Consumer 는 아무 변경 없이 그대로 처리합니다. 이게 팬인의 힘이에요.

**실무에서 자주 쓰는 팬인 패턴**

1. **감사(audit) / 로깅** — 여러 이벤트를 한 곳에서 기록
2. **집계(aggregation)** — 여러 소스 이벤트를 모아 대시보드 지표 생성
3. **이벤트 소싱의 스냅샷 빌더** — 여러 도메인 이벤트를 순서대로 리플레이해서 상태 재구성
4. **알림 허브** — `order-*`, `payment-*`, `stock-*` 을 모아 한 채널로 발송

**팬아웃 + 팬인 조합 — 파이프라인**

실전에선 이 둘을 섞어서 씁니다:

```
                 ┌── enrichment 서비스 ──┐
raw-events ──────┤                       ├──> processed-events ──> consumers
                 └── validation 서비스 ──┘
    (팬아웃)                                  (팬인)
```

원본 이벤트를 여러 처리기가 팬아웃으로 받아서 각자 가공한 뒤, 결과를 하나의 후속 토픽으로 팬인시켜 다운스트림에 전달하는 구조입니다.

**정리**

| 패턴 | 의미 | Kafka에서의 표현 |
|------|------|----------------|
| 팬아웃 | 1 이벤트 → N 처리기 | 같은 토픽, 다른 groupId 여러 개 |
| 팬인 (리스너) | N 토픽 → 1 처리기 | `topics = {...}` 배열 |
| 팬인 (토픽) | N 프로듀서 → 1 토픽 | Kafka 의 기본 동작 |

Kafka 의 pub/sub 모델 자체가 "생산자와 소비자를 토픽으로 분리" 하는 거라, 팬인/팬아웃 둘 다 별도 설정 없이 자연스럽게 표현됩니다. 이게 RabbitMQ 같은 큐 기반 시스템과 대비되는 Kafka 의 특징이에요.

**Assistant (09:48):**

한 줄로 요약하면 **RabbitMQ 는 "똑똑한 브로커, 멍청한 소비자"**, **Kafka 는 "멍청한 브로커, 똑똑한 소비자"** 입니다. 이 철학 차이에서 모든 특성이 파생돼요.

## 근본 모델의 차이

**RabbitMQ — 메시지 큐 (Message Queue)**

- 브로커가 메시지를 "배달" 함. Consumer 에게 push
- Consumer 가 ack 하면 브로커에서 **메시지 삭제**
- 라우팅 로직(exchange, binding, routing key)이 브로커 안에 있음
- 큐는 소비되면 비워지는 우편함

**Kafka — 분산 커밋 로그 (Distributed Commit Log)**

- 메시지를 파일에 append 하고 보관 (기본 7일). Consumer 가 pull
- Consumer 가 읽어도 **메시지는 그대로 남음** — 각 Consumer 는 자기가 어디까지 읽었는지(offset)만 기록
- 라우팅 로직 없음. 토픽 이름으로만 구분
- 토픽은 계속 쌓이는 로그 파일

## 주요 항목 비교

| 항목 | RabbitMQ | Kafka |
|------|----------|-------|
| 데이터 모델 | Queue (소비 시 삭제) | Log (소비해도 유지) |
| 전달 방식 | Push (브로커→컨슈머) | Pull (컨슈머→브로커) |
| 처리량 | 수만 msg/s | 수백만 msg/s |
| 지연시간 | 매우 낮음 (μs~ms) | 낮음 (ms) |
| 메시지 순서 | 큐 단위 보장 | 파티션 단위 보장 |
| 재처리 | 불가능 (이미 삭제됨) | 가능 (offset 되감기) |
| 라우팅 | 강력 (topic/direct/fanout exchange) | 단순 (토픽명 매칭만) |
| 팬아웃 | fanout exchange 로 명시적 설정 | groupId 여러 개로 자연 발생 |
| 메시지 우선순위 | 지원 | 미지원 |
| 지연 메시지 | 지원 (plugin) | 미지원 (직접 구현) |
| 프로토콜 | AMQP, STOMP, MQTT | 자체 프로토콜 (TCP) |
| 스케일링 | 수직(스케일업) 위주 | 수평(파티션 추가) |

## 예시로 감 잡기

**"주문이 들어오면 결제 서비스가 처리한다"** — RabbitMQ 가 자연스러움

```
Producer ──> [order.queue] ──> Payment Consumer (ack 후 삭제)
```

- 작업이 소비되면 끝. 재처리할 이유 없음
- Consumer 가 여러 개면 자동으로 부하 분산

**"주문 이벤트를 결제/알림/통계/분석/추천 시스템이 각자 처리한다"** — Kafka 가 자연스러움

```
Producer ──> [order-created log] ──> Payment    (offset=100)
                                 ──> Notify     (offset=100)
                                 ──> Analytics  (offset=95, 좀 느림)
                                 ──> Recommend  (offset=0, 나중에 붙어서 처음부터 읽는 중)
```

- 같은 데이터를 여러 팀이 각자 속도로 소비
- 새 서비스가 나중에 추가돼도 과거 데이터부터 리플레이 가능

## 언제 뭘 쓰나

**RabbitMQ 를 고르는 이유**
- **작업 큐(task queue)** — 이메일 발송, 썸네일 생성, PDF 렌더링
- **RPC/응답 패턴** — 요청-응답이 필요한 비동기 호출
- **복잡한 라우팅** — 조건별로 다른 컨슈머에게 전달
- **낮은 지연시간이 중요** — 실시간 명령 전달
- **메시지 우선순위**나 **지연 전송**이 필요

**Kafka 를 고르는 이유**
- **이벤트 스트리밍** — "무슨 일이 일어났다" 를 여러 시스템이 소비
- **로그 수집/분석** — 대용량 데이터 파이프라인
- **이벤트 소싱 / CQRS** — 상태 변경 이력을 그대로 보관
- **재처리 필요** — 버그 수정 후 과거 이벤트 다시 돌려야 하는 경우
- **초당 수십만+ 처리량**
- **여러 다운스트림 시스템**이 같은 데이터를 각자 소비

## 이 프로젝트에 Kafka 를 쓰는 이유

CLAUDE.md 에 **"대용량 주문 스트림 처리"** 라고 명시된 게 힌트예요. 사이렌 오더 같은 시스템은:

1. **팬아웃이 자연스러움** — 주문 하나가 결제 확정 + 재고 차감 + 매장 알림 + 통계 + 추천 학습 등으로 확산됨
2. **재처리 필요** — 하위 시스템(예: 통계 서버) 이 죽어도 복구 후 offset 되감아 리플레이 가능
3. **높은 처리량** — 피크 시간대 초당 수천~수만 주문
4. **순서 보장** — 파티션 키(예: `storeId`) 로 매장별 순서 유지 가능

만약 이걸 RabbitMQ 로 했다면 "재고 복구 실패 시 다시 시도" 같은 시나리오에서 이미 삭제된 메시지를 어디서 가져올지 고민해야 하고, 프로젝트에서 하고 있는 DLT 기반 보상 트랜잭션 설계도 훨씬 번거로워집니다.

## 요약

- **RabbitMQ = 우편배달부** — 편지를 받는 사람에게 배달하고 자기는 잊어버림
- **Kafka = 도서관** — 책을 꽂아두고 원하는 사람이 원할 때 와서 읽음. 다 읽어도 책은 그대로

작업 분배가 목적이면 RabbitMQ, 이벤트 로그가 목적이면 Kafka — 이 기준으로 고르면 대부분 맞습니다.

## 상황 1: **브로커(서버)가 죽은 경우** — 현재 설정에서 치명적

파티션은 브로커(Kafka 서버) 위에 얹혀 있어요. 브로커가 죽으면 그 위의 파티션이 사라집니다.

**현재 설정 (replicas=1) 에서**:

```
브로커A: 파티션 1 (원본만 존재)
브로커B: 파티션 2 (원본만 존재)
브로커C: 파티션 3 (원본만 존재)
```

브로커A가 죽으면 → **파티션 1 자체가 사라짐**. 트래픽이 파티션 2, 3 으로 자동 분산되지 **않아요**.

- 파티션 1 에 원래 가야 할 메시지들(예: `orderId % 3 == 0` 인 주문) 은 프로듀서가 발행 시도해도 실패
- 이미 파티션 1 에 쌓여 있던 미소비 메시지들은 브로커A 가 복구될 때까지 **읽을 수 없음**
- 프로듀서는 `TimeoutException` 을 받게 되고, 재시도 설정에 따라 다른 파티션으로 보내지 않고 그 파티션에 계속 재시도합니다

**왜 다른 파티션으로 안 넘어가나?**

Kafka 는 파티션을 **키 해싱**으로 결정해요:

```
파티션 번호 = hash(key) % 파티션수
```

`orderId=42, key=42` 라면 항상 파티션 1 로 가도록 계산됨. 이건 **같은 키의 메시지 순서를 보장** 하기 위한 설계라서, 파티션 1 이 죽었다고 파티션 2 로 보내면 순서가 깨져버립니다. 그래서 Kafka 는 "차라리 실패시킨다" 를 선택해요.

## 상황 2: **레플리카가 있는 운영 환경** — 자동 페일오버

`replicas=3` 이면 각 파티션이 3벌 존재합니다:

```
브로커A: 파티션1(leader), 파티션2(follower), 파티션3(follower)
브로커B: 파티션1(follower), 파티션2(leader), 파티션3(follower)
브로커C: 파티션1(follower), 파티션2(follower), 파티션3(leader)
```

브로커A 가 죽으면:
- 브로커A 가 leader 였던 **파티션 1 의 leader 를 브로커B 나 C 로 승격**
- 프로듀서와 컨슈머는 새 leader 로 자동 연결 전환
- **파티션 1 이 계속 살아있음** → 트래픽 손실 없음

이게 replicas 를 3으로 두는 이유이고, KafkaConfig.java:19 주석의 "운영은 3으로 늘림" 이 의미하는 바입니다.

## 상황 3: **Consumer 가 죽은 경우** — 이 경우가 질문 의도에 가까울 수도

브로커가 아니라 **Consumer 인스턴스** 가 죽은 경우라면 질문이 맞아요. 파티션은 살아있고, 그 파티션을 담당하던 Consumer 만 죽은 상황:

```
[리밸런싱 전]
Consumer1 ── 파티션 1
Consumer2 ── 파티션 2
Consumer3 ── 파티션 3
```

Consumer1 이 죽으면:

```
[리밸런싱 후]
(Consumer1 없음)
Consumer2 ── 파티션 2, 파티션 1
Consumer3 ── 파티션 3
```

Kafka 가 **rebalance** 를 트리거해서 남은 Consumer 2, 3 이 죽은 Consumer 의 파티션을 나눠 가집니다. 이 경우엔 질문대로 "2, 3 중 하나에게 작업이 넘어가는" 게 맞아요. 다만 정확히는 **파티션 1 의 담당자가 바뀌는 것**이고, 파티션 자체는 그대로 살아 있어요.

## 정리

| 무엇이 죽었나 | 파티션 1 은? | 트래픽 흐름 |
|--------------|-------------|-----------|
| 브로커 (replicas=1) | 사라짐 | 파티션 1 로 갈 메시지는 발행 실패 |
| 브로커 (replicas=3) | follower 가 leader 로 승격 | 무중단, 자동 페일오버 |
| Consumer 인스턴스 | 그대로 살아있음 | 다른 Consumer 가 파티션 1 인수 |

**질문의 답을 정리하면**:
- "파티션 1 이 죽었다 → 2, 3 이 나눠 갖는다" 는 **틀림**. 파티션은 서로 독립적이라 하나가 죽어도 다른 파티션이 대신 처리해주지 않습니다 (키 해싱 규칙 때문에)
- 만약 **Consumer** 를 뜻한 거라면 맞음 → rebalance 로 파티션 1 이 남은 Consumer 에게 재할당됨

## 종류 1: 브로커 죽기 **전에** 파티션 1 에 이미 쌓여 있던 메시지

이건 브로커의 **디스크에 저장되어 있어요**. Kafka 는 메모리 큐가 아니라 파일 기반 로그니까요.

```
브로커A 다운 시점 파티션 1 상태 (디스크):
offset 100: orderId=42
offset 101: orderId=45
offset 102: orderId=48   ← Consumer 가 여기까지 커밋 (committed offset)
offset 103: orderId=51   ← 미소비
offset 104: orderId=54   ← 미소비
```

브로커A 가 부활하면:
- 디스크에서 로그를 그대로 읽어옴 → 파티션 1 복구
- Consumer 는 마지막으로 커밋한 offset(102) 부터 이어서 poll
- **offset 103, 104 를 정상 소비** → 유실 없음

이게 Kafka 의 강점입니다. "Consumer 가 어디까지 읽었는지" 를 Kafka 자신이 `__consumer_offsets` 라는 내부 토픽에 기록해 두기 때문에, 브로커가 몇 시간 죽어 있다가 살아나도 이어서 처리할 수 있어요.

## 종류 2: 브로커 다운 중에 프로듀서가 **보내려다 실패한** 메시지

이건 프로듀서 쪽 처리에 달렸어요. Kafka 는 아무것도 안 해줍니다.

**프로듀서 재시도 동작**

```java
// Spring Kafka 기본 설정
retries = Integer.MAX_VALUE
delivery.timeout.ms = 120000  // 2분
```

- 프로듀서는 파티션 1 이 살아날 때까지 **최대 2분간 재시도**
- 2분 안에 브로커A 가 부활하면 → 재시도 성공, 메시지 저장됨 ✅
- 2분 넘게 다운되면 → `TimeoutException` 발생, 프로듀서 코드에서 catch 해야 함

**이 프로젝트에서는 어떻게 처리하나**

`PaymentService.pay()` 가 `order-created` 를 발행하는데, 만약 발행이 실패하면?

```java
// 대략의 흐름 (실제 코드 확인 필요)
paymentRepository.save(payment);          // ① DB 커밋
kafkaTemplate.send("order-created", ...); // ② Kafka 발행 실패 → 예외
```

여기서 문제가 생겨요:
- ① 은 이미 커밋됨 (결제 완료)
- ② 는 실패 → 주문 처리 이벤트가 소실됨
- 결과: **결제는 됐는데 재고는 안 빠지고 매장에도 안 보임**

이걸 방지하는 표준 패턴이 **Transactional Outbox 패턴** 이에요:

```
1. 트랜잭션 안에서:
   - payment 테이블에 INSERT
   - outbox 테이블에 "카프카로 보낼 메시지" INSERT   ← 같은 DB 트랜잭션
2. 별도 스케줄러가 outbox 를 읽어서 카프카로 발행
3. 발행 성공 시 outbox row 삭제/마킹
```

이러면 카프카가 다운되어도 outbox 에 남아 있어서 나중에 재발행 가능해요.

**이 프로젝트에 outbox 가 있는지는 확인이 필요합니다** — 원하시면 코드 찾아볼게요.

## 종류 3: replicas=1 에서 **디스크까지 손실된 경우**

브로커A 가 부활하는 게 아니라 **디스크까지 날아간 경우** (SSD 고장, 컨테이너 볼륨 삭제 등):

- 파티션 1 의 로그 파일 자체가 사라짐
- offset 100~104 의 메시지들은 **영구 유실** ❌
- 이게 replicas=1 의 진짜 위험

replicas=3 이었다면 다른 브로커에 복제본이 있어서 복구 가능. KafkaConfig.java:19 의 "운영은 3으로 늘림" 주석이 이걸 막기 위한 거예요.

## 정리 — 파티션 1 부활 후 메시지 운명

| 실패 유형 | 부활 후 어떻게 되나 |
|---------|------------------|
| 파티션 1 에 이미 저장된 미소비 메시지 | Consumer 가 커밋 offset 부터 이어서 소비 ✅ |
| 다운 중 프로듀서가 보내려던 메시지 (2분 내 부활) | 재시도로 정상 발행 ✅ |
| 다운 중 프로듀서가 보내려던 메시지 (2분 초과) | 프로듀서 예외 → outbox 없으면 유실 ❌ |
| 디스크까지 손실 (replicas=1) | 영구 유실 ❌ |
| 디스크 손실이라도 (replicas=3) | 다른 브로커의 복제본에서 복구 ✅ |

**핵심 포인트**

- Consumer 쪽 실패는 Kafka 가 알아서 복구해 줌 (offset 기반)
- Producer 쪽 실패는 앱이 책임져야 함 (outbox 패턴 등)
- 브로커 자체의 디스크 손실은 replicas 개수가 유일한 방어선

## "다 죽는" 시나리오 6가지

**1. 같은 물리 위치에 몰아둔 경우 (가장 흔함)**

```
데이터센터 A:
  브로커1 ── 파티션1(leader)
  브로커2 ── 파티션1(follower)
  브로커3 ── 파티션1(follower)
```

- 데이터센터 정전, 화재, 네트워크 스위치 고장 → **3벌 다 사라짐**
- 실제 사례: 2021년 OVH 데이터센터 화재로 여러 회사 Kafka 클러스터 전멸

**방어책 — Rack Awareness**

```properties
# 각 브로커의 server.properties
broker.rack=us-east-1a  # 브로커1
broker.rack=us-east-1b  # 브로커2
broker.rack=us-east-1c  # 브로커3
```

Kafka 가 이 정보를 보고 **레플리카를 서로 다른 rack(=AZ)에 분산 배치** 합니다. AZ 하나가 통째로 죽어도 나머지 2개 AZ 에 복제본이 있으니 살아남아요.

**2. 프로듀서 acks 설정이 느슨해서 복제 안 된 채 유실**

```java
// 프로듀서 설정에 따라 다름
acks=0    // 브로커 응답 안 기다림 → 다 죽으면 진행 중 메시지 다 유실
acks=1    // leader 만 확인 → leader 만 있고 follower 로 복제 전에 leader 죽으면 유실
acks=all  // 모든 in-sync 레플리카 확인 → 그나마 안전
```

이 프로젝트의 프로듀서 설정 확인은 필요한데, 기본값이 `acks=1` 이라서 명시적으로 `acks=all` 로 안 바꿨다면 **replicas=3 이어도 유실 가능성이 있어요**.

**3. 카스케이드 장애 (연쇄 붕괴)**

```
브로커1 죽음 → 트래픽이 브로커2, 3 으로 몰림
브로커2 과부하로 다운 → 트래픽이 브로커3 으로 집중
브로커3 도 과부하로 다운 → 전멸
```

한 대가 죽으면 남은 브로커의 부하가 늘어나서 도미노처럼 무너지는 패턴. **용량 계획을 항상 "N-1 개로도 버틸 수 있게"** 잡아야 방지 가능.

**4. 사람의 실수 (실제로 제일 흔한 원인)**

- `kafka-topics.sh --delete --topic order-created` 실수 실행
- Kubernetes 에서 Kafka StatefulSet 을 실수로 `kubectl delete`
- Terraform 으로 관리하다 리소스 destroy
- 볼륨 스냅샷 정책 오류로 디스크 이미지 삭제

**방어책**: `delete.topic.enable=false`, RBAC 로 삭제 권한 제한, 별도 백업(예: MirrorMaker 로 다른 클러스터에 복제)

**5. 소프트웨어 버그 / 잘못된 설정 배포**

같은 Kafka 버전을 쓰는 모든 브로커에 있는 버그가 특정 조건에서 터지면 전 브로커가 동시에 죽을 수 있어요. 롤링 업그레이드 중 새 버전에 버그가 있으면 순서대로 다 죽기도 하고요.

**방어책**: 카나리 배포 (한 대만 먼저 새 버전), 다운그레이드 절차 준비

**6. `min.insync.replicas` 를 안 걸어 놓은 경우**

이건 죽는 게 아니라 "죽어도 계속 쓰기가 되어서 데이터가 갈리는" 이슈:

```
정상: leader(브로커1) + follower(브로커2, 3) 모두 sync
브로커2, 3 다운
→ leader 만 남음. 프로듀서는 leader 에만 계속 씀
→ 나중에 leader 도 죽음
→ 브로커2, 3 이 먼저 부활 → 얘네가 leader 로 승격되면서
  브로커1 에만 있던 최신 데이터 유실
```

**방어책**:

```properties
min.insync.replicas=2   # 최소 2개 sync 상태여야 쓰기 허용
```

+ 프로듀서 `acks=all` 조합. 이러면 브로커 1대만 살아있을 땐 아예 쓰기를 거부(예외 발생)해서, 데이터 무결성을 지연시간과 맞바꾸는 구조가 돼요.

## 다 죽으면 무슨 일이?

**정말 3벌 다 유실된 파티션이 부활 못 하면**:

- 그 파티션에 있던 메시지는 **영구 유실**
- 프로듀서는 해당 파티션에 발행 시도 시 계속 실패
- Consumer 는 `LEADER_NOT_AVAILABLE` 에러 반복

**부활은 하지만 데이터가 갈린 경우 (unclean leader election)**:

```properties
unclean.leader.election.enable=false  # 기본값, 권장
```

이걸 true 로 두면 "sync 가 안 된 오래된 replica 도 leader 로 승격" 하는데, **최신 데이터 유실을 감수하고 가용성 확보** 하는 옵션이에요. 기본값 false 유지가 안전.

## 실무에서 쓰는 대비책 스택

| 계층 | 방어책 | 방어하는 시나리오 |
|------|--------|-----------------|
| 하드웨어 | Rack awareness (AZ 분산) | 데이터센터 장애 |
| 브로커 설정 | `replicas=3`, `min.insync.replicas=2` | 1대 죽어도 계속 서비스 |
| 프로듀서 | `acks=all`, `enable.idempotence=true` | 유실/중복 방지 |
| 컨슈머 | manual commit, idempotent 처리 | 중복 소비 대응 |
| 운영 | `delete.topic.enable=false`, RBAC | 사람 실수 |
| 백업 | MirrorMaker 로 별도 클러스터에 복제 | 클러스터 전멸 |
| 앱 레벨 | Transactional Outbox | Kafka 자체 장애 |

## 이 프로젝트 관점

현재 `replicas=1` 이니까 "레플리카가 다 죽는" 이슈는 아직 걱정 대상이 아니에요. 브로커 1대만 있으니 그게 죽으면 즉시 전멸 상태고, 개발환경이라 감수하는 트레이드오프죠.

운영 배포 시 체크리스트로 만든다면:

1. ✅ `replicas=3` 으로 변경 (KafkaConfig.java:19 주석 반영)
2. ✅ 3개 브로커를 서로 다른 AZ 에 배치
3. ✅ `min.insync.replicas=2` 브로커 설정
4. ✅ 프로듀서 `acks=all`, `enable.idempotence=true`
5. ✅ Transactional Outbox 도입 (Kafka 자체 다운 시 결제-이벤트 정합성 보장)
6. ✅ `unclean.leader.election.enable=false` 유지

여기까지 하면 "한꺼번에 다 죽는" 시나리오 대부분을 방어할 수 있고, 그래도 남는 리스크는 백업 클러스터(MirrorMaker)로 커버합니다.

## 무엇을 의미하나

`acks=all` + `enable.idempotence=true` 조합은 **"프로듀서 관점에서는 안전 설정"** 이에요:

1. 프로듀서가 메시지를 보내면 leader 뿐 아니라 **모든 in-sync 레플리카에 복제될 때까지 대기** 후 성공 응답
2. 재시도가 있어도 중복 저장 방지 (idempotence)
3. 브로커 하나가 죽어도 데이터 유실 위험 최소화

## 그런데 지금 상황에선 이 안전장치가 무력화됨

현재 `replicas=1` 이라서:

```
acks=all → "모든 in-sync replica 확인" → 근데 replica 가 1개뿐 → leader 하나만 확인
```

즉 **replicas=1 인 상태에서는 `acks=all` 이 `acks=1` 과 사실상 동일**합니다. 브로커A 가 응답한 직후 죽으면 다른 브로커에 복제본이 없으니 유실.

**추가로 놓치기 쉬운 함정 — `min.insync.replicas`**

브로커/토픽 레벨 설정인 `min.insync.replicas` 도 기본값이 `1` 입니다. 운영으로 넘어갈 때:

```yaml

→ follower 2개 다 죽어도 leader 혼자 계속 씀 → 나중에 leader 죽으면 유실
```

**세 개가 세트로 움직여야 진짜 안전**:
- `replicas=3` (KafkaConfig 에서 설정)
- `min.insync.replicas=2` (브로커 server.properties 또는 토픽 옵션)
- `acks=all` (프로듀서, 이미 기본값)

## 근거 3가지

**1. `@TransactionalEventListener` 는 기본이 동기**

```java
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void onOrderCreated(OrderCreatedEvent event) { ... }
```

이 리스너는 이벤트를 발행한 **원본 스레드에서 그대로 실행**됩니다. 즉 `PaymentService.pay()` (또는 어디서 이벤트를 발행하는 코드) 의 스레드가 트랜잭션 커밋 후 이 메서드까지 이어서 실행해요. 별도 스레드로 넘기려면 `@Async` 를 붙여야 하는데 없어요.

**2. 두 호출이 순차적으로 나열됨**

```java
boolean publishOk = publishOrderEvent(event);   // ① 완료까지 대기
boolean cartOk = clearCustomerCart(event);      // ② ① 끝난 뒤 시작
```

`publishOk` 를 다음 줄에서 참조하지 않더라도, 자바는 위에서 아래로 순차 실행이라 ① 이 리턴하기 전엔 ② 로 못 넘어갑니다.

**3. Kafka 발행조차 동기로 강제됨 — `.get()` 호출**

여기가 특히 중요한데, `OrderKafkaProducer.publish()` (line 24) 를 보면:

```java
kafkaTemplate.send(TOPIC, String.valueOf(orderId), orderId).get();
//                                                        ^^^^^
```

`kafkaTemplate.send()` 는 원래 `CompletableFuture<SendResult>` 를 반환하는 **비동기 API** 입니다. 그런데 `.get()` 을 붙였어요. 이건 **Future 가 완료될 때까지 현재 스레드를 블록** 시키는 호출이라, Kafka 브로커가 ack 를 돌려줄 때까지 여기서 멈춰요.

정리하면:

```
onOrderCreated 스레드
  │
  ├─ publishOrderEvent
  │    └─ kafkaTemplate.send().get()  ← 브로커 ack 올 때까지 블록 (수십 ms~수초)
  │    └─ return true
  │
  ├─ clearCustomerCart   ← 위가 끝나야 여기 진입
  │    └─ cartService.clearCart()
  │    └─ return true
  │
  └─ log.info(...)
```

## 왜 이렇게 짰나 — 의도된 동기 설계

Kafka 발행을 `acks=all` + `.get()` 조합으로 동기 대기시키는 이유는:

- **발행 실패를 즉시 catch 해서 메트릭 카운트를 올리기 위함** (line 51 `order.event.publish.failure`)
- 비동기로 `.send()` 만 호출하면 실패 콜백을 따로 등록해야 하고, 실패해도 이미 메서드는 리턴한 뒤라 흐름 제어가 복잡해짐
- 클래스 주석("Kafka 발행 실패 시 주문은 PENDING 상태로 DB에 남으므로 메트릭 알림으로 수동 재처리 필요") 이 이 설계를 전제로 함

## 부작용 — 지연시간 누적

동기 처리라서 **결제 API 응답 지연 = DB 커밋 + Kafka 발행 왕복 + 카트 삭제**가 다 합쳐집니다:

```

├─ 결제 처리 (DB)                     ~50ms
├─ 트랜잭션 커밋

│    └─ Redis 카트 삭제               ~5ms
└─ HTTP 200 반환                     총 ~65~155ms
```

Kafka 브로커가 느려지면 결제 API 응답도 같이 느려져요. 이걸 개선하려면:

**옵션 1: `@Async` 로 완전 비동기화**

```java
@Async
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void onOrderCreated(OrderCreatedEvent event) { ... }
```

응답은 빨라지지만 실패 시 사용자에게 알릴 방법이 없어짐.

**옵션 2: `publishOrderEvent` 와 `clearCustomerCart` 를 병렬 실행**

```java
CompletableFuture<Boolean> pubFuture = CompletableFuture.supplyAsync(() -> publishOrderEvent(event));
CompletableFuture<Boolean> cartFuture = CompletableFuture.supplyAsync(() -> clearCustomerCart(event));
CompletableFuture.allOf(pubFuture, cartFuture).join();
```

두 작업이 독립적이라 병렬화 가능. 지연시간이 `max(pub, cart)` 로 줄어듦.

**옵션 3: Kafka 만 fire-and-forget (`.get()` 제거) + 실패 콜백**

```java
kafkaTemplate.send(TOPIC, key, orderId)
    .whenComplete((result, ex) -> {
        if (ex != null) meterRegistry.counter("order.event.publish.failure").increment();
    });
```

응답은 빨라지지만 "발행 실패했는데 응답은 이미 200" 상황이 생김. Transactional Outbox 같은 별도 안전장치 필요.

## 실행 순서 상세

`@Transactional` 은 프록시로 감싸져 있어서, 메서드가 return 해도 **호출자에게 바로 안 돌아갑니다**. 프록시가 커밋을 마친 뒤에야 호출자에게 값이 전달돼요.

```
Controller.pay() 호출
   │
   ▼
[Spring 프록시] ────────────────────────────────────────┐
   │  ① 트랜잭션 시작 (BEGIN)                          │
   │                                                    │
   │  ② PaymentService.pay() 메서드 본문 실행           │
   │     ├─ paymentRepository.save(...)                 │
   │     ├─ payment.success(txId)                       │
   │     ├─ eventPublisher.publishEvent(...)  ← 이벤트  │
   │     │   (실제 발행 안 됨, TransactionSynchronization│
   │     │    큐에 대기)                                 │
   │     └─ return response;   ← 여기서 return 문 실행  │
   │        (아직 호출자에게 안 감!)                    │
   │                                                    │
   │  ③ 트랜잭션 커밋 (COMMIT)                          │
   │                                                    │
   │  ④ AFTER_COMMIT 리스너 실행 ← 여기서 onOrderCreated│
   │     ├─ publishOrderEvent (Kafka .send().get())     │
   │     └─ clearCustomerCart (Redis)                   │
   │                                                    │
   │  ⑤ 프록시가 response 를 호출자에게 리턴           │
   └────────────────────────────────────────────────────┘
   │
   ▼
Controller 가 PaymentResponse 받음
```

## 왜 이렇게 동작하나 — 프록시의 마법

`@Transactional` 은 컴파일러 마법이 아니라 **런타임 프록시** 로 구현됩니다. 실제로는 이런 코드가 자동 생성돼요 (개념적으로):

```java
public PaymentResponse pay(Long memberId, PaymentRequest request) {
    TransactionStatus tx = transactionManager.begin();
    try {
        PaymentResponse result = realPaymentService.pay(memberId, request);  // ← 원본 메서드
        transactionManager.commit(tx);        // ← 커밋
        // 커밋 후 등록된 TransactionSynchronization 콜백들이 여기서 실행됨
        // → AFTER_COMMIT 리스너들이 이 시점에 순차 실행
        return result;                        // ← 호출자에게 리턴
    } catch (Exception e) {
        transactionManager.rollback(tx);
        throw e;
    }
}
```

즉 원본 `pay()` 의 `return response` 는 **프록시의 로컬 변수에 값을 넘겨준 것**일 뿐이고, 호출자(Controller)에게 도달하는 건 프록시가 커밋과 후처리를 모두 마친 다음입니다.

트랜잭션 커밋이 성공하면 → 예약된 콜백이 발화 → 리스너 메서드가 실행되는 구조예요. 커밋 실패(예외) 시엔 콜백이 아예 실행되지 않고 이벤트도 조용히 버려집니다.

## 이 프로젝트에서 뭘 뜻하나


**즉 클라이언트가 "결제 성공" 응답을 받는 시점엔 이미**:

- ✅ Kafka `order-created` 발행 완료 (또는 실패해서 메트릭에 카운트됨)
- ✅ 장바구니 삭제 완료 (또는 실패)

**중요한 함의 3가지**

1. **응답 지연이 누적됨** — Kafka 브로커가 느려지면 결제 API 응답도 느려짐. 결제 자체는 이미 커밋됐는데도 사용자는 대기
2. **리스너에서 예외가 나도 결제는 취소 안 됨** — 이미 커밋된 뒤라 롤백 불가능. 그래서 리스너 안에서 try-catch 로 예외를 삼키고 메트릭만 기록하는 설계 (OrderEventListener.java:47-53)
3. **Kafka 발행 실패 = 유령 주문 가능성** — DB에 주문은 있는데 Consumer 가 처리 안 함. line 48 주석의 "수동 재처리 필요" 가 이 케이스


- 커밋 직후 리스너를 **별도 스레드에 던져놓고** 프록시가 즉시 리턴
- 응답 시간 = ~55ms (Kafka/카트 시간 빠짐)
- 대신 사용자에게 리턴한 후에 Kafka 발행 실패 시 알릴 방법이 사라짐

지금 프로젝트는 **동기 선택** = "응답이 조금 느려져도, 결제 응답을 받은 사용자는 이후 처리가 확정됐다고 신뢰할 수 있다" 는 트레이드오프예요.

## 짚고 갈 미묘한 지점 2가지

**① `pay() 커밋 → OrderCreatedEvent` 순서가 아니라 "이벤트 발행 예약 → 커밋 → 리스너 실행"**

사용자가 말한 "pay() 커밋 → OrderCreatedEvent" 는 결과적으로 맞지만 정확히는:

```
pay() 본문에서 publishEvent(...) 호출   ← 이 시점엔 아무것도 안 일어남
      ↓
프록시가 커밋
      ↓
Spring 이 등록해 뒀던 콜백을 실행 → 리스너 메서드 호출
```

즉 `OrderCreatedEvent` 는 커밋 *전*에 이미 만들어져 있고, 커밋 성공을 트리거로 리스너가 발화하는 구조예요. 커밋이 실패하면 이벤트는 그냥 버려지고 리스너는 실행되지 않습니다 (PaymentService.java:199-200 주석의 "커밋 실패 시 메시지 미발행" 이 이 뜻).

**② 컨슈머는 완전히 다른 스레드에서 비동기 실행**

`OrderEventListener` 까지는 원래 pay() 호출한 HTTP 스레드에서 순차 실행이지만, **`OrderKafkaConsumer.consume()` 은 완전히 별개 스레드**에서 돌아요:

- `publish()` = 브로커에 "이 메시지 저장해줘" 요청 후 ack 받고 끝. 컨슈머 호출 안 함
- `consume()` = Spring Kafka 가 백그라운드에서 별도 스레드로 브로커를 폴링하다가 새 메시지가 오면 호출

그래서 HTTP 응답이 나간 뒤 **밀리초~수 초 후에 컨슈머가 실행**됩니다. Consumer 처리 시간은 결제 API 응답 시간에 포함되지 않아요.

> pay() 커밋 → OrderCreatedEvent → OrderEventListener 가 publish → OrderKafkaConsumer 가 확인 후 처리

**1. `notify(orderId, ...)` — 개인 채널 (주문한 사람만)**

- "당신의 주문이 접수/실패됐다" 는 메시지

**2. `broadcastNewOrder(...)` — 브로드캐스트 (매장 대시보드)**

## 왜 이렇게 설계했나 — 두 단계 알림의 의미

**단계 1: HTTP 응답 (T=110ms)**
- "결제 성공" — 돈이 나갔음을 확인
- 이 시점엔 재고 차감이 아직 안 됨. 순수 결제 완료만 보장

**단계 2: SSE 알림 (T=155ms 이후, 언제 올지 모름)**
- "주문 접수 완료" — 재고까지 확보돼서 실제 매장 접수됨
- 또는 "주문 실패" — 결제는 됐지만 재고 부족으로 취소됨 (환불 처리됨)

이렇게 나눈 이유는 **결제 응답을 빨리 주기 위해서**예요. 만약 재고 차감까지 다 하고 응답하면:
- 사용자는 결제 버튼 누르고 몇 초 동안 로딩 화면
- 재고 락 경합이 심하면 타임아웃 발생 가능
- 결제 API 자체의 처리량 저하

Kafka 로 분리함으로써:
- 결제 API 는 최대한 빠르게 응답
- 실제 주문 처리는 백그라운드에서 대기열 형태로 순차 처리
- 사용자는 결제 완료 화면을 본 뒤 "주문 접수 중..." → "접수 완료!" 알림을 SSE 로 받음

## 왜 이렇게 나눴나 — 락 분리를 위한 설계

핵심은 **`deductStock` 이 메뉴 단위로 분산 락을 잡는다** 는 점이에요:

```java
// StockLockFacade.java:22
RLock lock = redissonClient.getLock("stock:lock:" + storeId + ":" + menuId);
```

락 키가 `stock:lock:{storeId}:{menuId}` 라서:
- **매장별 + 메뉴별로 독립적인 락**
- 아메리카노 락과 라떼 락은 서로 다름 → 동시에 잡아도 무관

만약 "주문 전체를 한 번에 차감" 하는 API 였다면 락 하나로 여러 메뉴를 커버해야 해서 **락 범위가 넓어지고 경합이 심해집니다**. 예: 매장 전체 락 하나로 잡으면 A 손님의 아메리카노 주문이 B 손님의 라떼 주문을 막게 돼요.

메뉴별로 나눠 잡으면:
- 아메리카노 주문 A → `stock:lock:1:아메리카노` 락
- 라떼 주문 B → `stock:lock:1:라떼` 락
- **완전히 병렬 처리 가능**

## 순차 호출의 함정 — 데드락 가능성

여러 메뉴를 순차로 락 잡을 때 조심할 점이 있어요:

**시나리오**: 손님 A 가 [아메리카노, 라떼] 주문, 손님 B 가 [라떼, 아메리카노] 주문 (동시)

```
스레드 A: 아메리카노 락 획득 → 라떼 락 대기
스레드 B: 라떼 락 획득 → 아메리카노 락 대기
                              ↓
                          데드락!
```

이 프로젝트는 어떻게 방어하고 있는지 확인해 보면:

```java
if (!lock.tryLock(5, TimeUnit.SECONDS)) {
    throw new StockLockException(...);
}
```

**`tryLock(5초)`** 를 씀. 5초 안에 획득 못 하면 `StockLockException` 을 던지고 롤백 → 데드락이 무한 대기로 이어지지 않고 예외로 빠져나옴. 그리고 `StockLockException` 은 **재시도 가능한 예외** 로 분류돼 있어서 (KafkaConfig.java 의 `addNotRetryableExceptions` 목록에 없음), Kafka 가 자동으로 다시 시도합니다.

더 근본적으로 방지하려면 **락 순서를 고정** (예: menuId 오름차순으로 정렬 후 획득) 하는 방법도 있는데, 이 프로젝트는 그 단계까지는 안 갔고 tryLock 타임아웃으로만 방어하는 상태예요.

## 두 층이 뭔지

**층 1 — Redisson 분산 락 (앱 ↔ Redis)**


**층 2 — @Version 낙관적 락 (JPA ↔ DB)**

- 재고 UPDATE 시 자동으로 `WHERE version = ?` 조건이 붙음
- 다른 트랜잭션이 먼저 수정했으면 rowCount=0 → `OptimisticLockException` 발생

## 왜 두 층이 필요한가 — 각자의 실패 모드

**Redisson 락만으로는 부족한 케이스**:

Redis 분산 락은 몇 가지 상황에서 **락이 새거나 두 프로세스가 동시에 락을 가진 것처럼 보일 수 있어요**:

1. **watchdog 갱신 실패** — 네트워크 순단으로 30초 안에 갱신 못 하면 만료. 원 소유자는 락 있다고 착각, 다른 프로세스는 새로 잡음
2. **GC pause / STW** — Java 프로세스가 수 초 멈춘 사이 락 만료 → 두 프로세스가 동시에 소유
3. **Redis 마스터 페일오버** — 마스터가 락 정보를 슬레이브에 복제하기 전에 죽으면 새 마스터는 락 없다고 판단 → 다른 프로세스가 획득

이런 극단 케이스에서 **두 프로세스가 동시에 재고 차감을 시도** 하면 lost update 발생 가능. 예:

```
프로세스 A: quantity 읽음 = 10 → 3 차감 → 7 저장 시도
프로세스 B: quantity 읽음 = 10 → 5 차감 → 5 저장 시도

DB 최종 상태: 5 또는 7 (마지막 UPDATE 만 반영) — 원래는 2가 되어야 함!
```

**@Version 이 이걸 잡아줌**:

```sql
-- 프로세스 A 의 UPDATE
UPDATE stock SET quantity = 7, version = 2 WHERE stock_id = 1 AND version = 1;
-- 성공 (rowCount = 1)

-- 프로세스 B 의 UPDATE (A 가 먼저 했으므로)
UPDATE stock SET quantity = 5, version = 2 WHERE stock_id = 1 AND version = 1;
-- 실패 (rowCount = 0, version 이 이미 2)
-- → OptimisticLockException 발생 → 트랜잭션 롤백
```

프로세스 B 의 차감이 무효화되고 예외가 던져지므로, 잘못된 값이 저장되지 않습니다.

```java
// StockService.java:92

// Stock.java:47-53
```

- 트랜잭션 격리로 dirty read 방지
- 엔티티가 스스로 "재고 부족 상태에서 차감 시도" 를 거부 (도메인 규칙)

이건 방어라기보단 **비즈니스 로직** 이라 주석에선 별개로 취급 안 한 듯해요.

| 층 | 방어 대상 | 어디서 |
|----|---------|-------|
| 1 | 동시 요청 직렬화 (같은 매장·메뉴) | Redisson 분산 락 |

## 왜 이 이중 구조가 필요한지 — 한 문장으로

**Redisson 락은 "정상 상황에서 빠르게" 동시성 제어, @Version 은 "락이 실패한 극단 상황에서도 데이터 정합성" 보장** — 서로 다른 실패 모드를 커버하는 조합이라 두 개를 다 두는 거예요. 하나만 있으면 성능이든 정확성이든 어느 쪽이 무너집니다.

## 멱등성 서비스의 방어 범위 — HTTP 진입점만

`IdempotencyService` 는 **결제 요청이 API 로 들어오는 순간** 만 보호합니다:

```
[클라이언트]              [PaymentController]           [IdempotencyService]
    │                            │                            │
    ├─ 결제 요청 (같은 카트) ──> │                            │
    │                            ├─ markProcessing(key) ──> │ setIfAbsent → true
    │                            │                            │ (Redis에 표시)
    │                            ├─ pay() 실행                │
    │                            │                            │
[클라이언트 네트워크 오류]        │                            │
    ├─ 재전송 (동일 키) ────────> │                            │
    │                            ├─ markProcessing(key) ──> │ setIfAbsent → false
    │                            │                            │ (이미 있음)
    │                            ├─ getResult(key) ──────> │ 저장된 결과 반환
    │                            │                            │
    │ <── 캐시된 응답 반환 ────── │                            │
```

이건 **"같은 사용자가 결제 버튼을 두 번 눌러도 하나만 처리"** 를 보장하는 거예요. **결제 API 앞단에서만 작동**.

## 그런데 `deductStock` 은 API 앞단이 아님

`deductStock` 이 실제로 호출되는 경로:

```
결제 API (멱등성 보호 O)
    ↓ 커밋
Kafka 발행
    ↓
[Kafka 브로커] ← 멱등성 보호 X, 여기부터는 IdempotencyService 관여 안 함
    ↓
OrderKafkaConsumer.consume()
    ↓
OrderProcessingService.process()
    ↓
deductStock() ← 이 시점엔 멱등성 키가 이미 만료됐거나 관계 없음
```

즉 **Kafka 발행 이후는 IdempotencyService 의 방어 범위 밖** 이에요.

## `deductStock` 이 중복 호출될 수 있는 3가지 시나리오

**시나리오 1: Kafka At-Least-Once 재전달 (가장 흔함)**

Kafka 컨슈머는 **최소 한 번(at-least-once) 전달** 을 보장해요. 이 말은 **한 번 이상**, 즉 **여러 번도 가능** 하다는 뜻입니다.

```
컨슈머가 orderId=42 메시지 수신
    ↓
process(42) 실행 → 재고 차감 완료
    ↓
컨슈머가 offset 커밋하려는 순간 JVM 다운! ⚡
    ↓
파티션 리밸런싱 → 다른 컨슈머가 인수
    ↓
새 컨슈머가 offset(커밋 안 됨)부터 다시 poll → orderId=42 다시 받음
    ↓
process(42) 다시 실행 → 재고 차감 또 시도?!
```

**시나리오 2: DefaultErrorHandler 재시도**

`KafkaConfig.java:45` 의 `FixedBackOff(1000L, 3)` — 3회 재시도. **부분 성공** 상황에서 위험:

```
process(42) 실행
    ↓ 아메리카노 차감 성공 ✅
    ↓ 라떼 차감 시도 → 순간 DB 커넥션 문제로 예외 발생 ❌
    ↓
DefaultErrorHandler 가 예외 캐치 → 1초 뒤 재시도
    ↓
process(42) 다시 실행 → 아메리카노 또 차감?!
```

**시나리오 3: 프로듀서 재시도로 인한 메시지 중복**

`kafka-clients 3.0+` 는 `enable.idempotence=true` 가 기본이라 프로듀서 재시도 시 중복이 방지되지만, 이 설정이 꺼져 있으면 같은 메시지가 브로커에 2번 저장될 수 있어요. 이 프로젝트는 기본값이라 방어됨.

## 그럼 진짜로 중복 차감이 되는가? — 아니요, 별도 방어책이 있음

`OrderProcessingService.java:68` — **바로 이 한 줄** 이 핵심 방어책이에요:

```java
private void processOrder(Long orderId, Orders order) {
    // Kafka at-least-once 재전달 방어 — 이미 차감된 주문은 무시
    if (order.isStockDeducted() || order.getStatus() == OrderStatus.CANCELLED) return;
    ...
}
```

**`Orders.isStockDeducted()`** 라는 boolean 플래그를 확인해서:
- 이미 재고 차감이 완료된 주문이면 즉시 return
- 취소된 주문이면 즉시 return

그리고 차감 완료 시 이 플래그를 세팅:

```java
// OrderProcessingService.java:81-82
order.markStockDeducted();
orderRepository.save(order);
```

## 3중 방어 구조 완성

정리하면 **`deductStock` 이 실제로 중복 실행되지 않도록 하는 방어책이 3층으로 쌓여 있어요**:

| 층 | 방어 대상 | 어디서 |
|----|---------|-------|
| ① IdempotencyService | 결제 API 중복 호출 (네트워크 재시도) | PaymentService.pay() 진입 |
| ② `isStockDeducted` 플래그 | Kafka 메시지 중복 전달 | OrderProcessingService.process() 진입 |
| ③ @Version + 분산 락 | 극단 경합 시 lost update | StockLockFacade + Stock 엔티티 |

**각 층이 방어하는 위치가 다름**:
- ① 은 **HTTP 앞단** (같은 사람이 2번 결제)
- ② 는 **Kafka 컨슈머 앞단** (Kafka 가 같은 메시지 2번 전달)
- ③ 은 **DB 앞단** (여러 프로세스가 같은 재고 동시 수정)

## 시나리오 시뮬레이션으로 검증

**시뮬레이션 1**: 같은 카트로 결제 버튼 두 번 누름
```
1차 요청 → IdempotencyService.markProcessing → 처리 → Kafka 발행
2차 요청 → IdempotencyService.markProcessing → false → 캐시된 응답 반환
       → Kafka 발행 X → process() 호출 X → deductStock X
```
✅ 층 ① 이 방어

**시뮬레이션 2**: Kafka 컨슈머가 처리 후 커밋 전에 죽음
```
컨슈머 A: process(42) 실행 → deductStock 완료 → isStockDeducted=true 저장
       → offset 커밋 전 다운 ⚡
컨슈머 B: 같은 orderId=42 재수신 → process(42) 진입
       → order.isStockDeducted() = true → 즉시 return
       → deductStock 호출 안 됨
       → offset 커밋 성공
```
✅ 층 ② 가 방어

**시뮬레이션 3**: 두 프로세스가 Redisson 락 페일오버 상황에서 동시 차감 시도
```
프로세스 A: 락 획득 → quantity=10 읽음 → 차감 → version 1→2
프로세스 B: 락 획득 (페일오버로 새는 상황) → quantity=10 읽음 → 차감 시도
       → UPDATE ... WHERE version=1 → rowCount=0
       → OptimisticLockException
```
✅ 층 ③ 이 방어

## 사용자 질문에 대한 정확한 답

**Q**: 결제 앞단에서 멱등성 처리를 하는데 `deductStock` 이 중복 실행될 수 있나요?

**A**: **`deductStock` 은 실제로 중복 실행되지 않지만, 그 이유는 IdempotencyService 때문이 아닙니다**.

- IdempotencyService 는 **결제 API 앞단** 만 보호 → 커밋 이후 파이프라인은 관여 안 함
- Kafka 는 at-least-once 라 컨슈머 재수신 가능 → 이걸 방어하는 건 별도 층인 **`Orders.isStockDeducted()` 플래그**
- 그러니까 "IdempotencyService 가 있어서 안전" 이 아니라, **각 계층마다 자기 층에 맞는 멱등성 장치를 따로 두고 있어서 안전** 한 거예요

## 왜 이렇게 분리했나 — 계층별 특성이 다름

한 곳에 몰아 넣지 않은 이유:

- **HTTP 요청 중복** — 네트워크 재시도, 사용자 더블 클릭. Redis TTL 기반이 자연스러움
- **Kafka 메시지 중복** — offset 관리, 리밸런싱. DB 의 영속 플래그가 자연스러움 (Redis TTL 로 하면 만료 시 방어 뚫림)
- **DB 동시 수정** — 락 페일오버, GC pause. `@Version` 이 자연스러움

각 계층의 실패 모드가 다르니 방어 도구도 달라야 합니다. 이걸 **defense in depth** (심층 방어) 라고 부르는 패턴이고, 이 프로젝트가 잘 지키고 있는 셈이에요.

**User (14:25):**

주문 pay() 시 redisson 으로만 멱등 처리 하는거 안전한걸까?

**Assistant (14:25):**

*[도구 호출: Read]*

**Assistant (14:27):**

결론부터 말하면 **개발/데모 환경에선 충분하지만, 실제 결제 서비스로 가면 부족합니다**. 특히 **돈이 걸린 결제** 라 Redis 단일 방어로는 몇 가지 시나리오에서 뚫려요.

## 현재 구현 분석

**멱등성 키 생성 (PaymentService.java:242-256)**:
```java
raw = memberId + storeId + method + orderType + customerRequest + toJson(cart)
key = SHA-256(raw)
```

**흐름**:
1. `markProcessing(key)` — Redis `SET NX EX 60s` (원자적)
2. 성공 시 결제 처리 → `saveResult(key, response)` (TTL 60초 갱신)
3. 실패 시 `delete(key)` — 재시도 허용

**설정**:
- PROCESSING_TTL = 60초
- RESULT_TTL = 60초

## 뚫리는 시나리오 6가지

**시나리오 1: Redis 마스터-슬레이브 페일오버 (가장 유명한 함정)**

이게 Martin Kleppmann 이 Redis Distributed Lock 을 비판한 그 유명한 이슈예요:

```
T=0    요청 A: markProcessing → 마스터에 키 저장 → true 반환
T=1ms  마스터 다운 (슬레이브에 아직 복제 전)
T=2ms  슬레이브가 마스터로 승격 (키 없음)
T=3ms  요청 A' (네트워크 재시도): markProcessing → 새 마스터엔 키 없음 → true 반환
       → 중복 결제 성사 💥
```

Redis 는 **비동기 복제** 라서 마스터→슬레이브 복제 사이 갭이 항상 존재. RedLock 알고리즘도 이 문제를 100% 해결 못 함.

**시나리오 2: PROCESSING_TTL 60초 만료**

```
T=0     markProcessing → true, 결제 시작
T=1s    PG 게이트웨이 호출 → 갑자기 느려짐 (PG 사 장애)
T=61s   TTL 만료 → 키 삭제
T=62s   사용자가 답답해서 새로고침 → 재요청
        markProcessing → true (키 없음) → 새 결제 시작
T=90s   원본 결제 응답 도착 → 결제 완료 처리
T=91s   재시도 결제도 완료 처리
        → 사용자 카드에 2번 청구 💥
```

실제 PG 사(토스, 카카오페이 등)는 대개 30초 이내에 응답하지만, 장애 시엔 몇 분씩 걸리기도 해요.

**시나리오 3: JVM 프로세스 크래시**

`PaymentService.java:232-238` 을 보면 catch 로 `delete(key)` 를 하는데, **크래시 시엔 catch 자체가 실행 안 됨**:

```
T=0    markProcessing → true
T=1s   PG 호출 성공 → 실제 결제 완료됨 (외부 시스템에 트랜잭션 존재)
T=2s   DB 커밋 직전 JVM 크래시 ⚡
       → catch 블록 실행 안 됨 → 키가 PROCESSING 상태로 남음
       → DB엔 결제 기록 없음
T=3s   로드밸런서가 다른 인스턴스로 요청 라우팅
       → 사용자 재시도 → markProcessing → false → DuplicateRequestException
T=63s  PROCESSING TTL 만료 → 재시도 가능
       → PG 상태 확인 없이 새 결제 진행 → 중복 결제 💥
```

**"결제는 됐는데 우리 DB 엔 없음"** 상태 = 회계 불일치 발생.

**시나리오 4: saveResult 실패**

```java
// line 210
PaymentResponse response = PaymentResponse.from(payment, order, 0);
idempotencyService.saveResult(key, response);  // ← Redis 순단으로 실패
return response;
```

DB 커밋은 이미 성공했지만 캐시 저장 실패. `saveResult` 가 예외를 던지면:
- catch 로 들어감 → `delete(key)` 실행 → 재시도 허용
- 재시도 시 새 결제 처리 → **중복 결제** 💥

`saveResult` 에 재시도 로직이 없고 예외 처리도 명시적이지 않아서 위험한 구간이에요.

**시나리오 5: 카트 내용의 미묘한 차이**

멱등성 키에 `toJson(cart)` 가 들어감. 카트 JSON 이 조금이라도 다르면 다른 키:

```
사용자가 결제 버튼 클릭 → 요청 A 전송 (카트: [아메리카노 1])
서버 응답 대기 중 사용자 짜증나서 카트 새로고침 (백그라운드로 재조회)
→ 이 과정에서 카트 items 순서, JSON 필드 순서가 미세하게 다르게 직렬화될 여지
→ 다시 결제 버튼 → 요청 B (카트: [아메리카노 1] 이지만 JSON 문자열이 다름)
→ 다른 키 → markProcessing → true → 중복 결제 💥
```

**JSON 직렬화의 결정성(determinism)** 이 보장돼야 하는데, Jackson 은 필드 순서, null 처리, 소수점 표기 등이 컨텍스트에 따라 다를 수 있어요.

**시나리오 6: 여러 인스턴스 + Redis 클러스터 스플릿브레인**

Redis 클러스터를 여러 노드로 운영하는데 네트워크 파티션이 발생하면:

```
클러스터 노드 A (파티션 1): 키 존재
클러스터 노드 B (파티션 2): 키 없음
로드밸런서가 인스턴스 A → 노드 A, 인스턴스 B → 노드 B 로 붙임
→ 두 요청 모두 markProcessing 성공 → 중복 결제 💥
```

## 실무에서 하는 방어 — Defense in Depth

돈이 걸린 결제는 **Redis 하나에 의존하지 않고 다중 방어** 를 씁니다:

**층 1: Redis 멱등성 (현재 있음)** — 정상 케이스 빠른 필터
- 장점: 빠름, 대부분의 재시도 방어
- 단점: 위 시나리오들

**층 2: DB UNIQUE 제약 (추가 권장)**

`payment` 테이블 또는 별도 `idempotency_log` 테이블에 유니크 인덱스:

```sql
CREATE TABLE payment_idempotency (
    key VARCHAR(64) PRIMARY KEY,     -- SHA-256 키
    payment_id BIGINT,
    created_at TIMESTAMP,
    ...
);

-- 결제 처리 시
INSERT INTO payment_idempotency (key, payment_id) VALUES (?, ?)
-- 중복이면 UNIQUE 위반 → 이전 결제 조회해서 반환
```

- Redis 가 뚫려도 DB 레벨에서 최종 차단
- DB 트랜잭션과 원자적으로 처리 가능

**층 3: PG 사의 트랜잭션 ID 활용**

실제 PG 사(토스, 카카오페이) 는 자체 멱등성 지원:

```java
kakaopayClient.approve(
    orderId,              // 우리 시스템의 주문 ID
    idempotencyKey        // 이걸 헤더로 넘기면 PG 사에서 방어
);
```

- 우리 서버가 크래시해도 PG 사에서 중복 승인 차단
- 대부분의 PG 가 이 헤더 지원 (Idempotency-Key)

**층 4: PG 상태 확인 로직 (재시도 전)**

재시도 시 PG 에 "이 주문 ID 로 이미 승인된 게 있는지" 조회 후 진행:

```java
if (redisMarkerExpired) {
    // 재시도 전 PG 상태 확인
    PgTransaction existing = pgClient.query(orderId);
    if (existing != null && existing.isApproved()) {
        // 이미 결제됨 → 우리 DB 만 복구
        return recoverFromPgState(existing);
    }
    // 정말 결제 안 됐으니 진행
    proceedPayment();
}
```

**층 5: Outbox 패턴**

DB 트랜잭션 안에서 "결제 승인 요청" 을 outbox 테이블에 저장 → 별도 워커가 PG 호출:

```
[결제 요청]
  ↓
INSERT payment (status=PENDING) + INSERT outbox (action=APPROVE)
  ↓ COMMIT
[별도 워커 프로세스]
  ↓
outbox 읽음 → PG 호출 → 결과에 따라 payment.status 업데이트 + outbox 삭제
```

- 이러면 크래시가 나도 outbox 에 남아있어서 재시도 가능
- 결제 승인이 정확히 1번 실행됨

## 이 프로젝트는 왜 층 1만 있어도 괜찮은가

**Mock PG 라서 진짜 결제가 안 일어남**:

```java
// PaymentService.java:189
boolean success = ThreadLocalRandom.current().nextInt(100) < 90;
```

실제 PG 호출이 없으니:
- 중복 결제 = 실제 돈 이중 청구 X (그냥 DB에 결제 로그 2개 생김)
- 시나리오 3(크래시 후 PG 상태 불일치) 발생 안 함
- 시나리오 4(saveResult 실패) 도 데이터 불일치 정도로 끝

**데모/포트폴리오 관점**:
- 동시성 제어 아이디어 시연 → 층 1만으로도 충분히 보여줌
- 코드 복잡도 낮게 유지
- Kafka + SSE + Redis + JPA 조합 자체가 이미 풍부한 학습 소재

## 실제 결제 서비스라면 필수 개선

우선순위 순:

**1. PG 사의 Idempotency-Key 헤더 사용** ⭐⭐⭐
- 가장 근본적 방어. 코드 몇 줄만 바꾸면 됨
- 우리 시스템이 아무리 꼬여도 PG 가 방어

**2. DB UNIQUE 제약으로 최종 차단** ⭐⭐⭐
- 별도 테이블에 idempotency_key 유니크
- Redis 뚫려도 DB 가 잡음

**3. 결제 전 PG 상태 확인 API 호출** ⭐⭐
- 재시도 시 이미 승인됐는지 확인
- "우리 DB 없음 + PG 승인됨" 케이스 복구 가능

**4. Outbox 패턴 도입** ⭐⭐
- 크래시 복구를 위해 필요
- 복잡도 증가하지만 대규모 서비스는 필수

**5. TTL 조정** ⭐
- PROCESSING_TTL 을 PG 최대 응답 시간의 3배 이상 (예: 5분)
- 짧게 두면 만료 후 재시도 위험, 길게 두면 데드락 후 대기 시간 증가 — 트레이드오프

**6. Redis 신뢰성 강화** ⭐
- Redis Sentinel/Cluster + AOF fsync=always
- 근데 이래도 페일오버 갭은 여전히 존재 → 다른 층 필요

## 정리

**질문 답**: **"안전한가?"** 에 대한 답은 상황에 따라 달라요.

| 환경 | 안전성 | 이유 |
|-----|-------|------|
| 개발/데모/포트폴리오 | ✅ 충분 | Mock PG, 트래픽 낮음, 데이터 손실 감수 가능 |
| 소규모 실서비스 (일 매출 백만원 이하) | ⚠️ 부족 | PG 사 헤더 + DB UNIQUE 정도는 필수 |
| 중대형 결제 시스템 | ❌ 부족 | 전체 Defense in Depth 필요 |

**핵심 인사이트**: Redis 멱등성은 **"실수/재시도를 필터링하는 첫 관문"** 으로는 훌륭하지만, **"돈 손실을 100% 막는 최후 방어선"** 으로는 부적합합니다. 결제 도메인에서는 항상 **여러 계층을 겹쳐 두고 각 계층이 서로 다른 실패 모드를 커버** 하는 게 원칙이에요.

이 프로젝트가 포트폴리오/학습 목적이라면 지금 구조로도 훌륭하고, 실서비스 이관 계획이 있다면 최소 **PG 사 Idempotency-Key + DB UNIQUE 제약** 두 개는 추가하는 걸 권장합니다.

## 언급 1: **Kafka 발행 실패 대비** (브로커 다운 관련 얘기 중)

`PaymentService.pay()` 흐름에서 이런 문제를 지적했었어요:

```
1. paymentRepository.save(payment);         // DB 커밋됨
2. kafkaTemplate.send("order-created", ...); // Kafka 발행 실패
   → 결제는 됐는데 재고는 안 빠지고 매장에도 안 보임
```

이걸 방지하는 표준 패턴으로 **Transactional Outbox** 를 제안했어요:

```
1. 트랜잭션 안에서:
   - payment 테이블에 INSERT
   - outbox 테이블에 "카프카로 보낼 메시지" INSERT   ← 같은 DB 트랜잭션
2. 별도 스케줄러가 outbox 를 읽어서 카프카로 발행
3. 발행 성공 시 outbox row 삭제/마킹
```

**해결하는 문제**: Kafka 자체가 다운돼도 outbox 에 남아있으니 나중에 재발행 가능 → DB 와 Kafka 사이의 일관성 확보.

또한 "실무에서 쓰는 대비책 스택" 표에서:

| 계층 | 방어책 | 방어하는 시나리오 |
|------|--------|-----------------|
| **앱 레벨** | **Transactional Outbox** | **Kafka 자체 장애** |

## 언급 2: **결제 멱등성 강화 (방금 대화)**

방금 답변한 "Redis 만으로 안전한가?" 에서 **층 5** 로 나왔어요:

```
[결제 요청]
  ↓
INSERT payment (status=PENDING) + INSERT outbox (action=APPROVE)
  ↓ COMMIT
[별도 워커 프로세스]
  ↓
outbox 읽음 → PG 호출 → 결과에 따라 payment.status 업데이트 + outbox 삭제
```

**해결하는 문제**: JVM 크래시 시나리오. PG 호출 중에 앱이 죽어도 outbox 에 남아있으니 워커가 이어서 처리 → **결제 승인이 정확히 1번 실행됨**.

## 두 용도의 공통점

두 경우 모두 **"DB 커밋과 외부 시스템 호출을 원자적으로 묶을 수 없다"** 는 근본 문제를 푸는 패턴이에요:

| 문제 상황 | Outbox 가 저장하는 것 | 워커가 하는 일 |
|---------|-------------------|-------------|
| Kafka 발행 실패 | "이 이벤트를 Kafka 로 보내야 함" | Kafka publish |
| PG 호출 실패 | "이 결제를 PG 로 승인 요청해야 함" | PG API 호출 |

**핵심 아이디어**: 외부 시스템 호출을 **DB 트랜잭션 안에서 "예약" 만 하고**, 실제 호출은 **별도 프로세스가 outbox 를 폴링해서 실행** — 이러면 앱이 어느 시점에 죽어도 outbox 에 흔적이 남아있어서 복구 가능.

## 이 프로젝트에 아직 없음

현재 코드엔 outbox 테이블도, 워커도 없어요. `PaymentService.pay()` 는 DB 커밋 후 `@TransactionalEventListener(AFTER_COMMIT)` → Kafka 발행 직접 호출 구조라, 이 사이 구간에 크래시나 Kafka 장애가 나면 유실 위험이 있는 상태입니다.

**대신 부분적 안전장치가 있어요**:
- `AFTER_COMMIT` 리스너 실패 시 `order.event.publish.failure` 메트릭 카운트 (OrderEventListener.java:51)
- 주석에 "수동 재처리 필요" 로 명시 (line 48)

즉 "완전 자동화된 outbox" 는 아니고 **"실패 감지 후 사람이 개입"** 하는 최소 방어. 실서비스라면 outbox 로 자동화해야 할 지점이에요.

원하시면 이 프로젝트에 outbox 패턴 도입 계획을 스케치해볼 수 있어요 — 어떤 테이블 구조로, 어떤 워커로, 어떻게 브로커 장애를 복구하는지. 필요하면 말씀 주세요.

## 왜 하필 이 지점에 Kafka 를 뒀나 — 6가지 이유

### 1. 재고 차감이 병목 후보 1순위 ⭐ 가장 큰 이유

사이렌 오더 같은 서비스의 트래픽 특성:
- **피크 시간대(출근/점심)** 초당 수백~수천 건 주문
- **인기 메뉴 편중** — "강남역 스타벅스 아메리카노" 같은 특정 조합에 요청 몰림
- 이 조합의 재고에 **분산 락이 걸림 → 요청들이 락 큐에서 대기**

만약 재고 차감을 결제 API 안에서 동기 처리하면:

```
사용자 A 결제 → 아메리카노 락 획득 → 3초 대기 (앞선 100건 처리 중) → 차감 → 응답
                                    ↑ 응답 시간에 3초 대기가 그대로 포함
```

Kafka 로 분리하면:

```
사용자 A 결제 → Kafka 발행 (수 ms) → 즉시 결제 완료 응답
                                       ↓
                    [백그라운드 Consumer 가 순차 처리]
                    파티션이 자연스러운 큐 역할 → 락 경합 없이 순차
```

**응답 시간이 락 대기와 독립됨**. 이게 결정적 차이예요.

### 2. Kafka 파티션 자체가 큐/직렬화 장치

Kafka 는 파티션 안에서 순서 보장이 되고, 하나의 파티션은 Consumer 한 개만 소비. 즉:

```
order-created 토픽 (파티션 3개)
  파티션 0 → Consumer 1이 순차 처리
  파티션 1 → Consumer 2가 순차 처리
  파티션 2 → Consumer 3이 순차 처리
```

**같은 매장/메뉴 주문을 같은 파티션에 몰아넣으면** (파티션 키 활용) 락 경합조차 줄일 수 있음. 이 프로젝트는 orderId 를 키로 쓰지만, 나중에 `storeId + menuId` 로 바꾸면 락이 아예 불필요해질 여지도 있어요.

### 3. Saga 보상 트랜잭션의 자연스러운 진입점

결제가 이미 성공한 상태에서 재고 차감이 실패하면 → **환불** 이 필요. 이게 Saga 보상 패턴인데, Kafka + DLT 조합이 이걸 구현하기에 최적:

```
정상 흐름:
  결제 성공 → order-created 발행 → Consumer 재고 차감 성공 → 완료

실패 흐름:
  결제 성공 → order-created 발행 → Consumer 재고 부족 → 재시도 3회 실패
     → order-created.DLT 로 이동
     → DLT Consumer 가 compensate() 실행 → 결제 취소 + 주문 취소
```

Kafka 의 재시도 + DLT 메커니즘이 **"이미 결제된 주문을 어떻게 안전하게 되돌릴까"** 문제를 인프라 레벨에서 해결해줌. 만약 동기로 짰다면 이 로직을 앱 안에서 다 짜야 하고, 부분 실패/재시도/보상까지 직접 관리해야 함.

### 4. 팬아웃 여지 확보

`order-created` 이벤트는 **하나의 이벤트를 여러 컨슈머가 각자 소비할 수 있는 브로드캐스트 지점** 이에요:

```
order-created 토픽
  ├─ groupId=whale-order            → 재고 차감 (현재)
  ├─ groupId=whale-analytics        → 매출 통계 (미래)
  ├─ groupId=whale-recommendation   → 사용자 추천 학습 (미래)
  ├─ groupId=whale-notification-fcm → 매장 사장 FCM 푸시 (미래)
  └─ groupId=whale-audit            → 감사 로그 (미래)
```

지금은 재고 차감만 붙어있지만, 나중에 어떤 downstream 이 붙어도 **결제 API 는 하나도 안 바뀜**. 이게 이벤트 스트림의 큰 이점.

만약 재고 차감을 결제 안에 넣었다면 나중에 통계/추천 시스템을 붙일 때 결제 코드를 계속 수정해야 함.

### 5. 결제와 재고의 트랜잭션 경계 분리

동기로 하면 트랜잭션 하나가 굉장히 길어져요:

```
[동기 방식]
BEGIN
  결제 DB 저장 (Payment, PaymentHistory)
  주문 DB 저장 (Orders, OrderItem)
  재고 락 획득 (5초 대기 가능)
  재고 차감 (Stock UPDATE)
  주문 상태 업데이트
COMMIT
```

이 긴 트랜잭션 동안:
- DB 커넥션 풀 점유
- 로우 락 유지 → 다른 트랜잭션 대기
- 롤백 발생 시 롤백 비용 증가

Kafka 로 분리하면:

```
[비동기 방식]
BEGIN
  결제/주문 DB 저장만
COMMIT (짧게 끝남)

--- 별도 트랜잭션 ---
BEGIN
  재고 락 + 차감
COMMIT (짧게 끝남)
```

**트랜잭션이 짧아지면 DB 처리량이 올라감**. 이건 성능뿐 아니라 데드락 확률도 낮춥니다.

### 6. 실패의 성격이 다름 → 처리 방식도 달라야

**결제 실패** (PG 오류, 카드 한도 초과 등)
- 사용자에게 즉시 알려야 함
- 재시도해도 대부분 여전히 실패
- → **동기 처리 + 즉시 응답** 이 맞음

**재고 차감 실패** (일시적 락 경합, DB 순단)
- 잠깐 기다렸다 다시 하면 성공할 확률 높음
- 사용자에게 즉시 알릴 필요는 없음 (결제는 이미 됐으니)
- → **비동기 재시도 + 실패 시 보상** 이 맞음

성격이 다른 두 작업을 같은 파이프라인에 두면 각자 최적화가 안 됨. Kafka 경계가 자연스럽게 이 둘을 분리해줍니다.

## 다른 경계를 골랐다면?

**대안 A: 결제 자체도 Kafka 로**

```
사용자 → 결제 요청 발행 (Kafka) → 즉시 응답 (접수됨)
       → Consumer 가 결제 처리 → SSE 로 결과 통지
```

- 장점: 결제 API 응답이 더 빠름 (수 ms)
- 단점: **사용자 경험 저하** — "결제 되긴 한 건가?" 를 SSE 로 확인해야 함. 결제는 사용자가 즉시 결과를 원하는 도메인
- 대부분의 서비스가 결제는 동기 처리하는 이유

**대안 B: 재고 차감도 결제와 함께 동기**

- 장점: 코드 단순, 원자성 자연스러움
- 단점: 위에 나열한 6가지 이유가 다 문제로 돌아옴

**대안 C: 재고 차감을 결제 전에 (선차감)**

- 장점: 결제 실패 시 환불 불필요
- 단점: **재고를 잡아놓고 결제 실패하면 재고 유령 예약 상태** — 복구 로직 필요. 결국 Saga 필요

**현재 방식 (결제 동기 → Kafka → 재고 비동기)**
- 장점: 각 도메인이 자기 성격에 맞게 최적화됨
- 단점: 사용자에게 "결제 완료" 응답 후에 "주문 실패(재고 부족)" 를 알려야 하는 UX 복잡성 → **SSE 로 해결**

## 이 프로젝트가 "결제 → 재고" 경계를 고른 이유 정리

한 문장으로: **"결제 응답 지연을 최소화하면서, 재고 차감의 실패/재시도/보상을 자연스럽게 처리하고, 향후 이벤트 소비자 확장 여지까지 확보하려는 절충안"**.

이 선택의 대가:
- 사용자는 "결제 완료" → 잠깐 후 "주문 접수 완료" 두 단계 UX 를 경험
- 재고 부족으로 인한 주문 취소가 결제 완료 후에 발생할 수 있음
- 이 대가를 **SSE 실시간 푸시 + Saga 보상 트랜잭션** 으로 커버

## 도메인이 달랐다면 다른 선택도 가능

**만약 티켓팅 서비스** (수량 극히 제한, 선착순) 였다면:
- 재고 차감을 **결제 전에** (선점 방식) 하는 게 정답
- 결제 실패 시 재고 반환 로직 필요
- Kafka 경계 위치가 달라짐

**만약 배달 주문 서비스** (재고 개념 없음, 매장 수락 필요) 였다면:
- Kafka 경계가 "주문 생성 → 매장 수락 요청" 사이에 옴
- 재고 차감 로직 자체가 없음

사이렌 오더는 **"재고는 있는데 병목/실패 가능성이 있고, 결제 응답은 빨라야 함"** 이라는 특정 조합이라 이 위치가 최적이었던 거예요.

## 보완할 3가지 세부사항

### 보완 1: 12번 조건은 **두 개의 OR**

```java
if (order.isStockDeducted() || order.getStatus() == OrderStatus.CANCELLED) return;
```

두 경우 모두 skip:
- **`isStockDeducted() == true`** → Kafka at-least-once 재전달 방어 (이미 처리된 주문이 또 왔을 때)
- **`status == CANCELLED`** → 이미 취소된 주문 (환불 처리 중이거나 이전에 취소된 주문이 뒤늦게 도착)

두 번째 조건이 있는 이유: **DLT compensate → cancel 로 상태가 CANCELLED 로 바뀐 뒤에도, 파티션 리밸런싱 등으로 같은 orderId 메시지가 재전달될 수 있음**. 이 경우 차감을 다시 시도하면 이미 취소된 주문에 재고를 차감하는 오류가 나므로 방어.

### 보완 2: 13번 "차감플래그 true" 는 **DB 반영까지 포함**

```java
order.markStockDeducted();   // 메모리상 엔티티 필드 변경
orderRepository.save(order); // DB UPDATE 실행
```

두 줄이 세트예요. **`markStockDeducted()` 만 하고 `save()` 를 안 하면 12번 검사가 무효화**됩니다. JPA 의 dirty checking 으로 트랜잭션 커밋 시 자동 반영되긴 하지만, 명시적으로 저장해서 확실히 반영시킴.

이 시점 이후로:
- 같은 orderId 로 메시지 재전달돼도 12번에서 걸러짐
- Kafka Consumer 가 다음 poll 에서 offset 커밋
- DB 와 Kafka 상태가 일관됨

### 보완 3: 14번 SSE 는 **2가지 종류**

```java
// 개인 채널 — 주문한 사용자에게만
orderSseService.notify(orderId, Map.of(
    "status", "SUCCESS",
    "orderId", orderId,
    "message", "주문이 접수되었습니다. 매장에서 확인 중입니다."
));

// 브로드캐스트 — 매장 대시보드/POS/바리스타 태블릿
orderSseService.broadcastNewOrder(OrderResponse.from(order));
```

**두 종류의 수신자**:
- `notify(orderId, ...)` → **주문한 사용자의 폰** ("당신의 주문이 접수됐다")
- `broadcastNewOrder(...)` → **매장 관리 화면** ("새 주문이 들어왔다, 만들어야 함")

두 SSE 가 하는 역할이 다릅니다. 사용자는 자기 주문만 관심 있고, 매장은 모든 새 주문을 봐야 하니까요.

## 각 층의 진짜 목적

### 층 1 — Redisson 분산 락: **"동시성"** 방어

목적은 **"같은 매장·메뉴에 대한 동시 다발 요청을 하나씩 직렬화"** 하는 것.

Kafka 유실과는 무관해요. 오히려 **Kafka 는 문제를 유발하는 쪽** 입니다:

```
Kafka Consumer 3개가 병렬 처리 중
   Consumer 1 → orderId=100 (강남매장, 아메리카노 1잔)
   Consumer 2 → orderId=101 (강남매장, 아메리카노 2잔)   ← 다른 파티션
   Consumer 3 → orderId=102 (강남매장, 아메리카노 3잔)   ← 또 다른 파티션
   
동시에 process() 실행 → 동시에 deductStock() 호출
   → 같은 stock row 를 3개 스레드가 동시에 UPDATE 시도
   → Lost update 위험!
```

**Kafka 파티션은 "파티션 안의 순서" 만 보장**. 같은 매장/메뉴 주문이 서로 다른 파티션에 흩어지면 병렬 처리되면서 경합이 생깁니다. Redisson 락이 이걸 방어해요:

```java
RLock lock = redissonClient.getLock("stock:lock:" + storeId + ":" + menuId);
```

락 키가 `storeId + menuId` 조합이라 **같은 매장의 같은 메뉴** 는 동시 처리 불가. 각각 락 대기 큐에 서서 순차 처리.

여기에 더해 **여러 인스턴스로 스케일아웃** 하면 JVM 락(synchronized) 은 무용지물이라 반드시 분산 락이 필요.

### 층 2 — tryLock(5초): **"무한 대기 방지"** (장애가 아니라 정상 경합 처리)

```java
if (!lock.tryLock(5, TimeUnit.SECONDS)) {
    throw new StockLockException("재고 처리 중입니다. 잠시 후 다시 시도해주세요.");
}
```

**5초 대기의 실제 상황**:

```
피크 시간 강남매장 아메리카노에 요청 100건 동시 몰림
   Consumer 1: 락 획득 → 100ms 처리 → 해제
   Consumer 2: 대기 → 락 획득 → 100ms 처리 → 해제
   Consumer 3: 대기 → 대기 → 락 획득 → ...
   ...
   Consumer 50: 대기 시간 = 5000ms 초과 → tryLock 실패 → StockLockException
```

즉 **"장애"** 라기보단 **"경합이 극심할 때 무한 대기 대신 예외로 빠져나옴"** 이 목적. 이 예외가 던져지면:

- `StockLockException` 은 `IllegalStateException` 을 상속하지 않도록 별도 클래스로 분리됨
- KafkaConfig 의 `addNotRetryableExceptions` 목록에 없음 → **재시도 대상**
- DefaultErrorHandler 가 1초 뒤 재시도 → 다음 시도 때는 큐가 줄어들어 성공 확률 높음

**"장애 시 5초"** 라기보다 **"경합이 심하면 5초 안에 못 잡을 수도 있는데, 그럼 재시도로 넘기자"** 는 흐름이에요.

### 층 3 — @Version 낙관적 락: **"층 1 이 실패한 극단 케이스"** 방어

Redisson 락이 완벽하다면 @Version 은 필요 없어요. 그런데 다음 3가지 상황에서 **락이 새거나 두 프로세스가 동시에 락을 가진 것처럼 보일 수 있음**:

1. **Redisson watchdog 갱신 실패** — 네트워크 순단으로 30초 안에 락 갱신 실패 → 만료. 원 소유자는 자기가 락 있다고 착각
2. **GC pause / STW** — Java 프로세스가 몇 초 멈춘 사이 락 만료 → 다른 프로세스가 획득
3. **Redis 마스터 페일오버 갭** — 마스터에 락 저장 후 슬레이브 복제 전 다운 → 새 마스터는 락 없다고 판단

이때 두 프로세스가 동시에 재고 UPDATE 를 시도하면:

```sql
-- 프로세스 A
UPDATE stock SET quantity=7, version=2 WHERE stock_id=1 AND version=1;  -- 성공

-- 프로세스 B (거의 동시)
UPDATE stock SET quantity=5, version=2 WHERE stock_id=1 AND version=1;  -- 실패 (version 이미 2)
```

B 의 UPDATE 가 실패해서 `OptimisticLockException` → 트랜잭션 롤백 → 데이터 정합성 유지.

**"마지막 점검"** 이라는 표현은 맞아요. 다만 왜 마지막인지가 중요한데, **"층 1 (Redisson) 이 정상 작동하면 층 3 (@Version) 은 발동될 일이 없음"** 인 관계예요. 이중 방어의 두 번째 층이라 평소엔 하는 일 없이 대기.

## Kafka 유실과 관련된 방어는 뭔가?

사용자가 "Redisson 락이 Kafka 유실 방어" 라고 생각한 건, 아마 **재고 차감 파이프라인 전체의 방어 계층** 을 통합해서 생각한 것 같아요. Kafka 유실은 별도 층이 담당합니다:

| 문제 | 방어 |
|-----|------|
| Kafka 메시지 유실 (프로듀서→브로커) | `acks=all` + `enable.idempotence=true` (기본값). 미래에 outbox 패턴 |
| Kafka 메시지 중복 전달 (at-least-once) | `Orders.isStockDeducted()` 플래그 검사 (12번 단계) |
| Kafka 처리 실패 | DefaultErrorHandler 3회 재시도 → DLT → compensate() |
| **동일 stock 동시 UPDATE (동시성)** | **Redisson 락** ← 여기 |
| Redisson 락 누수 시 lost update | @Version 낙관적 락 |

**Redisson 락은 "여러 스레드/인스턴스가 같은 재고에 동시 접근"** 문제를 해결하고, **Kafka 유실은 완전 다른 계층에서 처리**됩니다.

## 정확한 재정리

```
재고 차감 시 3층 방어:

층 1: Redisson 분산 락 (동시성 제어)
  ├─ 목적: 같은 매장/메뉴의 동시 요청을 순차 처리
  ├─ 실패 모드 방어: 여러 Consumer 병렬 처리, 여러 인스턴스 스케일아웃
  └─ 부속: tryLock(5초) = 무한 대기 방지

층 2: (Kafka 재전달 방어 — 12번 isStockDeducted 검사)
  └─ 실제로는 재고 차감 로직 밖에 있지만 논리적으로는 방어 체인의 일부

층 3: @Version 낙관적 락 (극단 상황 방어)
  ├─ 목적: 층 1 이 새는 상황에서도 데이터 정합성 유지
  └─ 실패 모드 방어: watchdog 실패, GC pause, Redis 페일오버 갭
```

## 락 보유 시간을 결정하는 요소

먼저 짚어야 할 게, `tryLock(5, TimeUnit.SECONDS)` 은 **락을 잡으려고 기다리는 최대 시간** 이지 락을 보유하는 시간이 아니에요. 대기자가 5초 안에 락을 못 잡는다 = **락 보유자가 오래 잡고 있어서** 예요.

락 보유자가 실제로 뭘 하고 있는지 보면 (`StockLockFacade.java:27-38`):

```java
if (!lock.tryLock(5, TimeUnit.SECONDS)) {...}
stockService.deductStock(storeId, menuId, amount);   // ← 이 시간만큼 락 유지
lock.unlock();
```

그리고 `StockService.deductStock` 은:
```java
@Transactional(timeout = 10)   // ← 최대 10초까지 트랜잭션 유지
public void deductStock(...) {
    Stock stock = stockRepository.findByStoreAndMenu(...);
    stock.deduct(amount);   // → UPDATE stock SET quantity=?, version=? ...
}
```

즉 **락 보유 시간 = 트랜잭션 커밋까지** (프록시가 커밋하고 리턴할 때까지). 이게 얼마나 걸리느냐에 따라 대기 큐가 얼마나 쌓이느냐가 결정돼요.

## 락이 5초까지 대기되는 시나리오 5가지

### 시나리오 1: 대기 큐 누적 (가장 흔함) ⭐

피크 시간대 강남매장 아메리카노에 100건 몰리는 상황:

```
각 요청의 락 보유 시간 = ~50ms (정상)
Consumer 1  : 락 획득 → 처리 → 해제 (50ms)
Consumer 2  : 대기 50ms → 획득 → 처리 (총 100ms)
Consumer 3  : 대기 100ms → 획득 → 처리 (총 150ms)
...
Consumer 100: 대기 4950ms → 획득 → 처리 (총 5000ms)
Consumer 101: 대기 5000ms → 5초 초과 → StockLockException
```

각 요청 자체는 빠르지만 **줄이 너무 길어서** 뒷사람이 5초를 초과하는 케이스. 이건 "장애" 가 아니라 **정상 경합** 이에요.

### 시나리오 2: DB UPDATE 자체가 느려짐 ⭐ 사용자가 지적한 그거

같은 100건이 몰려도 UPDATE 가 500ms 씩 걸리면:

```
Consumer 1: 락 획득 → 500ms → 해제
Consumer 2: 대기 500ms → 처리 500ms (총 1000ms)
...
Consumer 11: 대기 5000ms → 5초 초과 → 예외
```

**10건만 앞에 있어도 5초 초과**. UPDATE 가 느려지는 원인들:

- **다른 롱 트랜잭션이 같은 stock row 락 잡음** — @Version 아닌 다른 곳에서 SELECT FOR UPDATE 를 걸었거나
- **커넥션 풀 고갈** — HikariCP 가 커넥션 없어서 대기 (기본 30초 timeout)
- **DB CPU/IO 부하** — 다른 매장의 대량 배치가 DB 잡아먹는 중
- **인덱스 부재로 풀 스캔** — `findByStoreAndMenu` 가 인덱스 없이 돌면 stock 테이블 커질수록 느려짐
- **네트워크 지연** — DB 가 다른 AZ 에 있어서 왕복 지연 누적

### 시나리오 3: JVM GC pause / STW ⭐ 자주 잊음

락 보유 중 Full GC 가 발생하면 **STW 시간만큼 락이 유지**됩니다:

```
Consumer 1: 락 획득 → Stock UPDATE 시작
           → Full GC 발생 → STW 3초 ⏸
           → 재개 → UPDATE 완료 → 해제 (총 3050ms)
Consumer 2~4: 그동안 대기 → 3050ms 부터 순차 처리
```

이 프로젝트 Dockerfile 을 보면 JVM 힙이 작게 설정돼 있을 텐데, 트래픽 급증 시 GC 가 자주 발생하면 이런 pause 가 반복될 수 있어요.

### 시나리오 4: `@Version` 충돌 후 재시도 없음

이건 좀 반대 방향으로 문제인데, `@Version` 실패 시 자동 재시도 로직이 없어요:

```
Consumer A: 락 획득 → SELECT stock (version=1) → UPDATE (version=2) → 커밋 → 해제
Consumer B: 락 획득 (A 해제 직후) → SELECT stock (version=1 캐시된 상태에서 시작?)
           → UPDATE WHERE version=1 → rowCount=0
           → OptimisticLockException
```

실제로는 락으로 직렬화되니 이 상황이 잘 안 생기지만, **Redisson 락이 새는 극단 케이스**(watchdog 실패 등) 에서 발생 가능. 이 경우 Consumer B 는 예외 → Kafka 재시도 → 또 락 획득 → 다시 시도. **재시도 사이에 다른 요청이 계속 대기** → 대기 큐 누적.

### 시나리오 5: Redis 자체 지연

락 획득/해제 자체가 Redis 왕복이라 Redis 가 느려지면:

- Redis CPU 100% (다른 무거운 쿼리가 돌고 있거나)
- Redis 네트워크 순단
- Redisson 커넥션 풀 고갈

이 경우 락 로직 자체가 초 단위로 느려질 수 있어요. 정상 상태에선 락 획득 = 수 ms 이내인데 이게 500ms 씩 걸리면 대기 큐가 급속히 쌓임.

## 재밌는 점 — `timeout=10` 과 watchdog 의 관계

`@Transactional(timeout = 10)` 이 걸려있으니 **트랜잭션은 최대 10초까지만 유지** 됩니다:

```java
// StockService.java:92
@Transactional(timeout = 10)
public void deductStock(Long storeId, Long menuId, int amount) { ... }
```

그리고 Redisson watchdog 은 **락을 30초 주기로 갱신**하니, 10초 트랜잭션 정도는 여유롭게 커버.

즉:
- **정상 락 보유**: ~50ms
- **최대 락 보유** (트랜잭션 timeout 걸림): ~10초
- **watchdog 없이 락 만료** 되는 시나리오: 없음 (트랜잭션이 먼저 timeout 걸림)

**만약 트랜잭션이 10초까지 갈 정도로 DB 가 느려지면?**
- 트랜잭션 timeout 예외 발생 → 롤백 → finally 에서 unlock
- 하지만 그동안 대기 중이던 요청들은 이미 5초 초과로 다 예외 처리됨
- Kafka 재시도로 다시 큐에 들어감 → 문제 재발 가능

## 실제 발생 원인 분석 프레임

로그에 `StockLockException` 이 뜬다면 이 순서로 확인:

**1단계: 얼마나 자주 발생?**
- 특정 매장/메뉴에 집중되면 → 정상 경합 (시나리오 1)
- 매장/메뉴 상관없이 발생하면 → DB 나 인프라 문제 (시나리오 2, 5)
- 특정 시간대 집중되면 → GC (시나리오 3)

**2단계: 락 보유 시간 측정**

지금 코드엔 없는데, 추가하면 좋은 메트릭:
```java
long start = System.currentTimeMillis();
stockService.deductStock(...);
long heldMs = System.currentTimeMillis() - start;
Timer.builder("stock.lock.held").register(meterRegistry).record(heldMs, TimeUnit.MILLISECONDS);
```

- p50 이 100ms 미만이면 정상
- p50 이 500ms 이상이면 DB 느림 의심
- p99 만 튀면 GC 의심

**3단계: DB 슬로우 쿼리 로그**

- PostgreSQL `log_min_duration_statement = 500` 설정
- `UPDATE stock` 이 자주 뜨면 인덱스 확인, 다른 롱 트랜잭션 확인

**4단계: Consumer 병렬성 확인**

`OrderKafkaConsumer.consume` 에 `concurrency` 옵션이 없으면 파티션 수(3)가 최대 병렬성. Consumer 스레드가 부족해서 큐가 밀리는 상황일 수도 있어요.

## 정리

**"Redis 락을 5초 동안 잡고 있는 이유"** 를 정확히 하면:

- 한 요청이 5초 동안 락을 붙잡고 있는 건 **드묾** (트랜잭션 timeout 10초가 상한)
- 실제로 5초 초과가 나는 건 대개 **여러 요청이 대기 큐에 쌓여서** 뒷사람이 5초를 못 채우고 예외로 빠지는 경우
- 그 근본 원인 순위:
  1. **DB UPDATE 지연** (사용자 지적 대로, 가장 흔함)
  2. **정상 트래픽 폭주** (인기 메뉴 편중)
  3. **JVM GC pause**
  4. **Redis 자체 지연**

`tryLock(5초)` + `StockLockException` → Kafka 재시도 구조는 **"경합 상황을 즉시 실패시켜 다른 요청을 막지 않고, 재시도로 넘겨서 부하가 풀리길 기다린다"** 는 백프레셔(backpressure) 패턴이에요. 무한 대기하면 컨슈머 스레드 다 잠기고 전체 시스템이 멈추는데, 이걸 방지하는 안전장치죠.

GC pause 는머야 ? 가비지 콜렉터 ㅓ푸시 ? 파즈?

**Assistant (16:07):**

**GC = Garbage Collector (가비지 컬렉터), Pause = 포즈 (일시 정지)**. 발음은 "지씨 포즈".

## GC 란

Java 는 개발자가 메모리를 직접 해제하지 않아요. 대신 **JVM 이 자동으로 안 쓰는 객체를 청소** 하는데, 이 청소부가 **Garbage Collector (GC)** 입니다.

```java
User user = new User();   // 힙에 User 객체 생성
user = null;              // 참조 끊음 → 이제 "garbage"

// GC 가 언젠가 알아서 청소함 → 힙 메모리 회수
```

C/C++ 처럼 `free()` 를 직접 호출할 필요 없어서 편하지만, 대신 **GC 가 언제 어떻게 돌지 개발자가 통제하기 어려운** 트레이드오프가 있어요.

## GC Pause 란 — Stop-The-World (STW)

GC 가 힙 메모리를 정리하려면 **"어떤 객체가 살아있고 어떤 게 죽었는지"** 를 알아야 합니다. 그런데 정리하는 도중에도 애플리케이션 스레드가 계속 객체를 만들고 참조를 바꾸면 GC 가 헷갈려요.

그래서 GC 는 특정 단계에서 **"잠깐만! 모든 스레드 멈춰!"** 라고 외치고 애플리케이션을 일시 정지시킵니다. 이걸 **Stop-The-World (STW)** 라고 불러요.

```
[정상 실행 중]
스레드 1: 결제 처리 중...
스레드 2: 재고 차감 중...
스레드 3: HTTP 응답 중...

[GC 시작 → STW]
스레드 1: ⏸ 멈춤
스레드 2: ⏸ 멈춤
스레드 3: ⏸ 멈춤
    GC 작업 중... (수 ms ~ 수 초)

[GC 완료 → 재개]
스레드 1: ▶ 이어서 처리
스레드 2: ▶ 이어서 처리
스레드 3: ▶ 이어서 처리
```

**멈춰 있는 이 시간 = GC Pause = STW pause**.

## 얼마나 오래 멈추나 — GC 알고리즘별 차이

Java 는 여러 GC 를 선택할 수 있어요:

| GC 알고리즘 | 일반적 pause 시간 | 특징 |
|-----------|-----------------|------|
| Serial GC | 100ms ~ 수 초 | 옛날 알고리즘, 작은 앱용 |
| Parallel GC | 100ms ~ 1초 | Java 8 기본, 처리량 중심 |
| G1 GC | 10 ~ 200ms | Java 9+ 기본, 균형 |
| ZGC | ~1ms 이하 | Java 15+, 대용량 힙 |
| Shenandoah | ~1ms 이하 | RedHat 개발, 대용량 힙 |

Java 21 기본은 **G1 GC** 예요. 정상 상황에선 pause 가 10~50ms 정도로 짧지만:

- **힙이 가득 차서 Full GC 발생** → 몇 초까지 갈 수 있음
- **메모리 부족 상태에서 계속 GC 반복** (GC thrashing) → 앱이 사실상 멈춤

## 왜 이 프로젝트에서 문제가 되나

앞서 얘기한 Redisson 락 시나리오를 다시 보면:

```
Consumer 스레드가 락 획득 → Stock UPDATE 실행 중
    ↓
Full GC 발생 → 이 스레드도 STW 로 멈춤 ⏸ 3초
    ↓
그동안 락은 계속 잡혀 있는 상태 (unlock 코드도 멈춘 상태)
    ↓
다른 대기 중이던 요청들은 5초 tryLock 타임아웃 발생
    → StockLockException 대량 발생
    → Kafka 재시도 → 큐 급증 → 부하 증가 → GC 더 자주 발생
    → 악순환
```

**GC pause 중엔 Redisson watchdog 도 못 돌아요**. watchdog 은 앱 스레드가 돌리는 백그라운드 태스크인데 STW 로 다 멈춰있으니 락 갱신이 안 됨. 그래서:

- pause 가 30초 넘어가면 → watchdog TTL 만료 → 락이 새고 다른 프로세스가 획득 가능
- 정상적인 서비스에선 30초 넘는 pause 는 거의 없지만, **메모리 누수/OOM 직전 상황** 에선 실제로 발생

## Dockerfile 힌트

이 프로젝트 Dockerfile 을 봤을 때 힙 크기가 작게 설정돼 있으면 (예: `-Xmx512m`) 트래픽 몰릴 때 GC 발생 빈도가 높아져요. 트래픽 부하 테스트(k6) 돌릴 때 이 부분이 실제로 병목이 될 수 있습니다.

## 확인 방법

**JVM 옵션에 GC 로그 활성화**:
```
-Xlog:gc*:file=gc.log:time,uptime,level,tags
```

로그에 이런 게 찍히면 pause 시간 확인 가능:
```
[2026-07-23T14:23:01.500+0900][12.345s][info][gc] GC(15) Pause Young (Normal) (G1 Evacuation Pause) 128M->32M(512M) 45.678ms
                                                                                                              ^^^^^^^^^^
                                                                                                              이게 pause 시간
```

**Prometheus JVM 메트릭**:
- `jvm_gc_pause_seconds` — pause 시간 히스토그램
- `jvm_gc_pause_seconds_sum` / `jvm_gc_pause_seconds_count` — 평균 pause

이 프로젝트에 이미 Prometheus 가 붙어있으니 Grafana 대시보드에서 GC pause 지표를 볼 수 있어요.

## 실무에서 신경 쓸 만한 GC 관련 이슈

1. **Full GC 빈도** — 자주 발생하면 힙 크기 조정, 메모리 누수 확인
2. **pause 시간의 p99** — 평균이 낮아도 최악값이 크면 사용자 체감 나쁨
3. **알고리즘 선택** — 지연 시간이 중요하면 ZGC/Shenandoah 검토
4. **힙 크기 튜닝** — 너무 작으면 자주 GC, 너무 크면 한 번 GC 가 오래 걸림

## 한 줄 정리

**GC pause = Java 가 메모리를 청소하기 위해 애플리케이션 전체를 잠깐 멈추는 시간** (STW).

이 시간 동안:
- 요청 처리 안 됨 (지연시간 증가)
- 락 보유 중이면 그 락도 계속 유지 (다른 요청 대기 유발)
- Redisson watchdog 도 멈춤 (극단적으로 길면 락 만료 위험)

그래서 "락을 5초 잡을 만한 상황" 을 얘기할 때 GC pause 가 자주 등장한 거예요. DB 가 정상이어도 GC 만으로 락이 몇 초 잡힐 수 있으니까요.
