## 원인 분석

에러 메시지의 핵심 단서는 **컨테이너 이름이 `/56a673dc8611_whale-backend`** 라는 점입니다.

- 접두사 `56a673dc8611` = 그 뒤 컨테이너 ID(`56a673dc86115069...`)의 앞 12자리와 **정확히 일치**
- 이 `<컨테이너ID>_<원래이름>` 패턴은 Docker(특히 docker-compose)가 컨테이너를 교체할 때 **기존 컨테이너를 자동 rename 할 때 붙이는 패턴**입니다.

### 어떤 상황에서 생기나

`docker-compose up -d`는 같은 서비스의 기존 컨테이너를 교체할 때 다음 순서로 동작합니다:

1. 기존 `whale-backend` 컨테이너를 stop
2. **`whale-backend` → `<id>_whale-backend` 로 rename** (혹시 모를 롤백을 위해 잠시 보관)
3. 새 이미지로 `whale-backend` 컨테이너를 create
4. 보관해 둔 `<id>_whale-backend` 를 remove

3 또는 4 단계에서 실패(권한, 디스크, 네트워크 오류, 워크플로우 중단 등)하면 **rename된 좀비 컨테이너만 EC2에 남게** 됩니다. 다음 배포 때 compose가 또 같은 ID 접두사로 rename을 시도하다가 위와 같이 충돌하는 거예요.

### "분명히 삭제했는데?" 인 이유

- `docker ps`(실행 중인 것만)로 확인하고 지웠다면, **stopped 상태의 rename된 좀비**가 안 보였을 가능성이 큼. `docker ps -a` 로만 보입니다.
- 또는 docker-compose.prod.yml 의 `restart: unless-stopped` 정책 때문에 의도와 다르게 살아남았을 수도 있음.
- `docker rm` 대상에 그 좀비 ID/이름이 빠졌을 가능성.
