# 개념 정리 - 2026.05.27

---

## 1. Mock 결제

### 개념
실제 PG(결제대행사)와 연동하지 않고 결제 플로우만 구현하는 방식.
카드 정보나 실제 돈의 이동 없이 결제 성공/실패 시나리오를 시뮬레이션한다.

### 실제 결제 vs Mock 결제

| 항목 | 실제 결제 | Mock 결제 |
|---|---|---|
| 연동 대상 | 토스페이먼츠, KG이니시스 등 | 없음 |
| 카드 정보 | 실제 카드 번호 필요 | 임의 입력 또는 버튼 클릭 |
| 돈 이동 | 실제 출금 | 없음 |
| 구현 목적 | 실서비스 | 플로우 / 아키텍처 검증 |

### 흐름 예시
```
주문 확인
  → 결제 수단 선택 UI (카드/카카오페이 등 — 표시만)
  → "결제 완료" 클릭
  → 백엔드: 주문 상태 PENDING → ACCEPTED 변경
  → 완료 화면
```

### 왜 사용하는가
- 실제 PG 계약 없이도 결제 플로우와 예외 처리(Saga 패턴 등)를 검증할 수 있다.
- 나중에 실제 PG 모듈만 교체하면 바로 서비스 가능한 구조를 미리 만들 수 있다.
- 포트폴리오에서 결제 트랜잭션 흐름 설계 경험을 어필할 수 있다.

## 2. Saga 패턴

### 개념
분산 시스템에서 여러 단계로 이루어진 트랜잭션을 안전하게 처리하기 위한 패턴.
각 단계마다 **보상 트랜잭션(Compensating Transaction)** 을 정의해 실패 시 이전 상태로 되돌린다.

### 왜 필요한가
마이크로서비스나 외부 API가 포함된 트랜잭션은 DB의 `ROLLBACK` 한 번으로 되돌릴 수 없다.

```
① 재고 차감 (내부 DB)
    ↓
② 결제 요청 (외부 PG사 API)  ← 여기서 실패하면?
    ↓
③ 주문 확정 (내부 DB)
```

② 실패 시 ①에서 차감한 재고를 DB 롤백으로는 되돌릴 수 없다.
→ 명시적인 "재고 원복" 로직이 필요하다.

### 해결 방식

각 단계에 보상 트랜잭션을 정의한다.

```
① 재고 차감       실패 시 보상 → 재고 원복
        ↓
② 결제 요청       실패 시 보상 → 재고 원복 실행
        ↓
③ 주문 확정       실패 시 보상 → 결제 취소 + 재고 원복
```

결제 실패 → ②의 보상 실행 → ①의 보상 실행 → 원래 상태 복구

### 두 가지 구현 방식

#### Choreography (코레오그래피)
- 각 서비스가 이벤트를 발행/구독하며 자율적으로 다음 단계를 처리
- 도구: Kafka 이벤트
- 장점: 서비스 간 느슨한 결합
- 단점: 전체 흐름 추적이 어려움

#### Orchestration (오케스트레이션)
- 중앙의 오케스트레이터(Saga 클래스)가 각 단계의 실행 순서를 지휘
- 도구: Spring 내 Orchestrator 클래스
- 장점: 전체 흐름을 한눈에 파악 가능
- 단점: 오케스트레이터가 단일 장애점(SPOF)이 될 수 있음

### WhaleOrder 프로젝트 적용 예시

```
재고 차감 (Redis 분산 락)
    → Mock 결제 API 호출
        → 성공: 주문 확정 (PENDING 상태로 저장)
        → 실패: 재고 원복 + 주문 취소 상태로 저장
```

Kafka가 있으면 Choreography 방식, 없으면 Spring Orchestration 방식으로 구현.

## 관계 정리

```
Mock 결제  →  결제 성공/실패 시뮬레이션
Saga 패턴  →  결제 실패 시 이전 단계(재고 차감 등)를 보상 트랜잭션으로 되돌리는 흐름 설계
```

두 개념을 함께 구현하면 실제 PG 없이도 장애 내성 있는 결제 플로우를 검증할 수 있다.
