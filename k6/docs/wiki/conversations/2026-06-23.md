## 왜 검증 방식을 바꿨나 (핵심)

새 흐름은 `결제 200 → AFTER_COMMIT → Kafka → OrderProcessingService.deductStock()` 이라, **결제 200이 재고 차감 성공을 의미하지 않습니다.** 재고는 응답 이후 비동기로 차감되고 부족하면 주문이 CANCELLED로 보상됩니다.

그래서 예전처럼 "성공 응답 = N건"으로 분산락을 검증하면 거짓 결과가 나옵니다(결제는 재고와 무관하게 거의 다 200). 대신 `teardown()`에서 비동기 차감이 정착될 때까지 폴링한 뒤 `GET /api/stores/{storeId}/stocks`로 **실제 잔여 재고**를 읽어:
- `차감량 ≤ 초기 재고` && 음수 없음 → **분산락 정상**
- 음수 또는 초과 차감 → **분산락 오류(오버셀)**

`node --check` 문법은 통과했고, k6 실제 실행은 사용자 환경(Docker)에서 확인이 필요합니다. 결정 요약은 [docs/wiki/conversations/2026-06-23.md](docs/wiki/conversations/2026-06-23.md)에 기록했습니다.

한 가지 참고: 이 테스트는 비동기 차감을 검증하므로 **Kafka/컨슈머가 실제로 돌아가는 프로파일**에서 실행해야 의미가 있습니다. 만약 단일 인스턴스에서 Kafka 없이 직접 `process()`를 호출하는 구성이라면 그대로 동작하고, Kafka 경유라면 teardown 폴링(최대 ~20초)이 그 지연을 흡수합니다.

## 근본 원인: `stock.version` 낙관적 락 초기화 누락

[schema.sql:105](src/main/resources/db/schema.sql#L105)를 보면 두 version 컬럼의 정의가 다릅니다:

```sql
-- orders 테이블 (정상)
version  BIGINT  NOT NULL DEFAULT 0,   -- line 120

-- stock 테이블 (버그)
version  BIGINT,                        -- line 105 — nullable, 디폴트 없음
```

그리고 [data.sql:110](src/main/resources/data.sql#L110)의 stock INSERT는 `version` 컬럼을 안 넣습니다 → **시드된 모든 재고 row의 `version = NULL`**.

`Stock` 엔티티에는 `@Version Long version`이 붙어 있어서([Stock.java:24-25](src/main/java/com/whale/order/domain/stock/entity/Stock.java#L24-L25)), Hibernate가 재고를 **수정**할 때 이런 UPDATE를 만듭니다:

```sql
UPDATE stock SET quantity=?, version=1 WHERE stock_id=? AND version=?  -- ? = NULL
```

`WHERE version = NULL`은 SQL에서 **절대 참이 안 됨** → 0 rows 갱신 → `StaleObjectStateException` → 처리 안 된 예외 → **500**.

- `setStock`(어드민 PUT) → 500 ← 지금 보신 것
- **비동기 `deductStock`(주문 처리)** → 같은 `StaleObjectStateException` → OrderProcessingService가 "재고 부족"으로 오인 → 주문 취소
