# AI 페어 프로그래밍 워크플로우

> 이 프로젝트는 AI(Claude)와 **페어 프로그래밍**으로 개발했습니다. 핵심은 "AI를 썼다"가 아니라
> **AI의 출력을 어떻게 코드·문서와 정합하게 통제했는가**입니다. 그 장치와 실제로 오류를 잡아낸 사례를 정리합니다.

---

## 1. 협업 모델 — AI는 페어, 결정·승인은 사람

- AI는 코드 탐색·초안 작성·리뷰·디버깅 보조를 맡고, **설계 결정과 머지 승인은 사람**이 합니다.
- 모든 변경은 "원인 분석 → 방향 제시 → 사람 승인 → 적용 → 검증" 순서를 따릅니다.
- 외부로 나가거나 되돌리기 어려운 작업(커밋/푸시/스키마 변경)은 **반드시 사람 확인 후** 진행합니다.

## 2. 하네스 — 통제 장치가 "강제"되는 실행 계층

이 문서의 장치들(훅·권한·메모리 등)이 작동하는 이유는, 그것들이 **AI에게 보내는 부탁이 아니라 하네스가 집행하는 규칙**이기 때문입니다.

하네스(harness)는 모델(Claude)을 감싸는 **런타임**입니다. 모델이 직접 파일을 쓰거나 셸을 실행하는 게 아니라, 모델이 "도구 호출"을 내면 **하네스가 그 호출을 받아 권한 모드·훅을 거쳐 실제로 실행**합니다. 즉 모델과 시스템 사이에 항상 한 겹의 통제 계층이 있습니다.

- **권한 모드**: 사용자가 고른 모드 뒤에서 도구가 실행됩니다. 거부된 호출은 사람이 막은 것이며, AI는 우회할 수 없습니다. 반복되는 읽기 전용 명령은 allowlist로 미리 허용합니다(§6).
- **훅 인터셉트**: 도구 호출/세션 종료 같은 시점을 가로채 검증·로깅을 끼웁니다. 의사결정 로깅(§4)의 Stop hook이 대표 예입니다.
- **컨텍스트 관리**: 긴 세션은 하네스가 자동 요약해 다음 컨텍스트 창으로 이어 줍니다 → 토큰 경제(§6)와 직접 연결됩니다.
- **파일 상태 추적**: "편집 전 반드시 읽기" 같은 규칙을 하네스가 강제해, 모델이 보지 않은 파일을 덮어쓰는 사고를 차단합니다.

> 요점: 정합성·로깅·비용 장치가 **모델 바깥에서 결정적으로 동작**하므로, AI가 "깜빡하거나 무시"해도 무너지지 않는 가드레일이 됩니다. (이하 §3~§6은 이 계층 위에서 동작합니다.)

## 3. 문서-코드 정합성 정책 (`CLAUDE.md`)

프로젝트 루트의 [`CLAUDE.md`](../../../CLAUDE.md)에 답변 근거 정책을 못박아 두었습니다.

- 작업/질문 전 관련 wiki 문서를 **먼저 읽습니다** (`docs/wiki/Home.md`가 진입점).
- 답변은 항상 **(1) wiki + (2) 실제 코드**를 근거로 합니다.
- **두 소스가 충돌하면 코드를 진실로 채택**하고, wiki는 갱신 대상으로 표시합니다.
- wiki ↔ 코드 추적성: 문서는 관련 코드 경로를 `파일:라인`으로 링크해 문서에서 코드로 바로 점프할 수 있습니다.

> 이 정책 덕분에 "그럴듯하지만 코드와 다른 답변"이 구조적으로 걸러집니다. (§7 사례 참고)

## 4. 의사결정 로깅 (Stop hook)

세션 종료 시 [`.claude/hooks/save-conversation.js`](../../../.claude/hooks/save-conversation.js)가 **Stop hook**으로 실행돼
그날 대화를 `docs/wiki/conversations/YYYY-MM-DD.md`에 자동 누적합니다. (이 훅을 가로채 실행하는 주체가 §2의 하네스입니다.)

- 주요 결정은 "결정 요약" 섹션으로 주제 단위 저장 — **무엇을 왜 바꿨는지**가 날짜별로 남습니다.
- 나중 세션이 과거 결정의 맥락을 다시 읽어, 같은 논의를 반복하지 않습니다.

## 5. 메모리 + stale 감지

AI는 프로젝트별 파일 메모리(`~/.claude/projects/.../memory/`)에 4종 사실을 남깁니다.

| 종류 | 내용 |
|------|------|
| `user` | 사용자 역할·선호 |
| `feedback` | 작업 방식 지침(근거 정책, 대화 로깅 등) |
| `project` | 코드/git에서 안 드러나는 진행 맥락 |
| `reference` | 외부 리소스 포인터 |

핵심은 **stale 감지**입니다. 메모리를 읽을 때 "며칠 지난 기록이니 코드로 재검증하라"는 경고가 자동으로 붙고,
fact로 단언하기 전에 코드를 다시 확인하도록 강제됩니다. → 오래된 메모리가 그대로 출력되는 사고를 시스템 단에서 차단합니다.

## 6. 토큰 / 비용 경제

긴 세션에서 토큰·시간 낭비를 줄이는 장치들입니다.

- **권한 allowlist** (`.claude/settings.local.json`): 반복되는 읽기 전용/빌드 명령(`git`, `gradlew compileJava`, `grep` 등)을 미리 허용해 매번 확인 프롬프트로 끊기지 않게 합니다. (하네스의 권한 모드 위에서 동작 — §2)
- **wiki로 컨텍스트 압축**: 코드 전체를 다시 읽는 대신 설계 요약 문서를 1차 근거로 사용해 토큰을 절약하고 일관성을 확보합니다.
- **scratchpad 디렉토리**: 임시 산출물은 프로젝트 밖 격리 공간에 두어 저장소를 더럽히지 않습니다.

## 7. 정합성 장치가 실제로 오류를 잡은 사례 (정직 섹션)

이 워크플로우의 가치는 "AI가 틀릴 수 있다"를 전제하고 **그 오류를 잡는다**는 데 있습니다. 실제 사례입니다.

### ① 오래된 메모리 → 코드 재검증으로 교정
메모리는 "멱등성은 DB 기반, `OrderService.createOrder()`가 진입점, `POST /api/orders`는 데드코드"로 알고 있었습니다.
코드 재검증 결과 **이미 Redis(`SET NX EX`)로 이관**됐고, 진입점은 `PaymentService.pay()`이며, `createOrder`/`POST /api/orders`는 **제거된 상태**였습니다.
→ 메모리를 fact로 단언하지 않고 코드를 먼저 본 §3·§5 정책이 오답을 막았습니다. 이후 메모리를 현재 코드 기준으로 갱신했습니다.

### ② stock.version 오진단 → 가설 폐기
재고 설정 API(`setStock`) 500 에러를 처음엔 "`stock.version`이 NULL이라 낙관적 락이 깨졌다"로 추정했습니다.
라이브 DB를 직접 조회하니 `version=0`이라 **가설이 틀렸음**을 확인하고 폐기했습니다.
진짜 원인은 **메뉴 1(아메리카노)의 필수 옵션**(빈 옵션으로 장바구니 담기 거부)이었습니다.
→ "그럴듯한 추정"을 코드/DB 증거로 검증해 잘못된 수정으로 가지 않았습니다. (방어적으로 schema는 `version NOT NULL DEFAULT 0`으로 보강했습니다.)

### ③ 부하 테스트의 거짓 양성 발견
재고 동시성 테스트가 "결제 200 = 재고 차감 성공"으로 가정해, 비동기 차감(결제 → Kafka → `OrderProcessingService`) 구조에서
**최종 재고가 음수 차감(-90)인데도 "정상"으로 오판**하는 버그가 있었습니다.
검증 기준을 HTTP 응답 → **실제 잔여 재고 조회**로 바꾸고, 거짓 양성 가드를 추가해 교정했습니다.

## 8. 한계

- **자동 메모리 갱신은 없습니다.** stale 감지까지는 자동이지만, 재검증 후 메모리를 다시 쓰는 것은 사람이 트리거합니다.
- AI 출력은 항상 **사람 검토와 코드/DB 증거**로 확정합니다 — 위 사례들이 그 필요성을 보여줍니다.
- 대화 로그(`conversations/`)·학습 노트(`notes/`)는 **당시 시점 기록**이므로, 옛 구조가 나와도 정정 대상이 아니라 히스토리로 둡니다.
