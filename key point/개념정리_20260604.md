## 1. @Transactional Self-Invocation 문제

### 개념

Spring의 `@Transactional`은 **프록시 패턴**으로 동작한다.  
같은 클래스 내부에서 메서드를 호출하면 프록시를 우회해서 `@Transactional`이 무시된다.

### 프록시 패턴이란

실제 객체 앞에 대리인(프록시)을 세워서 호출을 가로채는 패턴.

```
외부 → 프록시.cancelOrder()
              ↓
       트랜잭션 시작
              ↓
       실제.cancelOrder() 실행
              ↓
       트랜잭션 커밋
```

### Self-Invocation 문제

```java
@Service
public class OrderService {

    @Transactional
    public void parent() {
        child(); // this.child() 와 동일 → 프록시 우회 → @Transactional 무시
    }

    @Transactional
    public void child() {
        // 트랜잭션 없음
    }
}
```

`this`는 프록시가 아닌 실제 객체를 가리키기 때문에 프록시를 통하지 않는다.

## 2. JPA vs MyBatis — 증상 차이

같은 self-invocation 문제지만 증상이 다르다.

### JPA (영속성 컨텍스트)

```java
@Transactional // self-invocation → 무시
protected void cancelOrder() {
    order.cancel();          // 메모리만 변경
    // Dirty Checking 없음 → DB 반영 안 됨
}
```

**결과**: 아예 저장이 안 됨.  
트랜잭션이 없으면 영속성 컨텍스트가 유지되지 않아 Dirty Checking이 동작하지 않는다.

### MyBatis

```java
@Transactional // self-invocation → 무시
public void parent() {
    insert1(); // SQL 즉시 실행 → 즉시 커밋
    insert2(); // SQL 즉시 실행 → 예외 발생
}
```

**결과**: insert1은 커밋, insert2는 롤백 → 데이터 절반만 저장된 불일치 상태.  
MyBatis는 영속성 컨텍스트가 없어서 메서드 호출 즉시 SQL이 실행된다.

| | JPA | MyBatis |
|---|---|---|
| **원인** | self-invocation으로 트랜잭션 없음 | 동일 |
| **증상** | 아예 저장 안 됨 | 일부만 저장됨 |
| **위험도** | 그나마 티남 | 데이터 불일치라 발견 어려움 |

## 3. @Transactional이 동작하지 않는 3가지 상황

```java
// 1. Self-Invocation
@Transactional
public void parent() {
    child(); // 프록시 우회
}

// 2. private 메서드 (오버라이드 불가)
@Transactional
private void doSomething() { ... }

// 3. final 메서드 (오버라이드 불가)
@Transactional
public final void doSomething() { ... }
```

| 상황 | 이유 | 동작 여부 |
|---|---|---|
| Self-Invocation | 프록시 우회 | X |
| private 메서드 | 오버라이드 불가 | X |
| final 메서드 | 오버라이드 불가 | X |
| public 외부 호출 | 정상 | O |

## 4. 올바른 사용법

### 문제 — 같은 클래스 내부 호출

```java
@Transactional
public void parent() {
    child(); // @Transactional 무시
}

@Transactional
public void child() { ... }
```

### 해결 1 — 부모만 @Transactional, 자식은 제거

```java
@Transactional
public void parent() {
    child(); // 부모 트랜잭션에 참여
}

private void child() { ... } // @Transactional 제거
```

### 해결 2 — 명시적 save() 추가 (JPA)

```java
protected void cancelOrder() {
    order.cancel();
    orderRepository.save(order); // 트랜잭션 없어도 명시적 저장
}
```

### 해결 3 — 클래스 분리

```java
// 별도 클래스로 분리하면 외부 호출이 되어 @Transactional 정상 동작
@Service
public class OrderCancelService {
    @Transactional
    public void cancelOrder() { ... }
}
```

## 6. 오늘 발견한 버그 (실제 사례)

`OrderProcessingService.cancelOrder()`가 같은 클래스의 `processOrder()`에서 호출되어 self-invocation 발생.

```java
// 버그
private void processOrder(...) {
    } catch (Exception e) {
        cancelOrder(orderId); // self-invocation → @Transactional 무시
    }
}

@Transactional
protected void cancelOrder(Long orderId) {
    order.cancel();
    // orderRepository.save() 없음 → DB 반영 안 됨 (JPA Dirty Checking 미동작)
}
```

**증상**: 재고 부족 시 로그에는 "주문 취소"가 찍히지만 DB는 여전히 PENDING 상태.  
**발견**: 통합 테스트에서 DB 상태를 직접 검증해서 발견.  
**수정**: `orderRepository.save(order)` 명시적 추가.

---

## 7. 통합 테스트로 발견하는 이유

```
수동 테스트 → 로그 확인 → "주문 취소" 로그 보임 → 정상으로 착각
통합 테스트 → DB 직접 조회 → PENDING 상태 확인 → 버그 발견
```

로그는 "함수가 호출됐는지"만 확인하지만  
통합 테스트는 "DB에 실제로 반영됐는지"를 검증하기 때문에 이런 버그를 잡을 수 있다.

## 8. Prometheus 커스텀 비즈니스 메트릭

### Prometheus란

애플리케이션의 상태를 숫자로 수집하고 저장하는 모니터링 도구.  
Grafana와 함께 사용해서 시각화한다.

```
Prometheus (숫자 수집/저장) → Grafana (그래프 시각화)
```

### TPS (Transactions Per Second)

1초에 처리하는 요청 수.

```
TPS = 100  → 1초에 주문 100건 처리
TPS = 500  → 점심시간 트래픽 급증
```

### DLQ vs DLT

```
DLQ (Dead Letter Queue) → RabbitMQ, AWS SQS 등에서 쓰는 용어
DLT (Dead Letter Topic) → Kafka에서 쓰는 용어 (같은 개념)
```

Kafka는 Queue가 아닌 Topic 구조이기 때문에 DLT라고 부른다.

### 메트릭 종류

| 종류 | 설명 | 예시 |
|---|---|---|
| **Counter** | 누적 횟수 (증가만 함) | 주문 처리 성공 횟수 |
| **Timer** | 소요 시간 측정 | 주문 처리 시간 |
| **Gauge** | 현재 값 (증감 가능) | 현재 접속자 수 |

### 이 프로젝트에 추가한 메트릭

```java
// Counter — 횟수 카운트
Counter.builder("order.processed")
    .tag("result", "success") // 태그로 성공/실패 구분
    .register(meterRegistry).increment();

// Timer — 처리 시간 측정
Timer.Sample sample = Timer.start(meterRegistry);
// ... 처리 로직 ...
sample.stop(Timer.builder("order.processing.time")
    .tag("result", "success")
    .register(meterRegistry));
```

| 메트릭 | 위치 | 의미 |
|---|---|---|
| `order.processed` (result=success/failure) | OrderProcessingService | 주문 처리 성공/실패 횟수 |
| `order.processing.time` | OrderProcessingService | 주문 처리 소요 시간 |
| `order.stock.shortage` | OrderProcessingService | 재고 부족 발생 횟수 |
| `payment.processed` (result=success/failure) | PaymentService | 결제 성공/실패 횟수 |
| `kafka.dlt.received` | OrderKafkaConsumer | DLT 수신 횟수 |

### 확인 방법

```
서버 실행 후
GET http://localhost:8080/actuator/prometheus
→ 추가한 메트릭이 텍스트 형식으로 노출됨
→ Grafana에서 가져다 대시보드로 시각화
```

### tag를 쓰는 이유

```java
// tag 없이
Counter.builder("order.processed.success").register(meterRegistry);
Counter.builder("order.processed.failure").register(meterRegistry);

// tag 사용 (권장)
Counter.builder("order.processed")
    .tag("result", "success")
    .register(meterRegistry);
```

tag를 쓰면 Grafana에서 `result=success` / `result=failure` 로 필터링하거나  
**성공률 = success / (success + failure)** 계산을 쉽게 할 수 있다.
