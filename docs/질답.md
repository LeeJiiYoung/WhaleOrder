## 1. 동시성 제어

### ★ Q1. 재고 오버셀 문제를 어떻게 해결했나요? (1분 버전)

> "수천 명이 같은 메뉴를 동시에 주문하면, 두 스레드가 재고 1을 동시에 읽고 둘 다 차감해 재고가 음수가 되는 lost update가 생깁니다.
> 저는 **3중 방어**로 풀었습니다.
> **1차는 Redisson 분산 락**입니다. 락 키를 `stock:lock:{storeId}:{menuId}` 로 잡아서 같은 매장·같은 메뉴 요청만 직렬화하고, 다른 메뉴 주문은 안 막아 처리량을 확보했습니다.
> **2차는 `@Transactional(timeout=10)`** 입니다. 트랜잭션이 어떤 이유로 멈춰도 10초 안에 강제 종료돼서 락 보유자가 무한 hang 되는 걸 막습니다.
> **3차는 Stock 엔티티의 `@Version` 낙관적 락**입니다. Redis가 죽어서 락이 유실되는 극단 상황에도 `UPDATE ... WHERE version=?` 로 한쪽만 성공하게 해서 DB 레벨에서 최후 방어를 합니다.
> 검증은 k6로 재고 10개에 20명 동시 주문을 걸어서 정확히 10건만 성공하는 걸 확인했습니다."

### ★ Q2. 분산 락 하나면 되는데 왜 `@Version`까지 이중으로 두나요? (핵심 꼬리질문)

> "분산 락은 **Redis라는 외부 컴포넌트에 의존**합니다. Redis 단일 노드가 죽거나 네트워크 파티션이 나면 락이 유실될 수 있습니다. 그 순간 두 트랜잭션이 동시에 재고 차감에 진입할 수 있는데, 이때 `@Version` 낙관적 락이 DB 레벨에서 한쪽을 `OptimisticLockException` 으로 떨어뜨립니다.
> 즉 **락을 못 믿는 상황을 가정한 설계**입니다. 정상 경로에선 분산 락이 경합 자체를 없애서 낙관적 락 충돌이 거의 안 생기고, 비정상 경로에서만 `@Version` 이 그물 역할을 합니다."

**주의 꼬리질문 — "그럼 `@Version` 충돌 나면 재시도하나요?"**
> "지금은 자동 retry가 없습니다. 충돌한 요청은 예외가 나고, 그 예외를 `OrderProcessingService` 의 Saga 보상 로직이 받아서 차감분 복구 + 주문 취소로 처리합니다. 다만 일시적 경합까지 취소시키는 건 과하니, 실무라면 1~2회 짧은 retry를 앞에 붙이는 게 개선점이라고 봅니다." *(← 한계를 정직하게 인정하는 게 정답)*

### Q3. 왜 `leaseTime` 을 생략했나요? watchdog이 뭔가요?

> "Redisson은 `tryLock(waitTime)` 처럼 leaseTime을 안 주면 **watchdog** 이 동작합니다. 기본 30초 lease를 걸고 10초마다 자동 갱신해서, 트랜잭션이 길어져도 작업 중에는 락이 안 풀립니다. 반대로 락 보유 스레드(JVM)가 죽으면 갱신이 멈춰 30초 뒤 자동 해제됩니다.
> 만약 leaseTime을 고정값으로 주면, 트랜잭션이 그보다 길어질 때 **작업 중인데 락이 풀려버리는** 위험이 있어서 watchdog에 맡겼습니다."

**꼬리질문 — "watchdog의 단점은?"**
> "보유자가 살아있는 한 무한 갱신되니, 특정 요청이 오래 물고 있으면 대기 요청들이 굶을 수 있습니다. 그래서 `@Transactional(timeout=10)` 으로 보유 시간 상한을 따로 걸어둔 겁니다."

### Q4. `waitTime` 5초, 그리고 `isHeldByCurrentThread()` 가드의 의미는?

> "`tryLock(5, SECONDS)` 는 최대 5초 락을 기다리고 못 얻으면 `false` 를 반환합니다. 이때 `StockLockException` 을 던져 사용자에게 '잠시 후 재시도'를 안내합니다 — 무한 대기로 요청이 쌓이는 걸 막는 거죠.
> `finally` 에서 `isHeldByCurrentThread()` 를 확인하고 unlock하는 이유는, **락을 못 얻은 스레드가 unlock을 호출하면 `IllegalMonitorStateException`** 이 나기 때문입니다. 내가 실제로 잡은 락만 푸는 안전장치입니다."

### Q5. 왜 DB 비관적 락(`SELECT FOR UPDATE`)이 아니라 Redis 분산 락인가요?

> "둘 다 검토했습니다. 비관적 락은 구현이 단순하지만 **DB 커넥션을 락 잡는 내내 점유**해서, 동시 주문이 몰리면 커넥션 풀이 고갈됩니다. 또 애플리케이션 인스턴스가 늘어나도 결국 DB 한 곳에 락 부하가 집중됩니다.
> 분산 락은 락 관리를 Redis로 빼서 DB 커넥션을 빨리 반납하고, 멀티 인스턴스에서도 일관되게 동작합니다. 대신 Redis 의존성이 생기니 앞서 말한 `@Version` 으로 보완했습니다."

## 2. Kafka / Saga 보상 트랜잭션

### ★ Q6. 주문 처리를 왜 Kafka로 비동기화했나요?

> "결제 API 응답 시간에서 **재고 차감·SSE 푸시 같은 후속 처리를 떼어내기 위해서**입니다.
> 흐름은 이렇습니다. `PaymentService.pay()` 가 결제 판정과 주문 저장까지만 트랜잭션 안에서 하고, 커밋된 뒤 `@TransactionalEventListener(AFTER_COMMIT)` 에서 Kafka `order-created` 토픽에 발행합니다. 실제 재고 차감은 Consumer가 비동기로 처리하죠.
> 이렇게 하면 사용자는 결제 결과를 빨리 받고, 주문 폭주 시 재고 차감은 Consumer가 자기 속도로 소화합니다. 파티션을 3개 두고 orderId를 키로 써서 같은 주문은 같은 파티션에 떨어지게 해 순서를 보장했습니다."

### ★ Q7. 결제/후속 처리가 실패하면 어떻게 되돌리나요? (Saga)

> "분산 트랜잭션(2PC) 대신 **Choreography 방식 Saga** 로 보상 트랜잭션을 씁니다. 실패 지점이 4가지라 각각 다르게 처리합니다.
> **① 결제 실패**: `pay()` 에 `noRollbackFor = PaymentFailedException` 을 걸어서, 예외를 던져도 `Payment(FAILED)` · `Orders(CANCELLED)` · 이력을 **일부러 커밋**합니다. 실패 시도 자체를 기록으로 남기려는 의도입니다.
> **② 재고 부족(차감 중)**: 여러 메뉴 중 차감 성공한 것만 `deducted` 리스트에 담아두고, 실패하면 **성공분만** 복구합니다. 그다음 `OrderCancelService.cancelOrder()` 로 주문 취소 + 결제 환불.
> **③ 재고 복구도 실패**: 200ms 간격 3회 재시도 후에도 실패하면 `StockRestoreFailure` 테이블에 남기고 어드민에게 SSE 경고를 보냅니다.
> **④ Consumer 3회 재시도 실패**: `order-created.DLT` 로 보내고, DLT Consumer가 `compensate()` → 주문/결제 취소를 태웁니다."

### Q8. 왜 결제 실패 시 전체 롤백이 아니라 FAILED 상태를 커밋해 남기나요?

> "정합성만 보면 롤백이 깔끔하지만, 그러면 **'결제를 시도했다가 실패했다'는 사실 자체가 사라집니다.** 고객 문의나 감사, 실패율 모니터링을 위해 시도 이력이 필요해서 의도적으로 `noRollbackFor` 로 `FAILED`/`CANCELLED` 종결 상태를 보존했습니다. 정합성과 관측성의 트레이드오프에서 관측성을 택한 거고, 대신 이 주문들은 확실히 `CANCELLED` 종결 상태라 재고엔 영향을 안 줍니다."

### Q9. Kafka 메시지가 중복 전달되면(at-least-once)?

> "Consumer 진입부에서 `if (order.isStockDeducted() || status == CANCELLED) return;` 로 **멱등하게** 막습니다. 이미 차감된 주문이 재전달돼도 이중 차감이 안 됩니다. 차감에 성공하면 `markStockDeducted()` 플래그를 세워서 이후 재전달을 무시합니다."

### ★ Q10. DB 커밋은 됐는데 Kafka 발행이 실패하면? (단골 함정)

> "지금 구조의 **가장 큰 약점**이고 솔직히 인정하는 부분입니다. `AFTER_COMMIT` 리스너에서 발행하니 '커밋 후 발행 실패'가 생기면 주문은 DB에 `PENDING` 으로 남는데 재고 차감 이벤트가 유실됩니다.
> 이때 유실 범위가 넓습니다. 재고 차감·관리자 실시간 푸시(`broadcastNewOrder`)·고객 접수 알림이 **모두 Consumer의 `process()` 안에 한 묶음**이라, 발행이 실패하면 셋 다 안 일어납니다. 결과적으로 **고객은 결제는 됐는데 접수 확인을 못 받고, 관리자 실시간 대시보드에도 안 뜨며(주문목록 조회하면 유령 `PENDING` 으로만 보임), 재고는 그대로**입니다.
> 현재 방어는 `log.error` + `order.event.publish.failure` **메트릭 카운터 증가**입니다. Prometheus/Grafana에서 이 카운터로 알림을 띄우고, 남은 `PENDING` 주문을 **수동 재처리**하는 걸 전제로 둔 구조입니다.
> 정석 해결은 **Transactional Outbox 패턴**입니다. 주문과 같은 트랜잭션으로 `outbox` 테이블에 이벤트를 저장하고, 별도 릴레이가 그걸 읽어 Kafka로 발행 + 발행 성공 표시를 하면, DB 커밋과 메시지 발행이 원자적으로 묶입니다. 트래픽이 커지면 이걸 도입하는 게 다음 스텝입니다."

**꼬리질문 — "Outbox 패턴이 뭔가요?"**
> "이질적인 두 저장소(PostgreSQL + Kafka)에 순차로 쓰는 **이중 쓰기(dual write)** 문제를 없애는 패턴입니다. Kafka 발행은 DB 트랜잭션에 못 넣지만, DB 테이블 INSERT는 같은 트랜잭션에 넣을 수 있죠. 그래서 **발행 대신 '발행할 이벤트'를 `outbox` 테이블에 주문과 같은 트랜잭션으로 INSERT** 합니다. 주문과 이벤트가 함께 커밋되니 '주문은 있는데 이벤트가 없는' 상태가 원천적으로 불가능해집니다.
> 그다음 **별도 릴레이**가 `outbox` 에서 미발행 row를 읽어 Kafka로 발행하고 `published=true` 로 마킹합니다. 릴레이가 잠깐 죽어도 row가 DB에 남아 있어서 **유실이 아니라 지연**으로 바뀝니다. 릴레이는 ① `@Scheduled` 폴링 방식(구현 단순) ② Debezium CDC로 DB WAL을 직접 읽는 방식(지연 적음, 인프라 추가) 중 고를 수 있습니다."

**주의 꼬리질문 — "그럼 중복 발행은 없나요?"**
> "있습니다. Outbox는 **at-least-once** 보장이라, 릴레이가 발행 후 마킹 전에 죽으면 같은 이벤트가 두 번 나갈 수 있습니다. 그래서 Outbox가 멱등성을 대체하는 게 아니라, **Consumer 멱등성(Q9의 `isStockDeducted()` 체크)과 짝**으로 써야 합니다. 즉 'exactly-once'가 아니라 '유실 없음 + Consumer 멱등'으로 정확성을 만드는 조합입니다." *(← at-least-once를 정확히 짚는 게 포인트)*

### Q11. DLT가 뭐고 언제 진입하나요? 재시도 무의미한 예외는?

> "DLT(Dead Letter Topic)는 처리에 최종 실패한 메시지를 모아두는 토픽입니다. `DefaultErrorHandler` 에 `FixedBackOff(1초, 3회)` 를 걸어서 3번 재시도 후에도 실패하면 `order-created.DLT` 로 보냅니다.
> 단, `IllegalArgumentException`/`IllegalStateException` 처럼 **재시도해도 결과가 같은 예외**는 `addNotRetryableExceptions()` 로 등록해서 재시도 없이 바로 DLT로 보냅니다 — 재고 부족 같은 건 3초 기다려봤자 소용없으니까요."

## 3. 실시간 푸시 (SSE)

### ★ Q12. 왜 WebSocket이 아니라 SSE인가요?

> "주문 상태 전송은 **서버 → 클라이언트 단방향**이면 충분합니다. 양방향 통신이 필요 없는데 WebSocket을 쓰면 핸드셰이크·연결 관리가 과합니다.
> SSE는 표준 HTTP 위에서 돌아서 프록시 통과가 쉽고, `EventSource` 가 **끊기면 자동 재연결**까지 해줍니다. `text/event-stream` 하나로 상태 전이(`status`), 재고 차감 결과(`result`), 어드민 브로드캐스트(`newOrder`)를 이벤트 이름으로 구분해 보냅니다."

### Q13. 워커가 먼저 끝나고 브라우저가 나중에 붙는 race condition은?

> "재고 차감 워커가 처리를 끝냈는데 브라우저가 아직 SSE 연결 전이면 결과가 유실됩니다. 그래서 결과를 바로 못 보내면 `pendingResults` 맵에 저장해두고, 브라우저가 `register()` 로 연결하는 순간 맵에서 꺼내 즉시 전송하고 `remove` 합니다. **연결 타이밍과 무관하게 결과가 전달**되도록 한 거죠."

### Q14. 유휴 연결 끊김과 메모리 누수는?

> "두 가지를 처리했습니다. **유휴 끊김**은 `@Scheduled(fixedDelay=25초)` 로 heartbeat 이벤트를 보내 프록시/LB가 죽은 연결로 오판해 끊는 걸 막습니다. **누수**는 `COMPLETED`/`CANCELLED` 같은 종료 상태에 도달하면 이벤트 전송 후 `emitter.complete()` 로 연결을 닫아 맵에서 정리합니다.
> 다만 `pendingResults` 는 아직 TTL이 없어서, 소비되지 않은 결과가 쌓일 수 있는 게 남은 약점입니다 — 정리 스케줄러가 필요합니다."

### ★ Q15. 서버를 여러 대로 늘리면 SSE는 어떻게 되나요? (스케일아웃)

> "현재 emitter를 인메모리 `ConcurrentHashMap` 에 들고 있어서 **단일 인스턴스 전제**입니다. 서버가 2대가 되면, 재고 차감 워커가 A서버에서 돌고 사용자의 SSE 연결이 B서버에 붙어 있으면 푸시가 유실됩니다.
> 해결은 **Redis Pub/Sub** 을 앞단에 두는 겁니다. 워커가 결과를 채널에 publish하면 모든 인스턴스가 구독하다가 자기한테 붙은 연결에만 흘려보내면 됩니다. 규모가 더 커지면 STOMP/WebSocket + 메시지 브로커로 가는 것도 선택지입니다."

## 4. Redis 활용

### Q16. Redis를 캐시 말고 어디에 썼나요?

> "6가지 역할로 썼습니다.
> ① **장바구니** — Hash, `cart:{memberId}`, TTL 24h. 수명 짧고 사용자별 CRUD가 잦아 RDB 부하를 피함.
> ② **리프레시 토큰** — String, `refresh:{memberId}`. 즉시 무효화 + TTL 자동 만료.
> ③ **주문 대기열** — ZSet, `order:queue`, timestamp를 점수로 써서 시간순 처리.
> ④ **메뉴 캐시** — `@Cacheable`, 변경 시 `@CacheEvict`. 조회가 압도적으로 많아서.
> ⑤ **분산 락** — Redisson RLock.
> ⑥ **멱등성 키** — `SET NX EX`.
> 데이터 성격에 따라 자료구조를 다르게 골랐다는 게 핵심입니다."

### Q17. 장바구니를 왜 RDB가 아니라 Redis에?

> "장바구니는 **수명이 짧고 정합성보다 속도**가 중요한 데이터입니다. 담았다 뺐다가 잦은데 이걸 매번 RDB에 쓰면 부하가 큽니다. Redis Hash에 `cart:{memberId}` 로 담고 TTL 24시간을 걸어서, 방치된 장바구니는 자동으로 사라지게 했습니다. 반대로 주문·결제·회원은 ACID가 필요하니 PostgreSQL에 뒀습니다 — 데이터 성격에 맞춰 저장소를 나눈 겁니다."

## 5. 멱등성 / 중복 결제

### ★ Q18. 같은 결제 요청이 두 번 오면(더블클릭/재시도)?

> "**SHA-256 멱등성 키 + Redis `SET NX EX`** 로 막습니다.
> `memberId·storeId·method·orderType·customerRequest·장바구니내용` 을 해싱해 키를 만들고, `markProcessing()` 에서 `setIfAbsent(PROCESSING, 60초)` 를 호출합니다. 이건 원자적이라 **최초 1명만 true** 를 받고 나머지는 '처리 중' 응답을 받습니다.
> 처리가 끝나면 결과를 JSON으로 60초 TTL에 저장해서, 재요청이 오면 저장된 결과를 그대로 반환합니다. 결제가 실패하면 키를 삭제해 재시도를 허용합니다."

**꼬리질문 — "왜 DB `ON CONFLICT` 가 아니라 Redis인가요?"**
> "처음엔 PostgreSQL `ON CONFLICT + delete/insert` 로 했는데, PROCESSING 레코드를 넣은 프로세스가 죽으면 그 키가 **영구히 점유**돼서 재시도가 막히는 race가 있었습니다. `SET NX EX` 는 원자적 삽입과 TTL을 한 줄로 처리해서, 죽은 PROCESSING 키도 60초 뒤 자동 소멸합니다. 코드도 단순해지고 무한 점유 위험이 사라졌습니다."

### Q19. 표시 금액 검증은 왜 하나요?

> "`pay()` 에서 클라이언트가 보낸 `expectedAmount` 와 서버가 장바구니로 재계산한 금액을 비교합니다. 메뉴 가격이 바뀌었거나, 다른 탭에서 장바구니를 수정했거나, 중간자 변조가 있으면 여기서 걸러집니다. **실제 청구는 무조건 서버 계산값만** 씁니다. 클라이언트 금액을 신뢰하지 않는다는 원칙입니다."

## 6. 인증 / 보안

### Q20. JWT + 카카오 OAuth2 인증 흐름은?

> "일반 로그인은 이메일/비밀번호로 Access + Refresh 토큰을 발급합니다. Access는 짧게, Refresh는 Redis `refresh:{memberId}` 에 저장해 TTL로 관리하고, 탈취/로그아웃 시 서버에서 즉시 무효화할 수 있습니다.
> 소셜 로그인은 카카오 OAuth2로 받아온 프로필을 우리 `Member` 로 매핑(신규면 가입)하고 동일하게 우리 JWT를 발급합니다. 요청마다 `JwtAuthenticationFilter` 가 토큰을 검증해 `SecurityContext` 에 인증을 심습니다."

### ★ Q21. (당신 코드의 실제 약점) `POST /api/orders` 가 아직 살아있던데, 진짜 주문 진입점은 결제 아닌가요?

> "맞습니다, 좋은 지적입니다. 이 프로젝트는 **선결제 구조**라 진짜 주문 진입점은 `PaymentService.pay()` 입니다. 결제가 성공해야 주문이 생성되죠.
> 과거의 `OrderService.createOrder()` / `POST /api/orders` 는 **결제 없이 주문을 만들 수 있는 레거시 데드 코드**이고, 지금 흐름에선 쓰이지 않을 뿐 아니라 인증만 뚫리면 무료 주문이 가능한 보안 구멍입니다. 그래서 엔드포인트를 제거하거나 최소한 차단하는 게 맞다고 보고, 정리 대상으로 인지하고 있습니다."

*(면접 팁: 이 질문은 면접관이 코드를 열면 실제로 나올 수 있다. **먼저 자진해서** "레거시 데드 코드가 남아있어 정리 예정"이라고 말하면 코드 관리 능력으로 가점된다.)*

## 7. 아키텍처 / 설계 판단

### Q22. 선결제(결제 → 주문 생성)를 택한 이유는?

> "노쇼와 미결제 주문을 원천 차단하기 위해서입니다. 결제가 성공한 주문만 DB에 생기니, '주문은 있는데 돈은 안 냈다'는 상태가 존재하지 않습니다. 재고 차감도 결제 성공 이벤트 이후에만 일어나 정합성이 단순해집니다. 대신 결제 실패 케이스를 잘 처리해야 해서 `noRollbackFor` 로 실패 이력을 남기고 Saga 보상을 붙인 겁니다."

### Q23. PostgreSQL과 Redis로 데이터를 나눈 기준은?

> "**정합성이 생명인 데이터 vs 속도가 생명인 데이터**로 나눴습니다. 주문·결제·회원은 ACID 트랜잭션이 필요해 PostgreSQL에, 장바구니·세션·대기열·실시간 상태처럼 수명이 짧고 초당 처리량이 중요한 데이터는 Redis에 뒀습니다. 각 저장소의 강점에 맞는 데이터만 올린 거죠."

### Q24. 이 시스템의 병목은 어디이고 어떻게 측정했나요?

> "Micrometer로 `order.processed`, `order.processing.time`, `payment.processed`, `order.stock.shortage` 같은 커스텀 메트릭을 심고 Prometheus + Grafana로 TPS와 Latency를 봤습니다. 부하는 k6로 걸었고, 특히 **재고 10개 / 20명 동시 주문** 시나리오로 동시성 정확성(정확히 10건 성공)을 검증했습니다. 병목은 락 경합 구간이라, 락 키를 매장 전체가 아니라 매장·메뉴 단위로 쪼개 경합을 줄이는 방향으로 잡았습니다."

### Q25. 트래픽이 10배 되면 어디부터 손보겠나요?

> "우선순위로는 ① SSE를 인메모리에서 **Redis Pub/Sub** 으로 바꿔 다중 인스턴스 대응, ② Redis 단일 노드를 **Sentinel/Cluster** 로, ③ Kafka 발행을 **Outbox 패턴** 으로 신뢰성 강화, ④ DLT Consumer와 파티션/replica 증설 순으로 보고 있습니다. 지금 구조는 단일 노드 전제라 이 지점들이 명확한 다음 스텝입니다."

|------|----------------|
| "락을 잡은 채 트랜잭션 커밋하면 락 유지 시간이 길지 않나?" | 그래서 락 키를 매장·메뉴로 좁히고 `timeout=10` 상한을 뒀다. 트레이드오프를 인지하고 통제 중. |

| "멱등성 키에 장바구니 내용이 들어가면, 카트가 조금만 달라도 중복으로 안 잡히지 않나?" | 맞다 — '동일 카트 재제출'만 잡는 설계. 진짜 중복 주문 방지는 이 정도면 충분하고, 필요하면 클라 발급 request-id를 추가로 받는다. |

| "결제가 Mock인데 실제 PG 붙이면?" | 네트워크 타임아웃·부분 성공·환불 멱등성이 새 변수. `PaymentHistory` 기반으로 PG 트랜잭션 ID 멱등 처리를 강화해야 한다. |
| "테스트는 어떻게 했나?" | 동시성은 k6 부하 테스트로 재고 정확성 검증, 비즈니스 로직은 단위/통합 테스트. |
