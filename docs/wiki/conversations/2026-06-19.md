## 왜 둘을 분리했나

| 구분 | 멱등성 (DB) | 분산락 (Redisson) |
|---|---|---|
| 목적 | 같은 요청 1회만 처리 + **결과 캐시** | 임계 영역 직렬화 |
| 보관 기간 | 24h (응답 본문까지) | 락 보유 3초 |
| 저장 데이터 | 상태 + JSON 본문 | 단순 락 |
| 핵심 연산 | `INSERT ON CONFLICT` (원자성) | `tryLock / unlock` |
| 실패 복구 | TTL 만료 + PROCESSING 타임아웃 | leaseTime 자동 해제 |

멱등성은 **결과 JSON** 까지 24시간 들고 있어야 해서 Redis 보다 DB 가 자연스러움 (TTL 도 `expires_at` 컬럼 + 스케줄러로 처리). 분산락은 짧고 빠른 직렬화라 Redisson 이 적합. — 둘 다 "분산 동시성 제어" 라는 점은 같지만 요구 특성이 달라서 다른 저장소를 골랐습니다.

`PaymentService.pay` (PaymentService.java:87-100) 의 흐름:
1. `idempotencyService.getResult(key)` — DB 에서 COMPLETED 응답 캐시 조회
2. `idempotencyService.markProcessing(key)` — DB INSERT 로 처리권 획득
3. 결제·주문 로직 실행
4. `idempotencyService.saveResult(key, response)` — DB 에 응답 본문 저장 (10분 TTL)

## 왜 PaymentService 가 직접 안 쓰고 우회하나

`@TransactionalEventListener(AFTER_COMMIT)` 가 핵심 이유:

1. **DB 커밋 보장 후에만 Kafka 메시지가 나감** — 결제 트랜잭션이 롤백되면 메시지도 안 나감 → 유령 주문 메시지 차단
2. **PaymentService 트랜잭션이 Kafka 응답을 기다리지 않음** — `KafkaTemplate.send().get()` 이 동기 호출이라 PaymentService 안에서 했으면 결제 트랜잭션 길이가 늘어남
3. **Kafka 발행 실패가 결제에 영향 없음** — 리스너는 `try/catch` 로 예외 삼킴. 주문은 PENDING 으로 DB 에 남고, 모니터링/재처리는 별도 (로그 + 메트릭)
4. **테스트/로컬 환경 분기** — `kafka.enabled=false` 면 `OrderKafkaProducer` 빈이 없어서 `Optional.isEmpty()` → 즉시 인메모리 `process()` 로 fallback. PaymentService 는 그런 분기 모름

요약하면: `PaymentService` → `ApplicationEvent` → `OrderEventListener(AFTER_COMMIT)` → `OrderKafkaProducer` → Kafka. PaymentService 코드에 Kafka 흔적이 없는 게 의도된 분리입니다.


## "연결 끊김" 을 어떻게 알아내나

브라우저 탭을 닫거나 네트워크 끊기면? Spring 이 자동 감지해서 콜백 호출:
- `emitter.onCompletion()` — 정상 종료 시
- `emitter.onTimeout()` — 5분 동안 활동 없음
- `emitter.onError()` — IO 에러 (소켓 끊김 등)

OrderSseService.java:60-61 에서 이 3 콜백 모두에 `emitters.remove(orderId)` 를 등록해뒀음 → **연결 끊기는 순간 Map 에서 자동 제거**. 그래야 메모리 누수가 안 생기고, 다음 푸시는 pendingResults 로 가게 됨.

### 1. SSE Emitter Pool 이 **cylinder3 (DB 모양)** 로 표시됨

### 2. **메트릭 스크래핑 화살표 방향**
edge `LIUpFxwcL3vg76uLq3Xt-133` 의 sourcePoint(1380, 410) → target=Monitoring. 즉 **Spring Boot → Prometheus** 방향으로 보임. 실제는 **Prometheus 가 `/actuator/prometheus` 를 pull** 합니다. 화살표 머리(target)가 반대여야 해요.

### 3. **Kafka Consumer / DLT Consumer 가 Spring Boot 컨테이너 바깥**
같은 JVM 안의 `OrderKafkaConsumer` 클래스 (consume + consumeDlt 메서드 2개) 인데, swimlane 바깥에 있어서 별도 서비스/프로세스로 보입니다. **Spring Boot swimlane 을 아래로 늘리거나 점선 컨테이너로 표시** 권장.

### 4. **Member / Auth 박스 색이 외부 API 와 동일 (#FFE812)**
Kakao OAuth Server, Kakao Map API 와 같은 노란색이라 **내부 컴포넌트인지 외부인지 헷갈립니다**. 흰색(#FFFFFF) 으로 통일하고 카카오 로그인 강조는 화살표만으로 충분.

### 5. **"Saga 보상 ①" 표기가 부정확**
edge 93 의 라벨: "결제 실패→Payment(FAILED)·Orders(CANCELLED) 이력 보존"
- 코드 (`PaymentService.java:79`): `@Transactional(noRollbackFor = PaymentFailedException.class)` — **하나의 트랜잭션 안에서 이력만 남기는 commit 패턴**이지 보상 트랜잭션이 아니에요.
- 진짜 Saga 보상은 ②(Kafka 컨슈머 재고 부족) + ③(DLT) 두 개뿐.
- ① 은 라벨을 **"결제 실패 → 이력 보존 commit (noRollbackFor)"** 정도로 바꾸거나 점선 색을 다른 것으로 분리.

### 6. **OrderSseService 의 "Heartbeat 25s" 가 오해 소지**
코드 (`OrderSseService.sendAdminHeartbeat`) 의 `@Scheduled` 는 **`adminEmitters` 만** 대상. 고객 채널(`emitters`, `statusEmitters`) 에는 heartbeat 없음. **"Heartbeat 25s (어드민)"** 로 표기.

### 7. Floating edge 다수 — source/target 미지정
edge `124, 126, 128, 132, 133, 137, 142, 154` 등은 sourcePoint/targetPoint 만 하드코딩되어 있어요. 박스 위치 조금만 옮겨도 화살표가 떠다닙니다. **각 edge 에 `source=` 또는 `target=` 속성으로 박스에 anchor 시키는 게 안전**.

### 8. **"재고 차감 요청 (동기 내부 호출)" 색이 Redis 분산락 색**

### 9. **"구독 등록" 화살표 출발점이 모호**
edge 132 가 sourcePoint(1000, 410) → SSE Emitter Pool. 진짜 흐름은 **React → `OrderQueueController.waitForResult` → `orderSseService.register()` → emitters Map** 인데, 다이어그램에선 클라이언트 출발이 안 보임. **React 박스에서 출발하는 별도 SSE 화살표가 있으면 더 정확**.

### 10. **Payment Service 박스 설명**
"결제 + 주문 생성 + 멱등성 검사 + Saga 보상"
- 멱등성 검사 ✅ (`IdempotencyService` 호출)
- Saga 보상은 PaymentService 가 아니라 OrderProcessingService/DLT Consumer 가 트리거. **PaymentService 는 보상의 "대상"(`cancelPayment`) 이지 보상자가 아님**.
- "Saga 보상" 단어를 빼고 **"결제 + 주문 생성 + 멱등성 + 환불 처리"** 가 정확.

### 11. **Order Service 박스 — 가장 왼쪽**
실제 결제 진입은 PaymentService 라서 흐름상 PaymentService 가 가장 왼쪽이면 자연스럽지만, 시각적 우선순위 정도 차이. 안 고쳐도 OK.

### 12. **Event Queue "Redis 큐"**
코드에 `EventQueueController` 가 있긴 한데 정확히 Redis 큐인지 분산락+카운터인지 확인 필요. 일단 wiki 의 `architecture/redis-usage.md` 참고해서 표현 일치시키세요.

### 13. **Order Service 박스에 5개 서비스 응축**
실제 코드의 `OrderService`, `OrderProcessingService`, `OrderCancelService`, `OrderQueueService`, `OrderSseService` 가 한 박스 — 단순화로 OK 지만, **OrderSseService 가 그 안에 있는지 별도 박스(Heartbeat 25s)인지 일관성 부족**. OrderSseService 가 별도 박스라면 다른 4개도 박스 나누거나, Order Service 박스 안에 작은 서브 박스로 표시.

## 사가(Saga) 패턴의 엄밀한 정의

Saga = 여러 개의 **독립적인 로컬 트랜잭션** T₁ → T₂ → … → Tₙ 로 쪼개진 긴 작업. 각 Tᵢ 에 대응하는 **보상 트랜잭션 Cᵢ** 가 있고, 중간에 실패하면 **이미 commit 된** 앞 단계들을 Cᵢ 로 되돌립니다.

핵심은 두 가지:
1. **이미 commit 된 상태가 있어야** 보상할 게 있음
2. **별도의 트랜잭션** 으로 되돌림

## ① "결제 실패 → 이력 보존" 의 실체

`PaymentService.pay()` 안쪽:
```java
@Transactional(noRollbackFor = PaymentFailedException.class)  // ← 한 개의 TX
public PaymentResponse pay(...) {
    // Payment 생성, Order 생성, Payment HISTORY PENDING 저장 ...
    
    if (success) { ... }
    else {
        payment.fail(...);             // 같은 TX 안
        savePaymentHistory(... FAILED); // 같은 TX 안
        order.cancel();                 // 같은 TX 안
        orderHistoryRepository.save(... CANCELLED); // 같은 TX 안
        throw new PaymentFailedException(...);
    }
}
```

이건 **모두 하나의 트랜잭션 안에서 일어나는 일**입니다:
- 별도의 commit 된 단계가 없음 (Mock PG 결제는 단순 `ThreadLocalRandom` — DB 트랜잭션 밖에서 일어난 외부 작업이 아님)
- `noRollbackFor` 덕분에 throw 해도 **롤백 대신 commit** 됨
- 결과: 하나의 TX 가 "이력 보존하면서 종료"

이건 사가 패턴이라기보다 **"실패 이력을 audit 로 남기는 commit 패턴"** 입니다. 같은 카테고리 이름으로는:
- "Outbox 패턴" 의 변형
- "Soft fail / Negative acknowledgment"

## 비동기로 처리하는 이유 (4가지)

1. **응답 속도** — 사용자가 결제하자마자 "주문 접수됨" 화면을 빠르게 봐야 함. 재고 차감을 동기로 했다면 분산락 대기시간이 응답시간에 포함됨.

2. **수평 확장** — Kafka 토픽 `order-created` 가 partition 3개 → consumer 인스턴스 최대 3개 병렬. 트래픽 늘면 Spring Boot 인스턴스 추가만으로 처리량 비례 증가.

3. **장애 격리** — 재고 차감 중 Redis 일시 장애 / 락 획득 실패 등이 발생해도 결제 자체는 이미 commit. Kafka 가 자동 재시도(1초 간격 3회) → 그래도 실패하면 DLT → Saga 보상.

4. **이벤트 기반 확장 여지** — 같은 `OrderCreatedEvent` 를 다른 컨슈머(예: 알림 발송, 적립금 적립, 매출 통계 집계) 가 추가로 구독해도 메인 결제 로직 수정 없음.

## 정확히 알아둬야 할 디테일

- **Kafka 페이로드는 `orderId` 만**. 컨슈머가 DB 에서 주문 본문을 다시 읽음 — 메시지에 옛 데이터 박혀있을 일 없음.
- **`stockDeducted` 플래그** 로 Kafka at-least-once 재전달 시 중복 차감 방어. 두 번째 호출은 `if (order.isStockDeducted()) return` 으로 무시.
- **분산락 key 가 `stock:lock:{storeId}:{menuId}`** — 다른 메뉴는 병렬 차감 가능. 같은 메뉴만 직렬화.
- **차감 중 `IllegalStateException` (재고 부족) 발생** → `notRetryableExceptions` 에 등록되어 있어서 1초 재시도 없이 즉시 DLT 로. 재시도해봤자 의미 없으니까.

## 왜 고객만 heartbeat 가 없나

각 채널의 수명 특성이 달라요:

### `/result` — 짧고 1회성
```java
SseEmitter emitter = new SseEmitter(5 * 60 * 1000L);  // 5분
```
- Kafka Consumer 가 보통 1초 이내에 재고 차감 끝냄 → 결과 즉시 전송
- 결과 받자마자 `emitter.complete()` → 연결 close
- 길어야 몇 초 사는 연결인데 25초 heartbeat 가 의미 없음
- 5분 timeout 은 "Kafka 가 죽었거나 컨슈머가 너무 느릴 때" 안전장치

### `/updates` — 짧은 ~ 중간
```java
SseEmitter emitter = new SseEmitter(30 * 60 * 1000L);  // 30분
```
- 사용자가 "주문 상세 페이지" 보고 있는 동안만 연결 유지
- 보통 점주가 PENDING → PREPARING → COMPLETED 까지 5~15분이면 처리
- COMPLETED 또는 CANCELLED 받으면 서버가 먼저 `emitter.complete()` 호출 → close
- 30분이면 대부분 정상 종결되니까 heartbeat 없이 timeout 으로 충분

### `/stream` (어드민) — 무제한
```java
SseEmitter emitter = new SseEmitter(0L);  // 타임아웃 없음
```
- 점주가 매장 영업 시간 내내 켜놓는 대시보드
- 6시간, 12시간 연결 유지 필요 → **중간 프록시/브라우저가 유휴 연결 끊는 걸 방지하려고** 25초 마다 heartbeat 송신
- `@Scheduled(fixedDelay = 25_000)` 에서 ping 전송

## 종료 흐름 — "계속 구독" 이 아니에요

고객 입장에서 SSE 가 닫히는 순간들:

```
주문 시점 — 두 채널 모두 OPEN
  ├ /result 구독 시작 ─┐
  └ /updates 구독 시작 ┤
                       │
[1초 후] Kafka 차감 완료
  /result 에 SUCCESS 푸시 → 서버가 emitter.complete()
  └ 1번 채널 CLOSE ✂

[5분 후] 점주 "제조 시작" 클릭
  /updates 에 PREPARING 푸시 → 연결 유지
  
[15분 후] 점주 "완료 처리" 클릭
  /updates 에 COMPLETED 푸시 → 서버가 emitter.complete()
  └ 2번 채널 CLOSE ✂  (TERMINAL 상태 감지 — OrderSseService.java:110)
```

코드로 확인 (OrderSseService.java:107-115):
```java
boolean terminal = TERMINAL.contains(status);  // {"COMPLETED", "CANCELLED"}
if (terminal) statusEmitters.remove(orderId);
if (terminal) {
    send(emitter, "status", json);
    emitter.complete();   // ← 연결 닫음
}
```

## 결론

- **고객 heartbeat: 없음**
- **고객 구독 지속 시간**:
  - `/result`: 보통 1~2초 (결과 받으면 끝)
  - `/updates`: 주문 종료(COMPLETED/CANCELLED)까지 또는 최대 30분
- **어드민만 25초 heartbeat 사용** — 장시간 idle 연결 끊김 방지용

만약 고객 화면이 30분 넘게 켜져있는데 주문이 안 끝나면? `/updates` 가 timeout 으로 끊깁니다. 그러면 클라이언트가 알아서 재연결(EventSource 의 자동 재연결 또는 fetch 수동 재시도)해야 하는데, 현재 프론트가 그걸 처리하는지는 확인이 필요해요 — 보통은 큰 문제 없지만 엣지 케이스로 챙겨볼 만합니다.
