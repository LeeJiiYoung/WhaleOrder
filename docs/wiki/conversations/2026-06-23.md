- 따라서 "N건만 성공" 분산락 검증을 HTTP 응답 카운트로 할 수 없음 → 실제 잔여 재고 조회로 검증해야 정확.

### 적용 (`k6/stock-concurrency-test.js`)
- `setup()`:
  - 어드민으로 `PUT /api/admin/stores/{storeId}/stocks/{menuId}` 호출해 재고를 `INITIAL_STOCK`(10)으로 세팅 → 테스트 결정성 확보(수동 세팅 불필요).

- `teardown()`: 비동기 차감 정착까지 폴링(`GET /api/stores/{storeId}/stocks`) 후 잔여 재고로 검증 — 음수/초과차감이면 분산락 오류, `차감 ≤ 초기재고`면 정상.

## 왜 검증 방식을 바꿨나 (핵심)

새 흐름은 `결제 200 → AFTER_COMMIT → Kafka → OrderProcessingService.deductStock()` 이라, **결제 200이 재고 차감 성공을 의미하지 않습니다.** 재고는 응답 이후 비동기로 차감되고 부족하면 주문이 CANCELLED로 보상됩니다.

그래서 예전처럼 "성공 응답 = N건"으로 분산락을 검증하면 거짓 결과가 나옵니다(결제는 재고와 무관하게 거의 다 200). 대신 `teardown()`에서 비동기 차감이 정착될 때까지 폴링한 뒤 `GET /api/stores/{storeId}/stocks`로 **실제 잔여 재고**를 읽어:
- `차감량 ≤ 초기 재고` && 음수 없음 → **분산락 정상**
- 음수 또는 초과 차감 → **분산락 오류(오버셀)**

`node --check` 문법은 통과했고, k6 실제 실행은 사용자 환경(Docker)에서 확인이 필요합니다. 결정 요약은 [docs/wiki/conversations/2026-06-23.md](docs/wiki/conversations/2026-06-23.md)에 기록했습니다.

한 가지 참고: 이 테스트는 비동기 차감을 검증하므로 **Kafka/컨슈머가 실제로 돌아가는 프로파일**에서 실행해야 의미가 있습니다. 만약 단일 인스턴스에서 Kafka 없이 직접 `process()`를 호출하는 구성이라면 그대로 동작하고, Kafka 경유라면 teardown 폴링(최대 ~20초)이 그 지연을 흡수합니다.

UPDATE stock SET quantity=?, version=1 WHERE stock_id=? AND version=?  -- ? = NULL
```

`WHERE version = NULL`은 SQL에서 **절대 참이 안 됨** → 0 rows 갱신 → `StaleObjectStateException` → 처리 안 된 예외 → **500**.

## 영향이 setStock만이 아닙니다 (더 중요)

읽기(GET 재고)는 version을 안 보니 멀쩡하고, **쓰기만 전부 깨집니다.** 즉:
- `setStock`(어드민 PUT) → 500 ← 지금 보신 것
- **비동기 `deductStock`(주문 처리)** → 같은 `StaleObjectStateException` → OrderProcessingService가 "재고 부족"으로 오인 → 주문 취소

→ 이 버그가 살아있는 한 **시드 재고에 대한 결제는 한 건도 재고 차감이 안 되고 전부 취소**됩니다. 분산락 테스트 이전에 정상 주문 자체가 안 됩니다.

## 핵심 논점

**SSE의 진짜 가치는 "실시간성"이고, 그건 부하가 아니라 동작으로 증명됩니다.** "주문 → 제조중 → 완료가 화면에 실시간으로 뜬다"는 건 브라우저에서 직접 보여주는 데모가 백 배 설득력 있습니다. k6로 50개 커넥션 열어서 "heartbeat 받음"을 세는 건, 정작 사용자가 보는 그 가치를 **증명하지 못합니다.**

그리고 부하 관점에서 의미 있는 단 하나(좀비 emitter 누수 없음)조차도 k6 클라이언트 메트릭으로는 안 보이고, **서버측 active emitter 수**(Grafana/actuator)를 봐야 합니다 — 즉 k6가 아니라 서버 관측의 영역입니다.

    queue.add(System.currentTimeMillis(), memberId); // score = join 시각(ms)
}
```

[EventQueueService.java:47-53](src/main/java/com/whale/order/domain/event/service/EventQueueService.java#L47-L53) — Scheduler가 꺼낼 때:
```java
public Collection<Long> pollNext(Long eventId, int count) {
    // score 오름차순(= 먼저 join 한 순) 으로 앞에서 count명
    Collection<Long> members = queue.valueRange(0, count - 1);
    members.forEach(queue::remove);
    return members;
}
```

`score`가 작을수록(= 먼저 들어온 ms) 앞 순번 → `valueRange(0, count-1)`가 그 순서대로 꺼냅니다.

## 짚어둘 두 가지

1. **"접속 순서"가 아니라 "join 요청 도달 순서"** 입니다. SSE 연결([subscribe](src/main/java/com/whale/order/domain/event/controller/EventQueueController.java#L105))은 순번을 *읽기*만 하지 순서를 정하지 않습니다. 로그인 시각도 무관합니다.

2. **같은 ms 동시 join 시 동점 처리** — score가 같으면 Redis Sorted Set은 value(`memberId`) 사전순으로 정렬합니다. 즉 밀리초 단위로 겹치면 도착 순서가 아니라 memberId 순으로 갈립니다. k6 테스트처럼 20명이 거의 동시에 join하면 이 동점이 자주 발생할 수 있어, "엄밀한 도착 순서"라기보다 "거의 동시 진입"에 가깝습니다.

→ 다만 최종 **구매 가능 인원(capacity)** 은 순번과 무관하게 `findWithLock` 비관적 락이 정확히 막으므로, 순서가 ms 단위로 흔들려도 "정확히 N명만 구매" 검증에는 영향이 없습니다.
