## 왜 이게 "가장 큰 약점"인가

DB 커밋(주문 저장)과 Kafka 발행이 **원자적으로 묶여있지 않기** 때문입니다. `AFTER_COMMIT`은 커밋이 끝난 뒤 발행하므로, 그 사이에 브로커가 죽거나 네트워크가 끊기면 주문만 살아남습니다. 그래서 Q10 답변의 결론인 **Transactional Outbox 패턴**(주문과 같은 트랜잭션으로 outbox 테이블에 이벤트를 저장 → 별도 릴레이가 발행)이 정석 해법인 겁니다.

## "메트릭 증가 = 프로메테우스에서 보고 수동으로 고쳐라"인가?

**절반만 맞습니다.** 지금 이 프로젝트는 "수동으로 고친다"는 맞지만, **"알림이 자동으로 온다"는 아직 아닙니다.**

```yaml
  scrape_interval: 15s

  - job_name: 'spring-boot'  # /actuator/prometheus 스크랩만
```

**alerting rule도, Alertmanager도 설정돼 있지 않습니다.** Prometheus는 15초마다 메트릭을 긁어오고 Grafana가 그림만 그립니다. 즉 현재 구조는:

- ❌ 하지만 **누가 대시보드를 들여다봐야만** 알아챕니다. 슬랙/메일로 자동 알림이 날아오진 않아요.

그래서 정확한 현재 상태는 **"메트릭은 남지만, 사람이 Grafana를 봐야 인지 → 그다음 남은 PENDING 주문을 수동 재처리"** 입니다. `OrderEventListener`의 주석("메트릭 알림 받고 수동 재처리")은 사실 **의도한 그림**이고, 알림 자동화 부분은 아직 안 붙어 있는 셈이죠.

### 추가로 알아둘 함정 하나
이 카운터는 `meterRegistry.counter("order.event.publish.failure")`로 **처음 실패가 나야 등록**됩니다(지연 등록). 그래서 첫 실패 전에는 `/actuator/prometheus`에 이 시계열이 **아예 안 뜹니다(0으로 미리 보이지 않음)**. 나중에 알림 규칙을 걸 때 "값이 없다 = 정상"과 "0 = 정상"을 구분해서 `increase(...) > 0` 형태로 잡아야 합니다.

### 제대로 하려면
1. **Prometheus alert rule** 추가 — 예: `increase(order_event_publish_failure_total[5m]) > 0`
2. **Alertmanager** 연결 → 슬랙/메일 자동 통지
3. 근본 해결은 역시 **Outbox 패턴** (그러면 애초에 유실이 안 생겨 알림 자체가 불필요)

## Transactional Outbox 패턴

한 줄 요약: **"메시지를 Kafka로 바로 쏘지 말고, 주문과 같은 DB 트랜잭션 안에서 outbox라는 테이블에 먼저 적어두고, 그걸 별도 프로세스가 읽어서 Kafka로 발행"** 하는 방식입니다.

### 왜 필요한가 — 지금 문제부터

지금 코드의 문제는 **두 개의 서로 다른 시스템(PostgreSQL + Kafka)에 순차적으로 쓴다**는 것입니다.

```
1. DB 커밋 (주문 저장)   ← PostgreSQL 트랜잭션
2. Kafka 발행            ← 커밋 후, 별개 동작 (AFTER_COMMIT)
```

1번과 2번 사이에 브로커가 죽으면 → 주문만 남고 이벤트는 유실. **이 둘을 하나로 묶을 방법이 없다**는 게 근본 원인이죠. (DB 트랜잭션은 Kafka를 롤백 못 하고, Kafka도 DB를 못 되돌림)

### Outbox의 아이디어

Kafka는 DB 트랜잭션에 못 넣지만, **DB 테이블에 INSERT 하는 건 같은 트랜잭션에 넣을 수 있습니다.** 그래서 발행 대신 "발행할 내용"을 테이블에 적습니다.

```
[결제/주문 트랜잭션] ← 이 안에서 원자적으로 둘 다 커밋
  ├─ orders 테이블에 주문 INSERT
  └─ outbox 테이블에 이벤트 INSERT  (예: {orderId: 42, topic: "order-created"})
        ↓ 커밋 (둘 다 성공 아니면 둘 다 롤백)

[별도 릴레이(Relay)]  ← 스케줄러 or CDC
  1. outbox에서 미발행(published=false) row를 읽음
  2. Kafka로 발행
  3. 성공하면 그 row를 published=true 로 표시 (또는 삭제)
```

핵심은 **주문과 이벤트가 같은 트랜잭션이라 "주문은 있는데 이벤트가 없는" 상태가 원천적으로 불가능**하다는 것. 릴레이가 잠깐 죽어도 outbox row는 DB에 남아있으니, 릴레이가 다시 살아나면 그때 발행합니다. → **유실이 아니라 지연**이 됩니다.

### 지금 코드에 대입하면

| | 현재 | Outbox 적용 후 |
|---|---|---|
| 이벤트 저장 | `@TransactionalEventListener(AFTER_COMMIT)` 로 커밋 **후** 발행 | `pay()` 트랜잭션 **안에서** `outbox` INSERT |
| 발행 실패 시 | 이벤트 영구 유실, 유령 PENDING | outbox에 남아있어 릴레이가 재시도 → 결국 발행됨 |
| 필요한 것 | (없음) | `outbox` 테이블 + 릴레이(스케줄러/Debezium) |

### 릴레이 구현 두 방식

1. **폴링 방식** — `@Scheduled`로 몇 초마다 `SELECT * FROM outbox WHERE published=false` → 발행 → 마킹. 구현 쉬움, 약간의 지연/DB 부하.
2. **CDC 방식 (Debezium)** — DB의 트랜잭션 로그(WAL)를 Debezium이 직접 읽어 Kafka로 흘려보냄. 애플리케이션 코드 손 안 대도 되고 지연 적음. 대신 인프라(Debezium 커넥터) 추가 필요.

### 주의할 점 (면접 대비 포인트)

- Outbox는 **at-least-once**입니다. 릴레이가 발행은 했는데 "발행함" 표시 전에 죽으면 **같은 이벤트를 두 번 발행**할 수 있음. → 그래서 여전히 **Consumer 멱등성(`isStockDeducted()` 체크, 질답 Q9)이 필요**합니다. Outbox가 멱등성을 대체하는 게 아니라 **둘이 짝**입니다.
- "정확히 한 번(exactly-once)"이 아니라 "유실은 없다(at-least-once)"를 보장하는 패턴이라는 걸 정확히 말하면 좋습니다.

## 유휴 연결 끊김(idle connection drop)이란

**"데이터가 한동안 안 흐르는 연결을 중간 장비가 죽은 걸로 판단하고 강제로 끊어버리는 현상"** 입니다.

### 왜 생기나

SSE는 **HTTP 연결을 계속 열어둔 채**, 이벤트가 있을 때만 서버가 찔끔찔끔 데이터를 흘려보내는 구조입니다. 문제는 그 연결 사이에 여러 중간 장비가 끼어 있다는 것:

```
브라우저 ── 프록시/로드밸런서(Nginx, ALB…) ── 서버
```

이 중간 장비들은 "연결을 무한정 열어두면 자원 낭비"라서 **idle timeout**을 갖습니다. 예를 들어 AWS ALB 기본값은 60초, Nginx도 `proxy_read_timeout` 기본 60초. **그 시간 동안 아무 바이트도 안 지나가면** → "이 연결은 죽었거나 방치됐다"고 판단 → **뚝 끊어버립니다.**

주문 상태가 "제조 중"에서 몇 분간 안 바뀌면 그 SSE 연결엔 데이터가 안 흐르죠. 그럼 실제로는 살아있는 연결인데 프록시가 오해해서 끊어버립니다. → 정작 "완료" 이벤트를 보내려는 순간엔 연결이 이미 없는 상태.

### 이 프로젝트의 해결책 — heartbeat

[OrderSseService.java:159-169](src/main/java/com/whale/order/domain/order/service/OrderSseService.java#L159-L169)

```java
/** 25초마다 heartbeat 전송 — 브라우저/프록시의 유휴 연결 끊김 방지 */
@Scheduled(fixedDelay = 25_000)
public void sendAdminHeartbeat() {
    adminEmitters.forEach((clientId, emitter) -> {
        try {
            emitter.send(SseEmitter.event().name("heartbeat").data("ping"));
        } catch (IOException e) {
            adminEmitters.remove(clientId);   // 이미 끊긴 연결이면 정리
        }
    });
}
```

**아무 의미 없는 `heartbeat("ping")` 이벤트를 25초마다 억지로 보냅니다.** 실제 주문 이벤트가 없어도 연결에 주기적으로 바이트가 흐르니, 프록시가 "이 연결은 살아있다"고 인식해서 안 끊습니다.

### 왜 하필 25초?

프록시 idle timeout(보통 60초)보다 **넉넉히 짧게** 잡아야 합니다. 60초 timeout인데 heartbeat를 55초마다 보내면 지연이 겹칠 때 아슬아슬하죠. 25초면 "타임아웃 도달 전에 최소 2번은 신호가 간다"가 보장돼 안전 마진이 큽니다.

### 덤 — heartbeat가 죽은 연결 청소도 함

`send`가 `IOException`을 던지면(= 브라우저가 이미 탭을 닫았거나 진짜로 끊긴 상태) `adminEmitters.remove()`로 맵에서 정리합니다. 즉 heartbeat는 **연결 유지 + 죽은 연결 감지**를 겸합니다. Q14에서 말한 "메모리 누수 방지"의 일부죠.

### 면접용 한 줄 정리

> "SSE는 오래 열어두는 연결이라 프록시·LB의 idle timeout(보통 60초)에 걸려 끊깁니다. 그래서 25초마다 heartbeat 이벤트를 흘려보내 연결에 트래픽을 유지하고, 동시에 전송 실패하는 연결은 그 자리에서 정리합니다."

한 가지 남은 약점: 지금 heartbeat는 `adminEmitters`(어드민 브로드캐스트용)**에만** 걸려 있습니다. 고객용 `emitters`(주문 결과)는 5분 타임아웃이라 그 안에 결과가 오면 괜찮지만, 상태 스트림 `statusEmitters`(30분)는 heartbeat가 없어서 장시간 상태 변화가 없으면 프록시에 끊길 수 있습니다. 면접에서 "모든 스트림에 heartbeat가 있나요?" 꼬리질문이 오면 이 점을 정직하게 짚으면 좋습니다. 이 내용도 질답.md Q14에 보충해둘까요?

## ACID = 트랜잭션이 지켜야 할 4가지 성질

트랜잭션 = "여러 작업을 하나의 논리적 단위로 묶은 것" (예: A계좌 출금 + B계좌 입금 = 한 트랜잭션)

| 글자 | 이름 | 의미 | WhaleOrder 예시 |
|------|------|------|------|
| **A** | Atomicity (원자성) | 다 되거나, 다 안 되거나. 중간 상태 없음 | 결제 성공 + 주문 저장이 **둘 다** 되거나 **둘 다** 롤백 |
| **C** | Consistency (일관성) | 트랜잭션 끝나도 DB 규칙(제약조건)이 안 깨짐 | 재고가 음수가 되는 상태로는 커밋 안 됨 |
| **I** | Isolation (격리성) | 동시에 도는 트랜잭션이 서로 간섭 안 함 | 두 명이 동시에 재고 차감해도 lost update 안 남 (Q1의 락·`@Version`) |
| **D** | Durability (지속성) | 커밋되면 서버가 죽어도 안 사라짐 | 결제 완료된 주문은 재부팅해도 DB에 남음 |

## 왜 "RDBMS의 특성"처럼 말하나

이 프로젝트 CLAUDE.md의 저장소 분리 기준이 바로 이겁니다:

> **주문·결제·회원 → PostgreSQL** (데이터 무결성이 중요, 엄격한 ACID 트랜잭션 필요)
> **장바구니·세션 → Redis** (수명 짧고 속도가 중요)

PostgreSQL 같은 RDBMS는 **ACID를 강하게 보장**하는 게 설계 목표라, "돈"처럼 절대 틀리면 안 되는 데이터를 여기 둡니다. 그래서 실무에서 "ACID = RDBMS 특성"이라고 뭉뚱그려 말하는 거예요. 하지만 엄밀히는:

- **트랜잭션의 성질**이지, DB 자체의 물리적 속성이 아님
- **모든 DB가 ACID를 주는 게 아님** — Redis, MongoDB 같은 NoSQL은 전통적으로 ACID를 **약하게** 보장하거나 일부만 지원 (속도·확장성을 위해 일관성을 양보 → 대신 **BASE**라는 대조 개념을 씀)

## 면접에서 조심할 점 (정확도)

- "Redis는 ACID가 없다"는 **틀린 말**입니다. Redis도 단일 명령은 원자적이고, `MULTI/EXEC`나 Lua 스크립트로 어느 정도 원자성을 줍니다. 다만 **RDBMS 수준의 완전한 트랜잭션(롤백, 격리 수준 선택 등)은 아님**이라고 말하는 게 정확합니다.
- MongoDB도 4.0부터 멀티도큐먼트 트랜잭션(ACID)을 지원합니다. "NoSQL = ACID 없음"은 옛날 얘기예요.

**정리 한 줄:** ACID는 "DB의 특성"이 아니라 **"(주로 RDBMS) 트랜잭션이 보장하는 4가지 성질"** 이고, 이 프로젝트가 주문·결제를 PostgreSQL에 둔 이유가 바로 이 ACID 보장이 필요해서입니다.

## SSE와 Redis는 담당 구간이 다르다

```
브라우저  ◀── SSE ──▶  서버      ◀── ??? ──▶  다른 서버
        (이건 Redis로     (여기 서버끼리
         못 바꿈)          통신을 Redis가 담당)
```

- **SSE** = 브라우저 ↔ 서버 구간의 **전송 프로토콜**. 브라우저는 Redis에 직접 연결 못 합니다(Redis는 백엔드 전용). 그러니 브라우저에게 실시간 푸시를 하려면 **SSE(또는 WebSocket)는 무조건 남아 있어야** 합니다.
- **Redis Pub/Sub** = 서버 ↔ 서버 구간의 **메시지 중계**. SSE를 대체하는 게 아니라 **SSE 뒤에 추가로 붙이는** 것.

즉 정확한 표현은 **"SSE를 Redis로 바꾼다"가 아니라 "SSE는 그대로 두고, 서버 간 연결을 Redis Pub/Sub으로 잇는다"** 입니다.

## 왜 Redis가 필요한가 — 지금 코드의 한계 (Q15)

지금은 emitter를 **인메모리**로 들고 있습니다:

[OrderSseService.java:21,34](src/main/java/com/whale/order/domain/order/service/OrderSseService.java#L21)
```java
private final ConcurrentHashMap<Long, SseEmitter> emitters = ...;
private final ConcurrentHashMap<String, SseEmitter> adminEmitters = ...;
```

`ConcurrentHashMap`은 **그 JVM 프로세스 안에만** 있습니다. 서버가 1대일 땐 문제없지만 2대가 되면:

```
[워커가 A서버에서 재고차감 완료]
    → A서버의 adminEmitters.forEach(...)로 broadcast
    → 근데 관리자 브라우저는 B서버에 SSE 연결돼 있음
    → A서버 맵엔 그 emitter가 없음 → 푸시 유실 ❌
```

**"이벤트를 만든 서버"와 "브라우저가 붙은 서버"가 다르면** 알림이 사라집니다.

## Redis Pub/Sub으로 해결하는 그림

```
[A서버] 워커 재고차감 완료
   └─▶ redis.publish("order-events", {orderId, ...})   ← 채널에 던짐
              │
     ┌────────┴────────┐  모든 서버가 구독 중
     ▼                 ▼
[A서버 구독]       [B서버 구독]
  내 맵에 있는       내 맵에 있는
  emitter에만 전송    emitter에만 전송  ← 관리자가 여기 붙어있음 ✅
```

핵심 로직: **"이벤트를 Redis 채널에 publish → 모든 인스턴스가 구독하다가, 자기한테 붙은 emitter에만 실제 SSE로 흘려보낸다."** 브로드캐스트가 한 서버 안이 아니라 **전체 클러스터**로 퍼지는 거죠.

## 정리

| | 지금 (인메모리) | Redis Pub/Sub 추가 후 |
|---|---|---|
| 브라우저↔서버 전송 | **SSE** | **SSE** (그대로) |
| 서버↔서버 전파 | 없음 (단일 서버 전제) | **Redis Pub/Sub** |
| 서버 2대 이상 | 푸시 유실 | 정상 동작 |

**면접용 한 줄:** "SSE는 브라우저-서버 전송이라 못 바꾸고, 인메모리 emitter 맵이 단일 인스턴스 전제라는 게 한계입니다. 스케일아웃하려면 이벤트를 **Redis Pub/Sub**으로 전 인스턴스에 뿌리고, 각 서버는 자기한테 붙은 SSE 연결에만 흘려보내면 됩니다. Redis가 SSE를 대체하는 게 아니라 그 **뒤단**을 맡는 구조죠."

참고로 규모가 더 커지면 Redis Pub/Sub 대신 **STOMP/WebSocket + 전용 메시지 브로커**(RabbitMQ 등)로 가는 것도 선택지입니다(Q15에도 언급돼 있음). Redis Pub/Sub은 구독자가 그 순간 없으면 메시지가 사라지는(fire-and-forget) 특성이 있어서, 더 강한 보장이 필요하면 브로커를 쓰는 거예요.

```
                        [보상: 스스로 주문/결제 취소]
```
**중앙 통제자가 없습니다.** 각 단계가 이벤트를 발행하면, 다음 단계가 그 이벤트를 **구독하다가 알아서 반응**합니다. "누가 시켜서"가 아니라 "이벤트가 왔으니까" 움직이는 구조.

### 오케스트레이션 (규모 커지면 도입할 방식)
```
        ┌─────────────────────────┐
        │   OrderSagaManager       │ ← 지휘자
        │   (흐름 전체를 알고 지시)  │
        └─────────────────────────┘
          │ 1.결제해   │ 2.재고차감해   │ 3.실패? 그럼 보상해
          ▼            ▼               ▼
       결제서비스    재고서비스      (지휘자가 순서대로 취소 명령)
```
**중앙에 SagaManager라는 지휘자**가 있어서, 각 서비스에 명령을 내리고 응답을 받아 다음 스텝을 결정합니다. 서비스들은 서로를 모르고, 지휘자만 전체 흐름을 압니다.

## 이 프로젝트 코드에 대입

지금은 **코레오그래피**입니다 (Q7):

```
PaymentService.pay()
  → @TransactionalEventListener로 order-created 이벤트 발행   ← "발행만" 함
       → OrderKafkaConsumer가 구독하다 반응 → process()          ← "구독해서" 반응
            → 실패 3회 → order-created.DLT로 이동
                 → DLT Consumer가 구독하다 반응 → compensate()   ← 또 "구독해서" 보상
```

[OrderKafkaConsumer.java:24,38](src/main/java/com/whale/order/domain/order/service/OrderKafkaConsumer.java#L24) — `@KafkaListener`로 이벤트를 **구독**해서 각자 반응하죠. **"전체 주문 사가를 관장하는 클래스"가 없습니다.** `PaymentService`는 재고 서비스의 존재조차 몰라요. 그냥 이벤트만 던지고, 재고 쪽이 알아서 주워갑니다.

## 트레이드오프 (면접 포인트)

| | 코레오그래피 | 오케스트레이션 |
|---|---|---|
| 통제자 | 없음 (이벤트 기반) | 중앙 Orchestrator |
| 결합도 | 낮음 (서비스끼리 서로 모름) | 지휘자가 모두를 앎 |
| 장점 | 단순, 서비스 추가 쉬움 | **흐름이 한눈에 보임**, 디버깅·추적 쉬움 |
| 단점 | **단계 많아지면 흐름 추적 어려움** ("이 이벤트 다음에 뭐가 도는지" 코드에 흩어져 있음) | 지휘자가 복잡해지고 단일 장애점 될 수 있음 |
| 적합 | 단계 적을 때 | 단계 많고 복잡할 때 |

이게 질답.md 압박 꼬리질문의 그 답변입니다:

> "Choreography Saga는 단계 많아지면 추적이 어렵지 않나?"
> → "그래서 지금은 단계가 적어 적합하지만, **늘어나면 명시적 Orchestrator(SagaManager) 도입이 맞다**고 본다."

## 왜 "추적이 어렵다"가 코레오그래피의 약점인가

지금 코드에서 "주문이 실패하면 무슨 일이 벌어지나?"를 알려면:
- `PaymentService`(발행) → `OrderKafkaConsumer`(구독) → `KafkaConfig`(DLT 라우팅) → `OrderProcessingService.compensate()`

**흐름이 4개 파일에 흩어져 있습니다.** 이벤트를 따라 코드를 점프해야 전체 그림이 보이죠. 단계가 2~3개일 땐 괜찮지만, 10단계가 되면 "지금 어디서 뭐가 도는지" 추적이 지옥이 됩니다. 오케스트레이션이면 SagaManager 한 파일만 읽으면 전체 흐름이 순서대로 보여서 이 문제가 없어요.

**한 문장 정리:** 코레오그래피는 "이벤트 보고 각자 알아서(분산 통제)", 오케스트레이션은 "지휘자가 시키는 대로(중앙 통제)". 이 프로젝트는 단계가 적어 코레오그래피를 택했고, 복잡해지면 Orchestrator로 전환하는 게 정석입니다.
