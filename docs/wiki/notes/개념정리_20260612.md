## 1. Spring Cache — @Cacheable / @CacheEvict

### 개념

DB 조회 결과를 Redis에 저장해두고, 같은 요청이 오면 DB를 거치지 않고 Redis에서 바로 반환한다.

```
캐시 없음                     캐시 있음
──────────────────────        ──────────────────────
요청 → DB 조회 → 응답          요청 → Redis 확인
요청 → DB 조회 → 응답              ├─ 있으면: Redis에서 반환 (DB 생략)
요청 → DB 조회 → 응답              └─ 없으면: DB 조회 → Redis 저장 → 응답
```

### 주요 어노테이션

```java
// 조회 — 결과를 캐시에 저장, 이후 같은 key면 캐시 반환
@Cacheable(cacheNames = "menus", key = "#category != null ? #category.name() : 'all'")
public List<MenuResponse> getMenus(MenuCategory category) { ... }

// 단건 조회
@Cacheable(cacheNames = "menu", key = "#menuId")
public MenuDetailResponse getMenu(Long menuId) { ... }

// 수정/삭제 — 해당 캐시 무효화
@CacheEvict(cacheNames = "menus", allEntries = true)
public MenuResponse updateMenu(Long menuId, ...) { ... }

// 여러 캐시 동시 처리
@Caching(evict = {
    @CacheEvict(cacheNames = "menus", allEntries = true),
    @CacheEvict(cacheNames = "menu",  key = "#menuId")
})
public void deleteMenu(Long menuId) { ... }
```

### key 표현식 해석

```java
key = "#category != null ? #category.name() : 'all'"
```

| 호출 | key 값 | 캐시 키 |
|------|--------|---------|
| `getMenus(BEVERAGE)` | `"BEVERAGE"` | `menus::BEVERAGE` |
| `getMenus(null)` | `"all"` | `menus::all` |

→ 카테고리별로 다른 캐시를 가짐. 전체 조회와 음료 조회가 서로 독립적.

## 2. Redis CacheManager 설정 — Jackson 직렬화 문제

### 문제 1: Java record는 역직렬화가 안 된다

`GenericJackson2JsonRedisSerializer`는 `@class` 타입 정보를 JSON에 포함시켜서 역직렬화한다.  
하지만 Java record는 `final` 클래스라 `NON_FINAL` 기본 타입 설정에서 `@class`가 삽입되지 않는다.

```
저장 시: {"name":"아메리카노","basePrice":4500}       ← @class 없음
복원 시: @class 없어서 타입을 모름 → LinkedHashMap으로 역직렬화
결과:    ClassCastException: LinkedHashMap cannot be cast to MenuDetailResponse
```

### 해결: 캐시별 타입을 명시한 Jackson2JsonRedisSerializer 사용

```java
// CacheConfig.java
RedisCacheConfiguration menuConfig = RedisCacheConfiguration.defaultCacheConfig()
    .entryTtl(Duration.ofHours(6))
    .serializeValuesWith(
        RedisSerializationContext.SerializationPair.fromSerializer(
            new Jackson2JsonRedisSerializer<>(objectMapper,
                objectMapper.getTypeFactory()
                    .constructCollectionType(List.class, MenuResponse.class))
        )
    );
```

| 직렬화 방식 | 장점 | 단점 |
|---|---|---|
| `GenericJackson2JsonRedisSerializer` | 타입 자동 처리 | record/final 클래스 역직렬화 실패 |
| `Jackson2JsonRedisSerializer<T>` | 타입 명시 → record 가능 | 캐시마다 설정 필요 |

### 문제 2: LocalDate/LocalDateTime 직렬화 실패

Jackson 기본 설정은 Java 8 날짜 타입을 처리하지 못한다.

```java
// ObjectMapper에 JavaTimeModule 등록 필요
objectMapper.registerModule(new JavaTimeModule());
objectMapper.disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS);
// 결과: "2026-06-12" (문자열) 로 저장됨
```

### 캐시 무효화 전략 (FLUSHALL)

직렬화 방식을 바꾸면 기존 Redis에 저장된 데이터 형식이 맞지 않아 역직렬화 오류 발생.  
설정 변경 후 반드시 Redis 캐시를 초기화해야 한다.

```bash
# Redis CLI 접속 후
FLUSHALL
```

## 3. 캐시 적용 기준 — 무엇을 캐시하고 무엇을 하지 않나

### 캐시 적합한 데이터

```
조건: 자주 읽히고, 변경이 드물고, 실시간성이 필요 없는 데이터

예시
  - 메뉴 목록/상세 (관리자 조회) → 메뉴 수정할 때 캐시 무효화
  - 매장 정보 (이름, 주소, 영업시간)
```

### 캐시 하면 안 되는 데이터

```
조건: 실시간으로 정확한 값이 필요한 데이터

예시
  - 재고 수량 → 캐시되면 "품절인데 주문 가능"처럼 보일 수 있음
  - 주문 상태 → 항상 최신 상태 필요
```

### 이 프로젝트 캐시 TTL 설계

```
menus (목록) : 6시간 — 메뉴 추가/수정/삭제 시 @CacheEvict로 즉시 무효화
menu  (단건) : 6시간 — 동일
```

TTL이 길어도 괜찮은 이유: 수정할 때마다 `@CacheEvict`로 직접 무효화하기 때문에  
TTL은 "마지막 안전장치"로만 동작한다.

## 4. 재고 + 메뉴 통합 조회 — JOIN vs N+1

### N+1 문제

```
메뉴 7개 조회 시
  쿼리 1: SELECT * FROM menu          (1번)
  쿼리 2: SELECT * FROM stock WHERE menu_id = 1  (7번)
  쿼리 3: SELECT * FROM menu_option WHERE menu_id = 1  (7번)
  총 15번 쿼리 발생
```

### 해결: LEFT JOIN + 배치 조회

```java
// MenuRepository — 메뉴 + 재고 한 번에
@Query("""
    SELECT m, s FROM Menu m
    LEFT JOIN Stock s ON s.menu = m AND s.store.storeId = :storeId
    WHERE m.isActive = true ORDER BY m.createdAt DESC
    """)
List<Object[]> findActiveMenusWithStock(@Param("storeId") Long storeId);

// MenuOptionRepository — 옵션 배치 조회 (IN절로 한 번에)
@Query("SELECT o FROM MenuOption o WHERE o.menu.menuId IN :menuIds")
List<MenuOption> findByMenuIds(@Param("menuIds") List<Long> menuIds);
```

```
개선 후
  쿼리 1: 메뉴 + 재고 LEFT JOIN  (1번)
  쿼리 2: 옵션 IN (1,2,3,4,5,6,7)  (1번)
  총 2번 쿼리
```

### LEFT JOIN을 쓰는 이유

```
INNER JOIN: 재고가 없는 메뉴는 결과에서 제외됨
LEFT JOIN:  재고가 없어도 메뉴는 노출 (재고 컬럼이 null로 옴)
            → 재고 미설정 메뉴도 정상 표시 가능
```

### 고객용 메뉴는 캐시하지 않는 이유

```
관리자 메뉴 조회 → 캐시 O (재고 정보 없음, 변경 빈도 낮음)
고객 메뉴 조회  → 캐시 X (재고 수량 포함, 실시간 품절 여부 반영 필요)
```

## 5. Micrometer 메트릭 타입 — Gauge vs Counter

### Gauge (현재값)

지금 이 순간의 값을 측정한다. Prometheus가 스크랩할 때마다 측정 함수를 호출한다.

```java
Gauge.builder("order.queue.size", orderRepository,
        repo -> (double) repo.countByStatus(OrderStatus.PENDING))
    .register(meterRegistry);
```

```
특징
- 올라가고 내려갈 수 있음
- 스크랩 시점에 측정 함수 실행
- 서버 재시작해도 현재 상태를 즉시 반영
- 예: 대기열 크기, 현재 접속자 수, DB 레코드 수
```

### Counter (누적값)

단조 증가하는 누적 횟수를 기록한다. 절대 내려가지 않는다.

```java
Counter.builder("order.created.total")
    .register(meterRegistry)
    .increment();
```

```
특징
- 항상 증가만 함 (서버 재시작 시 0으로 리셋)
- increment() 호출 시점에 값이 쌓임
- 예: 총 주문 수, 에러 발생 횟수, 재고 부족 횟수
```

### 핵심 차이

```
Gauge  → "지금 몇 개야?"   → order_queue_size = 5
Counter → "지금까지 몇 번?" → order_created_total = 847
```

## 6. Counter 사전 등록 — @PostConstruct

### 문제

Counter는 `increment()`가 처음 호출될 때 Prometheus에 등록된다.  
주문이 한 건도 없으면 `/actuator/prometheus`에 메트릭 자체가 없다.

```java
// 잘못된 방식 — 호출 시점에 등록됨
public void createOrder(...) {
    Counter.builder("order.created.total").register(meterRegistry).increment();
}
```

### 해결

`@PostConstruct`로 서버 시작 시점에 미리 등록한다.

```java
private Counter orderCreatedCounter;

@PostConstruct
public void initMetrics() {
    orderCreatedCounter = Counter.builder("order.created.total")
        .description("누적 주문 생성 수")
        .register(meterRegistry);  // 주문 0건이어도 바로 노출
}

public void createOrder(...) {
    orderCreatedCounter.increment();  // 필드로 재사용
}
```

```
장점
- 서버 시작 직후부터 /actuator/prometheus에 노출
- Counter 객체를 매번 생성하지 않아 성능도 좋음
- Grafana 패널에서 "no data" 없이 0부터 시작
```

## 7. Prometheus 수집 구조와 지연

### 스크랩 흐름

```
Spring Boot ←── Prometheus (15초마다 pull) ←── Grafana (패널 주기마다 pull)

Spring Boot: 값이 즉시 바뀜
Prometheus: 최대 15초 후 반영
Grafana:    Prometheus 갱신 + 패널 새로고침 주기 후 반영
```

### Gauge는 왜 지연이 생기나

Gauge는 lazy 측정이다. Prometheus가 스크랩할 때만 측정 함수를 실행한다.

```
주문 완료 처리 → DB 즉시 변경
            ↓ (최대 15초 대기)
Prometheus 스크랩 → countByStatus(PENDING) 실행 → 결과 저장
            ↓ (Grafana 새로고침)
Grafana 패널 업데이트
```

→ 버그가 아니라 정상 동작이다.

## 8. PromQL 핵심 함수

### 프로젝트에서 쓰는 메트릭 전체

| 메트릭 | 타입 | Prometheus 이름 |
|--------|------|-----------------|
| 대기 주문 수 | Gauge | `order_queue_size` |
| 누적 주문 수 | Counter | `order_created_total` |
| 주문 처리 성공/실패 | Counter | `order_processed_total{result="success/failure"}` |
| 주문 처리 시간 | Timer | `order_processing_time_seconds_*` |
| 재고 부족 | Counter | `stock_shortage_total_total` |
| Kafka DLT | Counter | `kafka_dlt_received_total` |

### 자주 쓰는 PromQL 패턴

```promql
# 현재 대기 주문 수 (Gauge 직접 조회)
order_queue_size

# 누적 주문 수 (Counter 직접 조회 — 서버 재시작 전까지 누적)
order_created_total

# 지난 1시간 주문 수 증가량
increase(order_created_total[1h])

# 초당 주문 생성 TPS
rate(order_created_total[1m])

# 5분 단위 주문 건수 (Time series 패널용)
rate(order_created_total[5m]) * 300

# 주문 처리 평균 소요 시간
rate(order_processing_time_seconds_sum[5m])
/ rate(order_processing_time_seconds_count[5m])
```

### Grafana 패널 선택 기준

| 목적 | 패널 타입 |
|------|-----------|
| 숫자 하나 크게 표시 (현재 대기 수, 오늘 주문 수) | **Stat** |
| 시간대별 추이, TPS 그래프 | **Time series** |
| 매장별 주문 수 비교 | **Bar chart** |

## 9. 로깅 전략

### 로그의 두 가지 목적

```
기록 (로그파일) : 나중에 문제 생겼을 때 원인 추적
인지 (알람)    : 지금 당장 이상한 게 생겼을 때 즉시 알림
```

→ 둘 다 있어야 한다. 로그만으로는 이상 징후를 놓칠 수 있다.

### 이 프로젝트에서 로그를 남기는 위치

```
OrderService        → [주문생성] [주문취소] [상태변경]
StockService        → [재고차감] [재고복구] [재고부족]
StockLockFacade     → [분산락] 락 획득 실패
OrderProcessingService → (이미 존재)
OrderKafkaConsumer  → (이미 존재)
RequestLoggingFilter → [API] HTTP 요청/응답 소요시간
```

### 로그 레벨 선택 기준

```
ERROR : 즉시 대응 필요 (서버 오류, 처리되지 않은 예외)
WARN  : 비정상이지만 서비스 지속 가능 (재고 부족, 락 획득 실패, 중복 요청)
INFO  : 정상 비즈니스 흐름 (주문 생성, 상태 변경, 재고 차감)
DEBUG : 개발 중 상세 추적 (SQL, 파라미터 바인딩)
```

### logback-spring.xml 구조

```
로그 파일 분리
  logs/order.log  → 전체 로그 (INFO 이상)
  logs/biz.log    → 비즈니스 흐름만 (OrderService, StockService 등)

장점: 장애 발생 시 biz.log만 열면 주문 흐름을 빠르게 추적 가능
```

## 10. HTTP 요청 로깅 — Filter vs Interceptor vs AOP

### 비교

```
Filter       : Servlet 앞단, Spring 컨텍스트 밖. 모든 요청 처리 가능
Interceptor  : Spring MVC 안, Controller 앞단. Spring Bean 접근 가능
AOP          : 메서드 레벨. 서비스 계층 로깅에 적합
```

### 이 프로젝트 선택 이유

HTTP 요청 로그는 `Filter`가 적합하다.
- Spring Security 필터 체인과 같은 레벨에서 동작
- 인증 실패 요청도 로그에 남음
- URI, 메서드, 상태코드, 소요시간 한 줄로 기록 가능

```java
// SSE, actuator 등 노이즈 경로는 제외
if (uri.contains("/stream") || uri.contains("/actuator")) {
    chain.doFilter(request, response);
    return;
}
// [API] POST /api/orders → 202 (143ms)
log.info("[API] {} {} → {} ({}ms)", method, uri, status, elapsed);
```

## 11. k6 부하 테스트

### 개념

```
부하 테스트 도구 비교
  JMeter  : GUI 기반, 설정 복잡, Java
  Gatling : 코드 기반, Scala, 리포트 상세
  k6      : 코드 기반, JavaScript, 가볍고 빠름 ← 선택
  Locust  : Python 기반
```

### k6 핵심 개념

```
VU (Virtual User) : 가상 유저. 동시 접속자 수
iteration         : 각 VU가 시나리오를 실행하는 횟수
executor          : 시나리오 실행 방식
  - ramping-vus       : 점진적 부하 증가
  - per-vu-iterations : VU마다 N번씩 실행 (동시성 테스트에 적합)
```

### setup() 함수의 역할

```javascript
export function setup() {
    // 테스트 시작 전 1회만 실행
    // 반환값이 default()에 data로 전달됨
    return { tokens };
}

export default function ({ tokens }) {
    // 각 VU가 실행하는 메인 시나리오
}
```

→ 테스트 계정 생성, 토큰 발급 등 사전 준비 작업을 여기서 처리한다.

### 동시성 테스트 설계 핵심

```
잘못된 설계                       올바른 설계
────────────────────────────      ────────────────────────────
100 VU가 같은 계정 공유            100 VU가 각자 다른 계정 사용
  → 같은 장바구니 충돌              → 독립적인 장바구니
  → 멱등성 키 동일 → 1건만 처리    → 멱등성 키 각자 다름
  → 성공 1건                       → 재고만큼 성공
```

### 멱등성 키와 동시성 테스트의 관계

이 프로젝트의 멱등성 키 구성:

```
memberId + storeId + orderType + customerRequest + 장바구니내용
```

같은 계정 + 같은 장바구니 → 멱등성 키 동일 → 100명 요청해도 1건만 처리  
→ 동시성 테스트 목적과 반대 결과가 나옴

### 결과 해석

```
재고 10개 설정 후 20명 동시 주문 결과

✅ 주문 성공: 10건   → 정상 (분산락이 초과 차감 방지)
❌ 주문 실패: 10건   → 정상 (재고 부족으로 거절)

성공 > 10건이면 분산락 오류 → 재고가 실제보다 많이 차감됨
```

## 12. Docker에서 localhost 접근

```
문제: Docker 컨테이너 안에서 localhost = 컨테이너 자신
       → 호스트 머신의 Spring Boot 서버에 접근 불가

해결: host.docker.internal 사용
       → 컨테이너 → 호스트 머신 접근 가능 (Windows/Mac 기본 지원)
```

```javascript
// k6 Docker 실행 시
const BASE_URL = 'http://host.docker.internal:8080';

// Prometheus docker-compose 설정도 동일
targets: ['host.docker.internal:8080']
```

### Gauge (현재값)

지금 이 순간의 값을 측정한다. Prometheus가 스크랩할 때마다 측정 함수를 호출한다.

```java
Gauge.builder("order.queue.size", orderRepository,
        repo -> (double) repo.countByStatus(OrderStatus.PENDING))
    .register(meterRegistry);
```

```
특징
- 올라가고 내려갈 수 있음
- 스크랩 시점에 측정 함수 실행
- 서버 재시작해도 현재 상태를 즉시 반영
- 예: 대기열 크기, 현재 접속자 수, DB 레코드 수
```

### Counter (누적값)

단조 증가하는 누적 횟수를 기록한다. 절대 내려가지 않는다.

```java
Counter.builder("order.created.total")
    .register(meterRegistry)
    .increment();
```

```
특징
- 항상 증가만 함 (서버 재시작 시 0으로 리셋)
- increment() 호출 시점에 값이 쌓임
- 예: 총 주문 수, 에러 발생 횟수, 재고 부족 횟수
```

### 핵심 차이

```
Gauge  → "지금 몇 개야?"   → order_queue_size = 5
Counter → "지금까지 몇 번?" → order_created_total = 847
```

## 2. Counter 사전 등록 — @PostConstruct

### 문제

Counter는 `increment()`가 처음 호출될 때 Prometheus에 등록된다.  
주문이 한 건도 없으면 `/actuator/prometheus`에 메트릭 자체가 없다.

```java
// 잘못된 방식 — 호출 시점에 등록됨
public void createOrder(...) {
    Counter.builder("order.created.total").register(meterRegistry).increment();
}
```

### 해결

`@PostConstruct`로 서버 시작 시점에 미리 등록한다.

```java
private Counter orderCreatedCounter;

@PostConstruct
public void initMetrics() {
    orderCreatedCounter = Counter.builder("order.created.total")
        .description("누적 주문 생성 수")
        .register(meterRegistry);  // 주문 0건이어도 바로 노출
}

public void createOrder(...) {
    orderCreatedCounter.increment();  // 필드로 재사용
}
```

```
장점
- 서버 시작 직후부터 /actuator/prometheus에 노출
- Counter 객체를 매번 생성하지 않아 성능도 좋음
- Grafana 패널에서 "no data" 없이 0부터 시작
```

## 3. Prometheus 수집 구조와 지연

### 스크랩 흐름

```
Spring Boot ←── Prometheus (15초마다 pull) ←── Grafana (패널 주기마다 pull)

Spring Boot: 값이 즉시 바뀜
Prometheus: 최대 15초 후 반영
Grafana:    Prometheus 갱신 + 패널 새로고침 주기 후 반영
```

### Gauge는 왜 지연이 생기나

Gauge는 lazy 측정이다. Prometheus가 스크랩할 때만 측정 함수를 실행한다.

```
주문 완료 처리 → DB 즉시 변경
            ↓ (최대 15초 대기)
Prometheus 스크랩 → countByStatus(PENDING) 실행 → 결과 저장
            ↓ (Grafana 새로고침)
Grafana 패널 업데이트
```

→ 버그가 아니라 정상 동작이다.

## 4. PromQL 핵심 함수

### 프로젝트에서 쓰는 메트릭 전체

| 메트릭 | 타입 | Prometheus 이름 |
|--------|------|-----------------|
| 대기 주문 수 | Gauge | `order_queue_size` |
| 누적 주문 수 | Counter | `order_created_total` |
| 주문 처리 성공/실패 | Counter | `order_processed_total{result="success/failure"}` |
| 주문 처리 시간 | Timer | `order_processing_time_seconds_*` |
| 재고 부족 | Counter | `stock_shortage_total_total` |
| Kafka DLT | Counter | `kafka_dlt_received_total` |

### 자주 쓰는 PromQL 패턴

```promql
# 현재 대기 주문 수 (Gauge 직접 조회)
order_queue_size

# 누적 주문 수 (Counter 직접 조회 — 서버 재시작 전까지 누적)
order_created_total

# 지난 1시간 주문 수 증가량
increase(order_created_total[1h])

# 초당 주문 생성 TPS
rate(order_created_total[1m])

# 5분 단위 주문 건수 (Time series 패널용)
rate(order_created_total[5m]) * 300

# 주문 처리 평균 소요 시간
rate(order_processing_time_seconds_sum[5m])
/ rate(order_processing_time_seconds_count[5m])
```

### Grafana 패널 선택 기준

| 목적 | 패널 타입 |
|------|-----------|
| 숫자 하나 크게 표시 (현재 대기 수, 오늘 주문 수) | **Stat** |
| 시간대별 추이, TPS 그래프 | **Time series** |
| 매장별 주문 수 비교 | **Bar chart** |

## 5. 로깅 전략

### 로그의 두 가지 목적

```
기록 (로그파일) : 나중에 문제 생겼을 때 원인 추적
인지 (알람)    : 지금 당장 이상한 게 생겼을 때 즉시 알림
```

→ 둘 다 있어야 한다. 로그만으로는 이상 징후를 놓칠 수 있다.

### 이 프로젝트에서 로그를 남기는 위치

```
OrderService        → [주문생성] [주문취소] [상태변경]
StockService        → [재고차감] [재고복구] [재고부족]
StockLockFacade     → [분산락] 락 획득 실패
OrderProcessingService → (이미 존재)
OrderKafkaConsumer  → (이미 존재)
RequestLoggingFilter → [API] HTTP 요청/응답 소요시간
```

### 로그 레벨 선택 기준

```
ERROR : 즉시 대응 필요 (서버 오류, 처리되지 않은 예외)
WARN  : 비정상이지만 서비스 지속 가능 (재고 부족, 락 획득 실패, 중복 요청)
INFO  : 정상 비즈니스 흐름 (주문 생성, 상태 변경, 재고 차감)
DEBUG : 개발 중 상세 추적 (SQL, 파라미터 바인딩)
```

### logback-spring.xml 구조

```
로그 파일 분리
  logs/order.log  → 전체 로그 (INFO 이상)
  logs/biz.log    → 비즈니스 흐름만 (OrderService, StockService 등)

장점: 장애 발생 시 biz.log만 열면 주문 흐름을 빠르게 추적 가능
```

## 6. HTTP 요청 로깅 — Filter vs Interceptor vs AOP

### 비교

```
Filter       : Servlet 앞단, Spring 컨텍스트 밖. 모든 요청 처리 가능
Interceptor  : Spring MVC 안, Controller 앞단. Spring Bean 접근 가능
AOP          : 메서드 레벨. 서비스 계층 로깅에 적합
```

### 이 프로젝트 선택 이유

HTTP 요청 로그는 `Filter`가 적합하다.
- Spring Security 필터 체인과 같은 레벨에서 동작
- 인증 실패 요청도 로그에 남음
- URI, 메서드, 상태코드, 소요시간 한 줄로 기록 가능

```java
// SSE, actuator 등 노이즈 경로는 제외
if (uri.contains("/stream") || uri.contains("/actuator")) {
    chain.doFilter(request, response);
    return;
}
// [API] POST /api/orders → 202 (143ms)
log.info("[API] {} {} → {} ({}ms)", method, uri, status, elapsed);
```

## 7. k6 부하 테스트

### 개념

```
부하 테스트 도구 비교
  JMeter  : GUI 기반, 설정 복잡, Java
  Gatling : 코드 기반, Scala, 리포트 상세
  k6      : 코드 기반, JavaScript, 가볍고 빠름 ← 선택
  Locust  : Python 기반
```

### k6 핵심 개념

```
VU (Virtual User) : 가상 유저. 동시 접속자 수
iteration         : 각 VU가 시나리오를 실행하는 횟수
executor          : 시나리오 실행 방식
  - ramping-vus       : 점진적 부하 증가
  - per-vu-iterations : VU마다 N번씩 실행 (동시성 테스트에 적합)
```

### setup() 함수의 역할

```javascript
export function setup() {
    // 테스트 시작 전 1회만 실행
    // 반환값이 default()에 data로 전달됨
    return { tokens };
}

export default function ({ tokens }) {
    // 각 VU가 실행하는 메인 시나리오
}
```

→ 테스트 계정 생성, 토큰 발급 등 사전 준비 작업을 여기서 처리한다.

### 동시성 테스트 설계 핵심

```
잘못된 설계                       올바른 설계
────────────────────────────      ────────────────────────────
100 VU가 같은 계정 공유            100 VU가 각자 다른 계정 사용
  → 같은 장바구니 충돌              → 독립적인 장바구니
  → 멱등성 키 동일 → 1건만 처리    → 멱등성 키 각자 다름
  → 성공 1건                       → 재고만큼 성공
```

### 멱등성 키와 동시성 테스트의 관계

이 프로젝트의 멱등성 키 구성:

```
memberId + storeId + orderType + customerRequest + 장바구니내용
```

같은 계정 + 같은 장바구니 → 멱등성 키 동일 → 100명 요청해도 1건만 처리  
→ 동시성 테스트 목적과 반대 결과가 나옴

### 결과 해석

```
재고 10개 설정 후 20명 동시 주문 결과

✅ 주문 성공: 10건   → 정상 (분산락이 초과 차감 방지)
❌ 주문 실패: 10건   → 정상 (재고 부족으로 거절)

성공 > 10건이면 분산락 오류 → 재고가 실제보다 많이 차감됨
```
